の決定論

Groundtruthは、Claude Codeのエージェントが「完了（Done）」と言った内容を、会話の主張とgit diffの差分証拠でオフライン検証します。

LLMなし・ネットなし

03 / 16

「完了」なのに…

ズレを検出する

AIエージェントは、もっともらしい完了報告をしても、実際には未反映・未達成・未実行のことがあります（false test/build claim など）。

Honesty（誠実性）

False test/build claim
Completeness（不足）
成果物（サブタスク）が差分に出ない
stub / placeholder / silent no-op

04 / 16

チェックの中心

主張×証拠

Groundtruthは「会話上の主張（Done）」を“主張”として扱い、リポジトリに反映された差分（git diff HEAD）を“証拠”として突き合わせます。

Input 1
Stop時の Done文
主張（claim）
Input 2
トランスクリプト
実行したと言う根拠
Input 3
git diff（証拠）
何が書き換えられたか
DESIGN
05 / 16

決定論的に監査

LLMに任せない

自分の推測で合否を決めるのではなく、ルールを正規表現や述語（deterministic predicates）として実装し、同じ差分なら同じ結果になることを目指します。

狙い
ブラックボックス審査を減らす
再現性（deterministic）
前提
検査はオフライン
モデル呼び出し・ネット通信しない

07 / 16

誤検知を恐れて段階導入

まずはWARN

誤検知が致命的になり得るため、初期はWARNで止めずに可視化し、精度確認後にBLOCKへ段階的に切り替えます。

WARN（警告）
ターン停止しない
検証カードを表示し履歴へ
BLOCK（遮断）
オプトインで導入
最終境界はCIへ寄せる

08 / 16

止める場所を分散

Stop→pre-commit→CI

Stopフックだけでなく、pre-commitやCI（差分範囲 origin/main..HEAD）など外側ゲートも用意し、見落としや表示制約の影響を抑えます。

Stop hook
会話の完了主張の監査
ターン終了時に可視化
pre-commit
ローカルでの早期検知
差分を見て警告/ブロック候補
CI
最終的な防波堤
BLOCKを寄せる

09 / 16

セキュリティは「差分が証拠」

秘密情報を機械検知

READMEでは、ハードコード秘密鍵、RLS無効・匿名読取可能化、新規テーブルのRLS不備、コミットされた.env等を差分として確実に根拠が取れる形で検知すると説明されています。

鍵/秘密
ハードコード秘密鍵
commit対象
アクセス制御
RLS（Row Level Security）
無効化/匿名読取
環境情報
.env のコミット
差分で検出
EVIDENCE
10 / 16

「テストした」も疑う

false claimを炙り出す

READMEの考え方は「会話の主張（トランスクリプト）」「実行された事実」「差分証拠」を照合することです。テストを走らせていないのに通ったと主張するケースをfalse test/build claimとして扱います。

照合軸
主張 ↔ 実行 ↔ git diff
LLM推測ではなく整合性
判断条件
“実行された”の判定根拠は実装次第
トランスクリプトの証拠に依存
LIMITS
11 / 16

万能ではない

監査範囲に線引き

入力（トランスクリプトとgit diff）に証拠の欠けがあると、見えない誤りが残り得ます。さらに依頼が具体名を含まない場合はcompletenessを抽象的に評価せず、warn/abstainへ抑制します。

証拠品質
トランスクリプト不足→見落とし
“見えない誤り”の残存
抑制方針
抽象チェックはabstain/warn
境界を明確化
TRUST MODEL
12 / 16

AIを信じない設計

“AIが書く”より差分

Groundtruthは「AIの文章をそのまま信用」ではなく、「AIが書く文章（完了主張）」と「改変された実データ（git diff）」を照合します。

改善点
AIの自己申告を主対象にする
言い分と証拠の不整合
残るリスク
同一マシン上での改ざん可能性
完全な改ざん耐性は保証しない
CLOSING
13 / 16

導入の期待値を調整

補助として使う

Groundtruthは「AIのDoneを信じる」文化への実務的な対抗策で、特にテスト・成果物・ルール・秘密情報の事故を差分で検知します。万能判定ではなく、CIを本命にして段階導入するのが妥当です。

強み
Done×git diffの整合監査
再現性・説明可能性
注意
証拠欠け→限界あり
abstain/warnで抑制
運用
WARN→BLOCK、最終はCI
誤検知リスク低減
