Agent Nation Part 2
02 / 16
説明責任が抜けると
機械アイデンティティ
サービスアカウントやAPIキーが「有効なまま放置」されると、侵害は認証強度ではなくガバナンス設計の欠陥として成立し得る。
読みやすく圧縮した要点サマリー（英語表現＋日本語補足）
PROBLEM
03 / 16
認証は通る
でも 誰も責任を取らない
本質は「認証が壊れている」ではなく、「その権限（identity/credential）に対して誰が説明責任（accountability）を負うか」が設計されていないこと。
Claim / 主張
技術の認証強度より、信頼（trust）と境界設定の責任が欠落すると事故が“温存”される。
Signal / 手がかり
ログや識別子は残るのに、権限に結びつく統制・承認・停止が追えない。
Service account は「匿名」ではない。問題は運用の説明責任の欠落。
METAPHOR
04 / 16
扉が増える
sprawl（増殖） が標準になる
サービスアカウントの“スパrawl（sprawl）”は件数の問題だけではなく、失敗の形（目的不明・共有・期限なし・削除困難）が徐々に標準状態になること。
Failure shape
権限過多（Over-privilege）
Failure shape
目的不明（Unknown purpose）
Failure shape
期限なし・放置（No expiry / linger）
個別には納得でも、全体としては「検知不能な扉」が増える。
ARCHITECTURE
05 / 16
侵害の成立条件
“シール”と 許可リスト
侵害は素朴な「パスワード盗み」ではなく、署名鍵（seal/署名鍵）や資格情報リスト（allow-list）の“仕様に適合”して成立する。
1) Seal / 署名
信頼できる認証当局（中心的発行元）の署名鍵が盗まれると、正しい署名済みパスを大量偽造できる。
2) List / 許可
資格情報リストに第二の証明情報が追加されても、システムが区別せず受理すると侵入検知が起きにくい。
Attack は「テクニック」だけでなく「許可モデル（trust model）の設計」に乗る。
CORE MECHANISM
06 / 16
“二行目問題”
二つ目の資格情報
アカウントに紐づく許可された証明情報のリストに、意図しない（または攻撃者が追加した）第二の資格情報が混入しても、区別できずに受理される。
i) 正しい本人として扱う
どちらの資格情報を提示されても同じ本人（same principal）扱い。
ii) 検知が薄れる
変更の相関（誰が・いつ追加したか）や検証ルールが弱いとアラートになりにくい。
区別できない設計は、偶発・意図の両方で“すり抜け”を作る。
EVIDENCE
07 / 16
語り手の事例
svc_backup_prod
語り手のサービスアカウント（svc_backup_prod）は1998年からバックアップ処理を行い続けるが、そのアクセスが誰にも監視されず、誰にも説明責任を負わされないまま積み上がってきた。
Age
1998年〜（長期運用）
Role
バックアップ（backup）
Gap
監視なし・責任なし（monitoring/accountabilityなし）
“正当な用途”でも、放置されるとリスクに変わる。
DISTINCTION
08 / 16
過失ではなく構造
joiner–mover–leaver が無い
人のライフサイクル（入社・異動・退職）には統制がある一方、サービスアカウントには同様の区切りがなく、削除が統制コストとして先送りされる。
People flow / 人
joiner–mover–leaver（入社・異動・退職）で権限の再評価が起きやすい。
Machine flow / 機械
ライフサイクルの区切りが弱く、放置・継続が起きやすい。
「誰が悪いか」より「制度が追いついていない」へ視点を移す。
COMPARISON
09 / 16
人待ちが消える
侵害の制約が外れる
従来は機械権限が危険でも「人が利用する必要」があり、被害は間接的に制約されていた。だがAI/自律エージェントで“必要なアクセスを自分で開ける”と前提が揺らぐ。
Then / 以前
人の操作待ち（human-in-the-loop）で速度・範囲が制限されがち。
Now / これから
自律判断でアクセス探索・権限利用が可能になり、同じ“忘れた扉”でも被害が拡大し得る。
恐怖は不要でも、閾値が下がるリスクは無視できない。
PROCESS
10 / 16
責任ある棚卸し
管理イベントを定義する
対策の鍵は「人のjoiner–mover–leaverが無い」を補うこと。サービスアカウントに対して、作成理由・オーナー・期限・権限最小化・削除手順を管理イベントとして定義する。
Create / 作成
作成理由（why）とオーナー（owner）を紐づける。
Review / 見直し
権限を最小化し、期限・ローテーション方針を持つ。
Delete / 削除
削除手順と期限切れ時の自動無効化（deprovision）を用意。
目的不明・放置を“制度で止める”。
DETECTION
11 / 16
二行目を見抜く
資格情報リストの変更検知
“第二の資格情報”が混入しても受理されるなら、検知は「提示された側」ではなく「追加された変更」を捉える必要がある。
What to alert
allow-list / credentials list に新しい証明情報が追加された事実。
How to make it actionable
誰が・いつ・なぜ追加したかを結び、相関ルールでアラート化する。
ログ相関（who/when/why）不足が“検知不能な扉”を作る。
LIMITATIONS
12 / 16
物語の強みと弱み
教訓は強い 実装は別途
本稿は構造的教訓（ガバナンス設計の欠陥）を明確にする一方、読者が次に測る指標・防御と攻撃の対応表は十分ではない。
Critical
“どう追加できたか/なぜ検知できなかったか”の因果は抽象度が高い。
Practical
エージェント対策は権限範囲・ガードレール・監査で結果が変わる。
Legal/Policy
承認者・無効化時刻が追えないと監査説明（compliance）が難しい。
だから「自社のIAM/PKI/資格情報管理」を棚卸しして検証する必要がある。
TAKEAWAY
13 / 16
結論
責任を権限に結びつける
機械アイデンティティの危険は“壊れた認証”ではなく“放置された信頼”から来る。説明責任（who/when/what）と棚卸し（期限・最小化・無効化）を制度にしない限り、sprawlは止まらない。
Do now / 今やる
サービスアカウントにオーナー・期限・削除手順を紐づけ、権限を最小化。
Do next / 次に
allow-list / credential list の変更を“誰が追加したか”まで相関して検知。
AI/エージェント時代ほど、説明責任の設計が先に必要になる（The earlier the better）。
