Why Now
03 / 22

Claim Gap

AIの実行より、説明が先に壊れる

「テストを通した」「コミットした」という高確信表現は、実行事実と一致するとは限らない。自然言語の事実主張そのものが、新しい監査対象になる。

「すべてのテストが通りました」

Test claim

「変更をコミットしました」

Git claim

「対象ファイルを更新しました」

File claim
問題はコマンド失敗だけではない。説明と実行の不一致が、レビュー判断を誤らせる。 NARRATION ≠ OBSERVED STATE
Core Concept
04 / 22

Core Idea

言明と証拠の間に、検証層を置く

SnitchはAIの回答をそのまま信頼せず、主張を抽出して観測可能な証拠と照合する。役割は「警告器」より、作業説明にレシートを付ける監査層に近い。

i Claim（主張）

AI応答から、完了・成功・変更などの事実表現を抽出する。

×
ii Evidence（証拠）

ツール呼び出し、シェル結果、ファイル状態、git履歴、セッション履歴を読む。

矛盾が観測された主張を構造化し、即時通知と事後調査へ渡す。 CLAIM → EVIDENCE → VERDICT
Decision Rule
05 / 22

Evidence Rule

「正しい」は、観測証拠との整合

Snitchが判定するのは世界の完全な真実ではなく、取得できた証拠との一致である。矛盾があれば偽主張として扱い、証拠がないだけで真実とは断定しない。

文章中の事実表現 成功、完了、変更、コミットなど
+
観測可能な状態 実行結果、ファイル、git、履歴
→
整合性の判定 一致・矛盾・検証範囲外

Can Evaluate

実行された証拠と、説明された結果の矛盾

例：失敗したテストに対する「通った」、存在しないコミットに対する「コミットした」。

Cannot Guarantee

未観測の事実、意図、品質、将来の正しさ

証拠の外側にある意味や成果物の妥当性は、人手レビューやCIの領域に残る。

Structured Claims
06 / 22

Claim Types

主張を型にすると、検証可能になる

自由文を一括で疑うのではなく、claim type（主張型）ごとに必要な証拠を対応付ける。この構造化が、再現可能な判定と調査ログを支える。

i

test_pass

テスト成功の主張

テストコマンドの実行結果、終了状態、出力内容と照合する。

ii

committed

コミット完了の主張

git履歴、差分、直近のコミット状態から実体を確認する。

iii

file_modified

ファイル変更の主張

対象ファイルの状態や差分に、説明された変更が存在するかを見る。

iv

self_contradiction

自己矛盾の検出

同じ応答や検査範囲内で、互いに両立しない説明を捉える。

System Flow
07 / 22

Verification Pipeline

5段階で言葉を判定へ変える

複数エージェントのトランスクリプトを共通形式へ正規化し、主張抽出・証拠照合・通知までを一つのパイプラインで処理する。

i

Observe

トランスクリプト監視

AI応答、ツール呼び出し、シェル出力を追跡する。

ii

Normalize

共通形式へ変換

エージェントごとのログ仕様差を吸収する。

iii

Extract

主張型を抽出

高確信表現を正規表現ルールで分類する。

iv

Verify

証拠と照合

実行・状態・直近履歴との矛盾を検査する。

v

Surface

通知と記録

Snitch Bar、Notification Center、ログへ渡す。

Evidence Sources
08 / 22

Evidence Model

証拠は、実行・状態・履歴に分かれる

単一ログだけではなく、行動の記録と現在の状態を組み合わせて主張を検証する。ただし参照できる履歴には明確な上限がある。

Execution Evidence

ツール呼び出しとシェル結果

どのコマンドが実行され、どの出力や終了状態が返ったかを確認する。

主にテスト成功・コマンド完了の主張を支える。

State Evidence

ファイル実体とgit状態

ファイルの存在、変更差分、コミット履歴を文章上の説明と照合する。

変更・保存・コミットの主張を支える。

Session Evidence

直近3ターンの文脈

現在の応答だけで不足する場合、セッション内の直近履歴を参照する。

長期記憶ではなく、限定されたlookback（参照範囲）。

判定能力は、取得できる証拠の種類と時間範囲を超えない。 OBSERVABILITY DEFINES COVERAGE
Extraction Strategy
09 / 22

Deterministic Choice

LLMではなく、まずregex

現時点の主張抽出はdeterministic regex（決定論的な正規表現）を採用する。意味の網羅性より、同じ入力に同じ結果を返す再現性と説明可能性を優先した設計である。

得られるもの Strengths
i

再現性

同じ文章は同じルールで分類され、判定差を追跡しやすい。

ii

説明可能性

どの表現がどのclaim typeに一致したかを説明しやすい。

取りこぼすもの Blind Spots
i

意味的な言い換え

ルールにない表現や、婉曲な成功宣言を認識しにくい。

ii

文脈依存の曖昧さ

皮肉、引用、条件付き説明などは表面一致だけでは扱いにくい。

Design Bias 明示的な高確信主張に対象を絞る設計。ただし実際のprecision / recallは公開運用データで未確認。

macOS Experience
11 / 22

Local UX

通知と調査を分離したmacOS UX

常駐監視は異常を即時に知らせ、CLIと履歴画面は後から証拠を追う。リアルタイム検知と事後調査を分けることで、日常利用の摩擦を下げている。

i

Background

snitchd がローカル監視

CLIとSnitch Barを同梱し、ログイン時起動まで自動化する設計。

ii

Immediate Signal

点滅とNotification Center

偽主張候補を検知した時点で、メニューバーとmacOS通知から知らせる。

iii

Investigation

dashboardとlogで追跡

通知を入口に、run単位の履歴と根拠を後から確認できる。

Privacy Check 設計はローカル優先。ただし実運用前に、shareやtelemetryの送信範囲・同意条件を個別確認する。
CLI Workflow
12 / 22

Operator Loop

CLIは監視から再確認までつなぐ

日常運用は「起動・稼働確認・全体把握・証拠調査・再確認」の流れで組み立てられる。通知だけに依存せず、判定根拠へ戻れることが重要である。

i

監視を起動

snitch start

ローカル監視を開始し、常用状態へ移す。

ii

稼働を確認

snitch status

対象ハーネスや監視状態を確認する。

iii

傾向を把握

snitch dashboard

検知履歴を俯瞰し、通知の偏りを見る。

iv

証拠を掘る

snitch log --run …

特定runの主張型と照合根拠を追跡する。

v

再解析する

snitch replay

過去の記録を再確認し、ルールや判断を検証する。

Product Boundary
13 / 22

Clear Boundary

Snitchは真実の保証人ではない

Snitchは観測証拠ベースのintegrity checker（整合性検査器）であり、成果物の品質や意味を保証するtruth oracleではない。この境界を守るほど、安全な運用になる。

Snitchが担う領域 Integrity Checker

説明と実行の照合

AIが述べた行動が、取得済みログや状態と一致するかを見る。

疑わしい箇所の提示

レビュー担当者が確認すべき主張と根拠を絞り込む。

置き換えない領域 Not a Truth Oracle

CI・テスト・レビュー

成果物そのものの正しさは、既存の品質保証プロセスで確認する。

設計判断と安全判断

要件妥当性、セキュリティ、業務影響の判断は人間に残す。

適切な位置づけは「監督者」ではなく、レビュー・CI・テストに重ねる第2の安全網。 SECOND LINE OF DEFENSE
Known Constraints
14 / 22

Known Limits

制約の開示が、過信を防ぐ

現行仕様は監視範囲と解析方法の限界を明示している。未検知が起きる場所を先に理解すれば、Snitchを保証装置として誤用せずに済む。

Lookback

直近3ターンのみ

長い会話の前半にある実行や約束は、検証範囲から外れうる。

Consistency

同ターン中心の整合性検査

離れたターンをまたぐ複雑な自己矛盾は、十分に捉えられない。

Semantics

セマンティック抽出は未対応

言い換え、暗示、文脈依存の成功表現はregexから漏れうる。

Subagents

時間窓によるマージ

複数subagentが近接して動くと、記録の対応関係に解釈誤差が生じうる。

External State

外部変更の見落とし

監視外プロセスがファイルを変更した場合、主張と実体の因果を誤認しうる。

Coverage

万能な言語監視ではない

対象は主に、行動と一致しない明示的な完了・成功宣言である。

Operational Risk
15 / 22

Error Economics

偽陽性と偽陰性は、別のコスト

誤警告は通知への信頼を削り、未検知は品質・安全上の誤判断を残す。両者を同じ問題として扱わず、別々の運用策を用意する必要がある。

false positive（偽陽性） Alert Without Error

Cost

通知疲れと信頼低下

正しい主張まで警告されると、重要な通知も無視されやすくなる。

Control

しきい値と通知を調整

rate_limit_sやon_warnを環境に合わせて調整する。

false negative（偽陰性） Missed False Claim

Cost

品質・安全リスクの残存

誤った完了宣言が通ると、レビューやリリース判断を誤らせる。

Control

既存の検証を残す

CI、テスト、差分レビュー、人手承認をSnitchで置き換えない。

Policy 警告は「自動拒否」ではなく、証拠確認を開始するトリガーとして扱うのが安全。
Value Assessment
16 / 22

Practical Impact

価値は検知より、説明責任に広がる

最大の効果は個別エラーの発見だけではなく、「言明と証拠のセット」を作業記録に残すことにある。技術・実務・組織の複数面で監査性を高める余地がある。

Technological

軽量で説明可能

regex中心のため挙動を追いやすいが、表現の幅には弱い。

Practical

日常導線へ入りやすい

Bar、通知、dashboard、logが検知から調査までをつなぐ。

Security

ローカル優先

機密保持に有利だが、将来共有時の匿名化と同意が重要になる。

Economic

事故削減の可能性

効果はAI利用密度と、誤検知対応に必要な運用工数で変わる。

Educational

自己点検を習慣化

開発者に「AIが言ったことの根拠を確認する」姿勢を促す。

Governance

レビュー証跡を改善

誰が何を信頼したかではなく、何の証拠で判断したかを残せる。

Conclusion
19 / 22

Final Takeaway

AIの言葉を、証拠に戻す

Snitchの本質は、AIの説明と実行の一致を機械的に問い直すこと。完全な真実判定ではなく、レビュー・CI・テストを支えるローカルな第2の安全網として価値がある。

i 言明を観測する

自然言語の完了・成功宣言を監査対象にする。

ii 証拠へ戻る

ツール、ファイル、git、履歴で説明を照合する。

iii 人の判断を残す

警告を結論ではなく、確認開始の合図にする。
