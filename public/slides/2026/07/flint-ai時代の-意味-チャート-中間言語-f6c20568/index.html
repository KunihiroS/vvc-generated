AI時代の“意味”で描く

Flint 中間言語

人が編集しやすいコンパクトなspecを、Vega-Lite/ECharts/Chart.jsへ自動でコンパイルして、信頼できるチャート生成を目指す可視化言語です。

Problem
03 / 16

低レベル指定の罠

spec が壊れる

従来は、スケール・軸・レイアウト・色などを低レベルに細かく指定しがちで、AIが作るとミスが増えます。結果として「見た目はそれっぽいが、意味がズレる」危険があります。

Architecture
04 / 16

仕様→コンパイル

意味から 描画へ

Flintは「仕様(spec) → コンパイル → レンダリング」を前提に、意味（semantic types）を入力として、バックエンド向けの完全な描画設定を導出します。

入力 / Input
データ + 列の意味（semantic types）
出力 / Output
Vega-Lite / ECharts / Chart.js 用の設定
Metaphor
05 / 16

“翻訳機”としてのspec

semantic types

semantic types（意味型）は、列が「時間・量・カテゴリ・符号付き概念」など何を表すかを示すラベルです。Flintはこれを手がかりに、軸や色の解釈を“翻訳”します。

Enumeration
06 / 16

自動推論の核

4つの 自動化

Flintの要点は、(i) 意味→設定の推論、(ii) レイアウト最適化、(iii) 低レベル依存の削減、(iv) 複数バックエンド吸収です。

i. 意味→設定
時間/量/カテゴリ/符号などを軸・書式・色へ反映
ii. レイアウト
小マルチ等の密度に応じて破綻を抑える
iii. 低レベル削減
チャート型とエンコーディング変更で一貫して更新
iv. バックエンド吸収
Vega-Lite/ECharts/Chart.jsの差を裏側で吸収
Evidence
07 / 16

例：time と diverging

見え方に直結

提示情報では、period（YearMonth）を時間として扱うためのパーサや、Profitの符号付き値に発散色（domainMid=0）を当てるなど、意味型が結果を決めます。

時間 / Time
YearMonth(period) → 時系列スケール + 軸書式
符号付き / Signed
Profit → redblue 発散 + 中点（0）
Process
08 / 16

破綻しない“配置”

密度に 強く

小マルチの数や密度が変わっても、弾性レイアウトモデルの考え方で、キャンバスに収まるようにバンド幅やサイズを調整します。

課題
カテゴリ増で読めなくなる / 画像が潰れる
対策
弾性モデルで配置パラメータを自動最適化
Comparison
09 / 16

“上位互換”ではない

Flint ≠ Vega-Lite

FlintはVega-Liteの上位互換と断定できません。むしろ最終的にVega-Lite/ECharts/Chart.jsへコンパイルする“中間言語”です。

Flintの位置
統一インターフェース + 意味→描画の推論
バックエンド
最終レンダリング実行（表現・APIの差は吸収）
Distinction
10 / 16

semantic types がないと

解釈が 崩れる

semantic typesがないと、時間として扱うべきか、量として並べるべきか、符号付きゆえ発散色にすべきかを推論しづらくなります。すると軸・書式・色が誤り得ます。

時間 / Time
軸スケールや書式がズレる
量 / Quantity
集計や表現の意味が変わる
符号 / Signed
中点（0等）の扱いが変わる
Evidence
11 / 16

“統一”で差を隠す

表現差は 裏で吸収

バックエンド差異は統一インターフェースで隠蔽されるとされています。例えば、Vega-Liteにネイティブがない表現はEChartsへ切り替える、という思想です。

狙い
同じspec思想で移植・運用を進める
メリット
スタック制約を受けにくい / 実装が簡素化
Process
12 / 16

導入で確認すべきこと

信頼性を 検証

提示情報だけでは、semantic typesの体系・誤推論の評価・曖昧入力の挙動・エラー説明が不明です。導入時はガードレールと診断を前提に確認するのが安全です。

型の境界
例：Quantity/Count、単位、タイムゾーン、欠損の解釈
診断性
なぜその軸・色になったか / 失敗時に説明できるか
品質指標
読み取り正答率、誤り率、破綻条件の有無
制約
ブランド色・厳密スケール・アクセシビリティ要件の調整権限
Closing
13 / 16

意味→自動コンパイル

結論： “それっぽい”から

Flintの主眼は、semantic typesの推論と自動コンパイルで、見栄え・一貫性・運用しやすさを底上げすることです。導入では、誤推論や欠損・単位などの境界条件を必ず検証してください。

参考：Vega-Lite / ECharts / Chart.js
バックエンド候補として提示されている可視化基盤（Flintは最終的にこれらへコンパイルする位置づけ）
参考：semantic types（意味型）
列の意味を指定し、軸・書式・色（発散スキームの中点等）を推論する中核コンセプト
参考：MCP Server（エージェント連携）
AIエージェントのワークフローで使う前提として言及されている提供形態
