Causari causality

狙い / Goal
説明可能性（Explain）と検証可能性（Verify）
中核 / Core
content-based join（内容結合）＋proof（証明）＋seals（受領書）

03 / 16
Gitの“差分”だけでは足りない
なぜ 追えない
Gitは「いつ・誰が・何を変えたか」に強い一方、AIが変更するまでに見ていたプロンプトや推論の流れまでは追いにくいです。
観測欠落
プロンプト／トークン
推定の壁
推定コスト／根拠の連鎖
運用課題
監査・デバッグで“整合”が問われる
結論：AIコードをブラックボックスにしない仕組みが必要です。
metaphor
04 / 16
“ログ”ではなく“因果の線”
追跡は 結合
Causariは、AIの自己申告に依存せず、別ストリームの観測（LLM出力とファイル変更）を内容で突き合わせて“因果的に結ぶ”発想です。
re proxy（観測）
LLMトラフィック（prompt/completion等）
re watch（観測）
ファイル書き込み（inserted lines等）
ポイント：エージェントの協力なしで事実を集める方向。
architecture
05 / 16
content-based join（因果フィンガープリント）
結び目は 内容
挿入されたコード行を直前のcompletion内で探索し、マッチを“指紋（fingerprint）”として因果に結びます。
結合対象
ファイルの“書き込み行” ↔ completionの“一致する断片”
独立性
プロンプト・トークン・コスト・応答内容 と 書き込み を別ストリームで取得
注意：入力データ上では、衝突耐性や確信度（confidence）のアルゴリズム詳細は未確定です。
process
06 / 16
why / trace / impact で地図を描く
因果は 上流 と 下流
特定のコード行について、導入元（why/trace）と波及範囲（impact）を因果に基づいて提示します。
re why（導入者/理由）
その行を書いたのは“誰/何”
re trace（上流）
間接寄与まで“寄せて”辿る
re impact（下流）
ブラスト半径（blast radius）を提示
用途：復元・デバッグ・リバート判断を“結果だけ”から“起点の意図と依存”へ。
evidence
07 / 16
proof と Crovia Seals
改ざんに 強く
Ed25519署名・ハッシュ連鎖・オフライン検証可能な領収書（receipt）で、ログが“正しいと検証できる”証跡を作ります。
Crovia Seals
IETFドラフトの受領書形式（receipt）
proof（証明）
署名付きの検証可能成果物（fail-closed を志向）
狙い：監査・説明責任で“あるからではなく、正しいと確かめられる”状態へ。
comparison
08 / 16
ログ可視化 vs 因果追跡
差は 結合 と 検証
従来は“見えるログ”で終わりがちですが、Causariは“どの要求から生まれたか”を検証可能にする方向性です。
従来（例）
変更結果（diff）中心、上流の意図は推定しがち
Causari
re proxy + re watch を content-based join で結合
従来
監査で“正しさ”の独立検証が難しいことがある
Causari
proof / seals によりオフライン検証を志向
結論：説明可能性（causality）と検証可能性（proof）をセットで狙う。

成功を“スキル”へ圧縮
re skill distill
検証済みの過去イベントを蒸留（distill）し、署名付きで配布・再利用できる形にします。
distill（蒸留）
成功イベントを圧縮して“再現しやすい単位”へ
export / import / pull
チーム共有と取り込み（trust）を制御
trust（信頼）
未知の署名キーは読み込まない（改変も拒否する方向）
期待効果
同じ失敗を繰り返しにくくする
運用面の理解が必要：粒度や評価ラダーは入力データだけでは不明です。

content-based join

12 / 16
私見：価値とリスク
説明責任 × 再現性
Causariは単なる可視化ではなく、上流/下流を因果で追って監査やデバッグに耐える形に落とそうとしています。
強み / Strength
エージェント協力なしの観測＋署名付き証跡
リスク / Risk
誤結合・欠落データ・ツールチェーン変形で正確性が揺れる
導入判断
追跡精度／欠落時挙動／証明運用を段階評価
結論：導入は“検証設計込み”で進めるのが現実的です。
