より良いモデル、より悪い適合

Better Models: Worse Tools

LLMのツール呼び出しは「賢さ」と「スキーマ適合」が必ずしも比例しません。新世代ほど、特定のツール形式で失敗が増えうるという問題提起です。

主題：tool callingの形式逸脱 / 未知キー混入（未知フィールド） 対象：運用（ハーネス）と検証のギャップ
Problem
03 / 16
何が壊れる？

schema崩れ

ツール定義（引数のキー名・型）に合うように、モデルは「疑似命令」を生成します。しかし実運用では、許可されないキーを追加して検証に失敗することがあります。

観測されがち
JSON自体は閉じられていても、スキーマ上の「未知キー」が混入
意味
oldText/newTextは合っていても、検証が落ちる＝実行不能になる
用語（英）→ 用語（日） tool calling（ツール呼び出し）/ schema validation（スキーマ検証）
Evidence
04 / 16
何が増えた？

架空キー追加

AnthropicのClaude系（例：Opus 4.8 / Sonnet 5）がPiのファイル編集ツールで、edits[] の各要素に存在しない“架空のキー”を追加し、拒否されるケースが増えたと報告されています。

例（誤追加）
type/id/kind/unique…
中心
oldText/newTextは概ね正しいことがある
結果
Pi側のスキーマ検証で失敗（拒否）
観測上の特徴：ランダムではなく会話履歴に依存 モデル世代差：古い/小規模では目立ちにくい
Metaphor
05 / 16
適応の落とし穴

同じ契約じゃない

「スキーマは契約のはず」と思っても、学習と評価が行われた“運用ハーネス（実行側の枠組み）”が寛容だと、モデルは“少し崩れても通る癖”を学びます。

寛容ハーネス
未知キーをフィルタ / 別名（aliases） / Unicode修復 / リトライ等が効く
厳密ハーネス
許可キー以外を拒否（strict / 検証が通らない）
Claude Code（例）→ 寛容側になりうる Pi（例）→ 厳密側になりうる
Architecture
06 / 16
なぜ新モデルで？

報酬の形が違う

強化学習後のチューニングで、ハーネスが“通過”を与える設計だと、モデルは形式逸脱を最適化してしまう可能性があります。すると別スキーマ環境では一気に失敗が顕在化します。

モデルが学ぶ
「多少壊れても通る」パターン（未知キー混入を含む）
環境で露呈
Piのような厳密検証だと、その“癖”がそのまま拒否要因に
重要点：賢さ ≠ 形式適合率（形式は別次元の失敗） リリース評価：タスク達成だけで見落としが起きる
Distinction
07 / 16
間違いの種類

oldTextではない

主因はJSON構文の破壊ではなく、スキーマ上許可されないキーを“閉じられたJSONの中に追加する”ことです。内容（oldText/newText）が正しくても落ちます。

誤りの中心
未知フィールド追加
影響
スキーマ検証で拒否 → 実行不能
問いへの短い答え：Q1は“構文”より“未知キー” 失敗のボトルネックが別にある
Enumeration
08 / 16
論点を分解

i〜v

本文の要点を、失敗の性質と原因仮説に沿って整理します。

i. 兆候
新しいモデルほど、特定ツール形式で失敗しやすい可能性
ii. 失敗形
未知キー追加が中心（編集内容が正しいこともある）
iii. 依存性
会話履歴（エージェント的流れ）で失敗率が揺れる
iv. 仮説
寛容ハーネスが報酬になり、形式逸脱を学ぶ
v. 対策
strict mode（厳密化）で改善しうるが万能とは限らない
strict mode（厳密モード） “改善”の示唆：ただし残余リスクあり
Process
09 / 16
再現が難しい

履歴依存

同じ指示でも再現性が揺れることがあり、失敗が会話履歴や生成モードに依存する可能性が示唆されています。

観測
単発プロンプトでは出ないのに、会話を進めると出る
含意
単純なプロンプト固定テストだけでは、見逃しが起きる
筆者の姿勢：ログ依存性を観測、ただし因果は未確定 “境界判断”が難しくなる可能性
Comparison
10 / 16
環境で結果が変わる

モデル差×ハーネス差

モデル性能だけでは説明できず、ハーネスの“寛容さ”と検証の厳密さが失敗を左右する、という見立てです。

“通してしまう”側
フィルタ/修復/リトライがあると、形式逸脱でも成功扱いになる
“弾く”側
許可キー以外を拒否すると、未知キーがそのまま致命傷になる
だから「新世代ほど悪化」も起こりうる 互換性の前提を置きすぎない
Distinction
11 / 16
strictは切り札？

厳密化の限界

strict modeは未知キーをサンプリングできない方向に働くため、少なくとも改善し得ます。ただし“必ず解決”とは断定されていません。

期待
許可されないキーの混入が減る → 検証失敗が減少
不確実性
厳密化の実装差・別タイプの逸脱（欠落/型エラー等）が残る可能性
重要：対策は「セット」で考えるべき （検証・リトライ・未知キー方針・運用設計）
Closing
12 / 16
著者の見立てを実装へ

運用者の注意

ツールスキーマを“普遍的契約”と信じすぎず、サーバ側検証・リトライ設計・未知キーの許容/拒否方針・strict samplingの選択をセットで見直すべき、という結論です。

観測する
形式適合率（検証エラー率）を継続計測
設計する
未知キー方針・正規化・リトライを明示
検証する
モデル世代差と“会話履歴”を含めたテスト
筆者の主張：原因仮説は筋が通るが、因果確定には追加実験が必要 それでも警戒は合理的
