COVER
02 / 16
ブラウザで動画を合成

WebMotion

フレーム番号に対して描画を決定的にし、ブラウザ内でMP4をクライアント完結する発想です。

English core: deterministic (決定的) / frame-accurate (フレーム厳密) / client-side encode (クライアント側エンコード)

対象: 一般ユーザー〜技術者（Remotion代替の文脈）
PROBLEM
03 / 16
ズレが怖い

プレビュー不一致

動画編集・レンダリングは、プレビューと書き出しで見た目がズレることが制作の不確実性になります。

よくある負担
headless Chrome / FFmpeg / レンダーファームなどの外部実行
現場の痛点
シーク（例: frame=42）で結果が変わると、デバッグが遅くなります
“まず動く”より“同じ結果が出続ける”が重要、という問題設定
METAPHOR
04 / 16
フレームは関数

frame → 同一画像

「現在のフレームに見えているものはすべて、そのフレームの純粋関数である」というルールが中核です。

Seeker
42フレーム目へシークしても同じ
Preview
プレビューと同じピクセル再現を目標
Export
書き出しでも同じ画像
Japanese note: 決定性（deterministic / 決定的）とは「同じ入力（frame）→同じ出力（画像）」のこと
ARCHITECTURE
05 / 16
HTMLで宣言

動画を構文で書く

`w-composition` / `w-sequence` / `w-animate` などの要素で、動画をDOM/HTMLのように記述します（HTML中心の制作モデル）。

宣言的（declarative / 宣言）
w-composition：全体の器 / シーン定義
モーション（motion / 動き）
w-animate：フレーム範囲の補間（tween）として定義
テキスト
w-text：文字要素
反復・参照
w-for / w-defs：定義を名前で再利用
EVIDENCE
06 / 16
契約がある

renderFrame契約

フレームインデックスを受け取って描画する renderFrame のような小さなコンポーネント契約が、決定性を支えます。

入力
frame番号（例: frame=42）
出力
そのフレームの描画結果（Canvas/DOMラスタライズ等）
破りやすい点
外部の時刻・外部ランダム・環境差（フォント/レンダリング）を入れると、決定性が崩れるリスク
日本語の要点: “純粋関数ルール”を守る設計が前提
PROCESS
07 / 16
ブラウザ内エンコード

MP4をクライアント完結

書き出しでは、ブラウザ内でエンコード（WebCodecs等）を行う方針です。外部レンダリングが不要になる一方、対応と性能がボトルネックになり得ます。

要点
headless/FFmpeg/レンダーファームなし
前提
Chromiumベース推奨
必要要素
WebCodecs H.264 / OffscreenCanvas
ARCHITECTURE
08 / 16
2つの道

Canvas と html-in-canvas

描画バックエンドはCanvas中心だけでなく、DOMをHTMLのまま扱いフレームごとにラスタライズする html-in-canvas 系も示されています。

CanvasRenderer
描画をCanvasへ：フレームごとにラスタ
html-in-canvas
DOM→HTML表示→ラスタライズ：既存Web資産を活かす可能性
重要（品質保証）
決定性が「どの環境までピクセル一致するか」は、特にラスタライズ差で要検証
DISTINCTION
09 / 16
どこで崩れる？

決定性の落とし穴

決定性が成立するには「frame由来だけで計算する」必要があります。README要約だけでは保証範囲の詳細は不足しがちです。

外部時間
現在時刻（now）依存はNG（frame→結果が変わる）
外部乱数
Math.random等はNG（frameから再現できる設計が必要）
環境差
ブラウザ/GPU/フォント/サブピクセルでピクセル差が出る可能性
Action: “ターゲット環境での一致テスト”が導入判断の鍵
COMPARISON
10 / 16
用語ミニ辞典

英日で把握

WebMotionの核は、決定性とframeベースのモーション定義です。読むときの頻出語を短く整理します。

deterministic / 決定的
同じframe入力で同じ見た目（画像）
frame-accurate / フレーム厳密
シークしてもフレーム単位で結果が合う
tween / 補間
あるフレーム範囲に対する補間でモーションを作る
client-side encode / クライアント側エンコード
ブラウザ内でMP4エンコードまで完結
PROCESS
11 / 16
制作フロー

作って出す

典型的には、宣言的なシーン定義→フレームごとの描画→ブラウザ内でMP4へ、という流れになります。

i. 定義
w-composition / w-sequence で構成
ii. モーション
w-animate 等でframe範囲のtweenを定義
iii. 描画
renderFrame でフレーム別にCanvasまたはDOMラスタ
iv. 書き出し
WebCodecs/OffscreenCanvasでMP4エンコード
AIコーディング向け “skills”/制作レシピ学習も用意される旨が示されています
EVIDENCE
12 / 16
賛否の軸

強みと条件

WebMotionは“プレビューと書き出しの差”を仕様レベルの決定性で潰す狙いが強い一方、環境差・エンコード依存を見極める必要があります。

強み
frameを指定すれば同一画像：キャッシュ・デバッグ・反復が速くなる
条件/リスク
一致の範囲は不明瞭：フォント/ラスタライズ差、ブラウザ対応、重いシーンで体験が変わる可能性
導入前に必要
想定ブラウザ群・使用フォント・外部画像/動画・解像度/fps・重さを前提にデモ/検証
