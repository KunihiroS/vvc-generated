Executive Summary
03 / 20

Executive verdict

有望な基盤。だが、まだ導入済み品質ではない

Cactusは速度・効率・導入容易性を同時に狙う、実装指向のモバイルAI基盤である。製品採用には電力、熱、品質安全域、クラウド移行の追加検証が必要だ。

i

統合された開発導線

Engine、Graph、Kernels、Quants、Transpiler、CLI、各種バインディングまでを一つのエコシステムとして提供する。

ii

定量的な設計判断

端末別の速度・RAMと量子化後の精度を並べ、効率と品質のトレードオフを可視化している。

iii

残る運用上の空白

長時間稼働、端末差、セキュリティ監査、ハンドオフの再現性などはREADMEだけでは判断できない。

Current position

PoC候補として高評価。 CQ4を初期基準に、実ワークロードで品質と端末運用を通過した範囲だけ段階導入する。

Problem Framing
04 / 20

The mobile constraint

「端末かクラウドか」の二択では解けない

端末は即時性・プライバシー・オフライン性に強いが、計算資源と電力が限られる。クラウドは高性能だが、通信遅延・費用・データ移転を伴う。

On-device

端末内で完結

低遅延、オフライン、データ最小化に有利。一方でRAM、帯域、発熱、バッテリー、モデルサイズが上限になる。

Hybrid local first
Cloud

クラウドで処理

大規模モデルと弾力的な計算資源を利用可能。一方でネットワーク依存、従量課金、プライバシー管理が増える。

Design response

軽量化して端末で処理し、端末では安全に完遂できない要求だけクラウドへ送る。 Cactusの中心命題はこの分業にある。

Core Idea
05 / 20

Local-first inference

ローカルを既定にし、必要時だけ移行する

量子化モデルを端末で実行し、信頼度や制約に応じてクラウドへハンドオフする。この構成が速度・費用・プライバシーの折衷点をつくる。

Step i 端末で入力を処理

テキスト、音声、画像などをローカルのモデル／カーネルで推論する。

Step ii 結果と制約を判定

confidence_threshold、失敗、処理時間、ポリシーを確認する。

Step iii 必要時のみクラウド

cloud_handoffを実行し、タイムアウト内で代替結果を取得する。

Routing criteria 推論の信頼度 通信・遅延の許容値 データ送信ポリシー
Architecture
06 / 20

One integrated stack

価値はエンジン単体ではなく、層全体にある

モデル変換、実行、最適化、量子化、アプリ統合までを分断しない。独自モデルを実機に持ち込み、計測する工程が一つのスタックに収まる。

Cactus Engine 推論実行、API、ローカル／クラウド制御の中心
Graph ネイティブな計算グラフと演算の構成管理
Kernels ARM・Apple系を含む端末向け低レベル最適化
Quants CQによる低ビット量子化と端末コスト削減
Transpiler 外部モデルをCactus向け形式へ変換する導線

Application surface

C++／Python実装を核に、モバイル・クロスプラットフォーム環境から利用できる。

Swift Kotlin Flutter React Native Python Rust

OpenAI互換APIにより、既存の呼び出し設計を活用しやすい。

Developer Experience
07 / 20

From model to device

変換から評価まで、一つのCLIでつながる

Cactusはデモ用ランタイムに留まらず、モデル準備・サービング・ベンチ・テストまでの再現可能な開発導線を提供する。

cactus run

対象モデルを実行し、端末上の挙動と出力を素早く確認する。

cactus convert

モデルを変換し、量子化やターゲット環境への適合を進める。

cactus serve

APIとして公開し、アプリや既存クライアントから統合評価する。

cactus benchmark / test

速度、メモリ、品質、回帰を同じ工程内で比較・検証する。

OpenAI-compatible API 呼び出しインターフェースの移行負担を抑えられる。ただし端末固有のモデル管理、権限、更新設計は別途必要。
Capability Scope
08 / 20

Beyond text generation

対象はLLMだけではない。複合AIを端末へ

音声認識、画像入力、RAG、ツール呼び出しまでを想定する。単発のテキスト生成ではなく、実アプリに近い推論フローが射程に入る。

i
Language

LLM・対話

ローカル生成、会話、要約などを端末上で実行し、応答遅延とデータ移転を抑える。

ii
Audio

Transcribe — 音声認識

音声入力の文字起こしを対象に含む。実用化には騒音、話者差、長時間音声での誤差評価が必要。

iii
Vision

VLM — 画像理解

画像とテキストを組み合わせる入力を想定。端末カメラとの統合はプライバシー上の利点が大きい。

iv
Augmentation

RAG・ツール呼び出し

外部知識や機能を接続し、回答だけでなく検索・操作を含むアプリケーションフローを構成する。

Quantization
09 / 20

Efficiency has a price

CQ量子化は中核技術。低ビット化は無料ではない

CQはRotation-and-codebook方式で、概ね4bit〜2bitへモデルを圧縮する。RAMと実行コストを下げる一方、ビット幅を攻めるほど品質リスクが増える。

Core mechanism より小さく、より速く

量子化は重み表現を小さくし、端末のメモリ帯域と計算負荷を抑える。モバイル実行を成立させる鍵だが、用途ごとの許容誤差が選定条件になる。

CQ4 初期基準に適したバランス型 主要ベンチでは比較的高い再現率を維持する傾向。
CQ3.26 効率を一段高める中間型 タスク間の劣化幅を実データで確認する必要がある。
CQ2 省メモリを優先する攻めた設定 一部タスクで急激な精度低下が見られ、安全域が狭い。
CQ = Rotation-and-codebook quantization. ビット幅の呼称と具体設定は対象モデル・リポジトリの版を確認すること。
Performance Evidence
10 / 20

Speed on Apple silicon

READMEの性能値は、端末AIの実効性を示す

Gemma系モデルの例では、Mac M5 Maxで高いprefill／decode速度が提示される。iPhone 15・17系を含む端末別ベンチも、モバイル実行の成立可能性を支える。

Prefill · Mac M5 Max

2964tps

入力文脈をモデルへ取り込む処理速度。長い入力の初期応答時間に影響する。

Decode · Mac M5 Max

154tps

トークンを逐次生成する処理速度。ユーザーが感じる生成テンポに直結する。

What it supports

Appleシリコン向け最適化とネイティブカーネルが、高いスループットにつながる可能性を示す。

What it does not prove

文脈長、バッチ、モデル版、熱状態、電源条件が変われば結果も変わる。自社条件での再測定が前提。

tps = tokens per second. 数値は提供資料に記載されたリポジトリREADMEのベンチマーク要約に基づく。
Quality Evidence

The quality cliff

CQ2では、品質が緩やかでなく急落しうる

CQ4は主要ベンチで比較的高い品質を保つ一方、CQ2ではARC-C 24.73という低下例が示される。最小ビットを選ぶことが最適化とは限らない。

Reported example 24.73 ARC-C · CQ2 提供資料に記載された精度表の例。低ビット条件でタスク性能が大きく崩れる可能性を示す。
CQ4
比較的高い再現率を保つ傾向。実務PoCの出発点として現実的。
CQ3.26

CQ2

バーは概念図であり、ベンチマーク得点の直接比較ではない。評価は提示タスク・約3 seedに限られるため、実ワークロード検証が必要。
Cloud Handoff
12 / 20

Fallback is governance

ハンドオフは機能ではなく、統治ルールである

低信頼時にクラウドへ切り替える設計は成功率を高めるが、データ移転・遅延・費用も発生させる。分岐条件は監査可能でなければならない。

i Local inference

端末で推論し、出力・信頼度・所要時間・エラー状態を取得する。

ii Policy decision

confidence_thresholdと送信可否、通信状態、制限時間を照合する。

iii Cloud handoff

cloud_handoffを実行し、cloud-timeout-ms内で結果を返す。

Traceability

なぜクラウドへ送ったかを記録する。

Data control

送信対象とマスキング範囲を定義する。

Calibration

信頼度自体が正しく校正されているか測る。

Economic View
13 / 20

Savings shift the work

クラウド費は減らせる。代わりに検証費が増える

ローカル推論は従量課金と通信依存を抑える可能性がある。一方、複数端末・複数量子化設定の評価と運用が新しいコストになる。

Potential gains

得られる経済価値

i
クラウド推論量の削減

端末で完結する要求が増えるほど、API従量課金と通信量を抑えられる。

ii
応答体験の改善

ネットワーク往復を避け、待ち時間による離脱や失敗を減らせる可能性がある。

iii
既存API資産の活用

OpenAI互換インターフェースにより、統合の初期負担を抑えやすい。

New cost centers

増える実装・運用負担

i
端末マトリクス検証

SoC、RAM、OS、温度、バッテリー状態ごとに性能と安定性を測る必要がある。

ii
量子化チューニング

モデルとタスクごとに、速度・容量・品質の安全域を探索する工数が生じる。

iii
二重経路の運用

ローカルとクラウド双方の監視、障害対応、モデル更新、ログ管理が必要になる。

Risk Review
14 / 20

The missing evidence

採用を止めるのは性能不足より、運用の空白だ

READMEは製品説明として有用だが、実デプロイの保証ではない。採用判断では、技術・セキュリティ・法務・安定性の未確認項目を明示する。

Operations

電力・熱・長時間安定性

連続推論時の消費電力、サーマルスロットリング、バックグラウンド実行、同時利用時の挙動が未評価。

判断質問：代表端末で30分以上、目標SLAを維持できるか。
Security

クラウド移行とデータ境界

音声・画像・テキストがいつ外部へ移るか、ログに何が残るか、送信禁止データをどう止めるかを定義する。

判断質問：分岐理由と送信内容を監査・再現できるか。
Legal

モデル規約とデータ法制

Cactus本体のLICENSEだけでなく、Hugging Face上のモデル規約、gated modelのトークン、音声データ法制を確認する。

判断質問：配布形態と商用利用が全ライセンスに適合するか。
Stability

実験機能と互換性

任意HFモデル変換はexperimentalとされ、OS別最適化や一部コア機能も変更されうる。

判断質問：固定版で回帰試験とロールバックを運用できるか。
Validation Plan
15 / 20

Four gates to adoption

ベンチ再現ではなく、製品条件で通過判定する

PoCでは速度だけを測らない。品質、端末運用、ハンドオフ、ガバナンスを独立したゲートとして評価する。

i

製品品質ゲート

長文要約、長期対話、専門語、長時間音声、画像入力を実データ分布で評価する。

Measure 正解率、失敗率、重大誤答率、量子化前後の差
ii

端末性能ゲート

低・中・高性能端末で初回応答、decode、RAM、電力、温度推移を測定する。

Measure p50／p95遅延、最大RAM、Wh、温度、速度低下率
iii

ハンドオフゲート

低信頼、通信不良、クラウド遅延、タイムアウト、ローカル再試行を故障注入で確認する。

Measure 移行率、成功率、誤移行率、総遅延、追加コスト
iv

統治・適合ゲート

送信可否、暗号化、保持期間、削除、監査ログ、モデルライセンスを運用手順へ落とす。

Measure 監査可能性、規約適合、権限分離、インシデント対応時間
Recommendation
16 / 20

Adopt in stages

Cactusは「試す価値が高い」。
答えは段階導入である

CQ4を起点に、クラウド移行を監査可能な形で限定実装し、代表端末と実データで四つのゲートを通す。合格した用途だけを本番へ広げる。

i

PoC

1モデル・1用途・代表端末に限定し、CQ4と非量子化基準を比較する。

ii

Limited rollout

端末範囲を広げ、ハンドオフ、監視、失敗時フォールバックを検証する。

iii

Production

品質安全域、熱・電力、法務・セキュリティ条件を満たした機能だけ公開する。
