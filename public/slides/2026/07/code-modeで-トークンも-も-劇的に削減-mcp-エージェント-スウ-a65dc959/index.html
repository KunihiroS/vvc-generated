COVER
実測でわかった経路設計の勝ち筋

Code Mode

MCP（Model Context Protocol）でツールを扱うエージェントにおいて、raw逐次呼び出しより「Code Modeのまとめ処理」が トークンもコストも劇的に減らせることを報告ベースで整理します。

03 / 16

PROBLEM
中間データがコンテキストへ

rawの重さ

rawなツール呼び出し（JSON等の生データを都度モデルへ入れる）では、 ツール結果の再投入が続き、トークン消費と費用が膨らみます。
Key idea
トークン＝請求・計算の単位
Failure mode
中間（生）JSONが積み上がる

04 / 16

METAPHOR
運ぶ量を減らす

荷物改革

Code Modeは「実行（処理）は現場でまとめて行い、モデルには要約だけ渡す」発想です。 モデルが受け取る“荷物（中間データ）”が小さくなるほど効きます。
Code Mode（仮）
サンドボックス実行 → return（要約のみ）
raw sequential（仮）
26回分の生JSONを都度コンテキストへ

タスク

EVIDENCE
文字量：桁違い

約126倍少ない

スクリプト経由のモデル到達データは約25,811文字（約6,450 tokens相当）で、 raw逐次では約3,259,649文字（約815,000 tokens相当・推定/含む）でした。
Code Mode
~25,811 chars（~6,450 tokens）
raw sequential
~3,259,649 chars（~815,000 tokens）

07 / 16

DISTINCTION
削減率：99.2%

ほぼ全消し

結論として、モデルに渡る中間データ量は約99.2%削減（=約126倍の差）に収束しました。 これは“圧縮”というより“データ経路”の差が本質です。
削減比率
約99.2%（中間生データを渡さない）
根本要因
モデルへ入るトークン量が激減

EVIDENCE

$も差が出る

Claude Sonnet 5（入力 $3 / 1M tokens）で換算すると、 スクリプト側は約$0.02、raw側はfloorで約$2.44と見積もられました。
モデル単価
Claude Sonnet 5：入力 $3 / 1M tokens
換算結果
Code Mode ~ $0.02 / raw floor ~ $2.44

09 / 16

PROCESS
なぜrawは全部測らない？

検証の自己矛盾

raw側を“本番会話のまま”26回分すべて同一コンテキストに入れると、 検証が破綻し得るため、サンプル測定→平均→外挿でfloorを作ったと説明しています。
スクリプト側
durationMsと返却文字数をライブ実測
raw側
SDKで生データ量を別測定し外挿（floor）

DISTINCTION

導入→磨きへ

彼らのスウォームは既に、system.agent.context_modeなどのrubricで「大量ツール呼び出しを避け、スクリプトを選ぶ」方針を持っていました。 この報告は“Code Mode採用”の正しさではなく、そのrubricを数字で磨く段階に進める材料です。
以前

11 / 16

COMPARISON
一般化の見取り図

効く条件

削減率（約99.2%）の“比”はモデル依存性が小さいと示唆され、主因はトークン量の差です。 効果は「ツール結果の大きさ」「モデルへの再投入頻度」「最終要約の圧縮度」に依存します。
大きいほど有利
ツール結果が巨大（生JSON）
再投入されるほど
後続ターンでコンテキストに残る
最後に圧縮
列挙→統計化→要約が相性良

12 / 16

EVIDENCE
懸念点：raw floorと外挿

数字の前提

raw側の$2.44はfloorで、実際には後続ターンでも生結果が再投入され得るため上振れの可能性があります。 また外挿（サンプル→平均→実総数）なので、データ形状が変われば比率も動きます。
rawコスト
floor推定（実コストは高くなる可能性）
外挿リスク
レスポンス/分布が違うと削減比率が変動

13 / 16

CLOSING
実務のアクション

rubricを磨く

結論は「魔法ではなく経路設計」です。まずは“モデルに渡す中間生データ”を最小化し、 rubricでスクリプト選択条件（例：一括処理すべき分量やパターン）を運用ログで調整しましょう。
やること
中間JSONをモデル経由にしない設計（Code Mode）
測ること
入力トークン・実行失敗/リトライ・要約品質
Code Mode（コードモード / code execution）という英語用語には短い日本語補助として「コード実行」を付けると理解が揃います。
