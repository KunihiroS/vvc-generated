Sandboxは安心？
境界の正体
AIエージェントの安全対策は「サンドボックスがあるか」だけでは足りない。鍵は、隔離境界がどこで“信頼”として引き継がれるかだ。
SAIL 2.0 Framework（Pillar Research “Week of Sandbox Escapes”の論点整理）/ 日本語圧縮版
problem
03 / 16
プロセス隔離の落とし穴
失敗は後段で起きる
サンドボックス内で実行が制限されても、エージェントが書いたファイルや設定が後でIDE拡張・Git連携・タスク実行で“正当な成果物”として扱われると、実質的に境界が越えられる。
直接突破
隔離プロセスを破る攻撃（伝統的な想定）
process escape
信頼の突破
“境界外の作用”が、ホストの後段で発生（成果物を信頼）
trust handoff / trust escalation
メッセージ：境界はプロセスだけで定義できない。
metaphor
04 / 16
隔離はバトン
信頼が バトン
エージェントは“実行者”である以前に“作者”でもある。境界の本質は、作者が書いた成果物が次のコンポーネントに渡る瞬間の信頼にある。
i. 書く
コード/設定/フック/タスク定義を生成
artifact creation
ii. 解釈される
IDE拡張・Git・実行パイプラインが読む
interpretation
iii. 実行される
ホスト権限で動作（隔離の効果が薄れる）
host execution
結論：endpoint actor（エンドポイントの行為主体）として設計・監査する。
architecture
05 / 16
攻撃面の再定義
Trust Handoff
“サンドボックス境界”をプロセスではなく信頼の引き継ぎとして見ると、評価軸（何を・いつ・誰が信頼するか）が明確になる。
評価するもの
i) エージェントが書ける範囲
ii) 成果物を信頼するホスト構成要素
iii) 到達可能なローカルデーモン（例：Dockerソケット）
評価の目的
i) 承認（approval）をスキップする条件の検出
ii) ポリシーが副作用に効いているか検証
iii) テレメトリ（監視ログ）がどこで発火するか可視化
ポイント：ベンダー機能だけでなく、企業側のガバナンスで閉じる。
enumeration
06 / 16
失敗は4系統
収束する パターン
失敗パターンは4つに収束するとされる。どれも“プロセス隔離”より“信頼の引き継ぎ”で説明できる。
i. denylist失敗
OSの複雑さに追従できず、ブロック漏れが残る
denylist sandbox
ii. 設定が実行
ワークスペース設定ファイルが“実行コード”になる
config-as-exec
iii. allowlist過信
コマンド名だけでは、引数・副作用をモデル化できない
name-only allowlist
iv. 特権デーモン
サンドボックス外のローカルデーモンが信頼ブリッジになる
privileged local daemon
この4分類が、監査質問票の骨格になる。
distinction
07 / 16
denylist型の限界
ブロック漏れ
OSの機能・経路は膨大で、denylistは「常に1項目足りない」状態になりやすい。エージェントは組み合わせ探索で“想定外”を突く。
なぜ崩れる
i) 選択肢が多すぎる
ii) 隠れ経路（indirect path）が残る
iii) 追従コストが高い
OS complexity / indirect paths
監査の観点
i) denylist適用範囲の棚卸し
ii) “迂回”の到達可能性テスト
iii) 成果物→後段処理の信頼点を別途監査
trust boundary testing
メッセージ：denylistだけでは“境界の実在”を保証できない。
evidence
08 / 16
設定がコードになる
config-as-exec
ワークスペースの設定ファイルが、後でホストが“実行”するトリガになり得る。エージェントは破壊ではなく“設定の埋め込み”で成立する。
例：IDE
拡張・タスク連携が設定を解釈
IDE extension
例：Git
フック/連携が成果物を起点に動く
git hooks
例：タスク実行
タスクランナーが設定を“正規手順”で走らせる
task runner
メッセージ：隔離の外で起動される“仕組み”を評価する。
process
09 / 16
allowlistは呼び出し条件まで
nameだけでは
allowlistがコマンド名を許しても、引数・カレントディレクトリ・設定・フックで挙動と副作用は激変する。安全性は“完全な呼び出し条件”で判断すべきだ。
崩れる要因
i) 同名でも挙動が異なる
ii) 引数で機能が拡張される
iii) 作業ディレクトリ/設定が影響する
invocation context
実装に必要な粒度
i) 許可条件を引数レベルへ
ii) 副作用（file/net/exec）を制約
iii) 成果物→後段処理の鎖も含めて検証
side-effect model
メッセージ：allowlistは“呼び方”と“影響範囲”まで。
evidence
10 / 16
ローカルデーモンが橋になる
privileged daemon
サンドボックス外にある特権ローカルデーモン（例：Dockerソケット）が、エージェントの信頼ブリッジになり得る。隔離プロセスだけ守っても迂回が残る。
Docker
socket到達で実行環境が拡張
Docker socket
パッケージ
ローカルCLI/ビルドが成果物を取り込む
package managers
言語サーバ
解析・ビルド・補完が自動実行を誘発
language server
メッセージ：到達可能な“実行環境”を中心に監査する。
comparison
11 / 16
脅威モデルの違い
プロセス中心 → 引き継ぎ中心
旧来の見方は「隔離プロセスの突破」だが、SAIL 2.0は「ホストが書いた成果物をどれだけ信頼するか」を中心に置く。
従来：隔離突破
何が問題か：サンドボックスを破る方法
監査：侵入・脱獄（escape）への対策
SAIL：信頼突破
何が問題か：後段で“正当”として実行される連鎖
監査：信頼点、承認、テレメトリ、到達可能デーモン
メッセージ：安全性の評価対象が“連鎖”に移る。
process
12 / 16
企業が確認すべきこと
監査質問の核
ベンダーと自組織の両方で「信頼の引き継ぎ」を測る必要がある。少なくとも以下は問うべきだとされる。
監査：チェーン
i) エージェントの書き込み範囲は何か
ii) その成果物を誰が/どこで信頼するか
iii) 実行までのタイミングと条件は何か
監査：運用と証跡
i) 承認をスキップする条件はあるか
ii) ポリシーが副作用に効いているか
iii) テレメトリはいつ・どこで発火するか
メッセージ：隔離“有無”ではなく“連鎖の止め方/追跡”を設計する。
closing
13 / 16
まとめ
Trustを観測せよ
SAIL 2.0が示すのは、エージェントの脅威を「プロセスの制限」から「信頼の連鎖」に戻す必要性だ。4つの失敗パターンは、監査・可視化・ガバナンスの設計に直結する。
失敗1
denylist追従不能
denylist
失敗2
設定が実行へ
config-as-exec
失敗3/4
allowlist過信 / 特権デーモン
allowlist + privileged daemon
次の一歩：自組織のIDE/Git/タスク/ローカルデーモンの信頼点を棚卸しし、テストで境界の実在を確かめる。
