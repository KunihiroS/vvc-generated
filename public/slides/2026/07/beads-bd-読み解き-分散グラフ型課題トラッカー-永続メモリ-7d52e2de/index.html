永続メモリ × 依存グラフ
Beads（bd）
Beadsは、AIコーディングエージェントが長期タスクで文脈を失わないための「永続・依存関係つきメモリ」を、Dolt駆動の課題トラッカーとして提供します。
読み解く観点：source of truth／ready→claim→close／衝突回避ID／メモリ要約（compaction）／運用モード選択
problem
03 / 16
散逸する計画
TODOは壊れる
MarkdownのTODOや手作業の計画は散逸しやすく、複数エージェントが関わると整合性が崩れます。
現象
文脈の喪失／重複作業／進捗の不確かさ
原因
単純なテキスト更新では競合と依存が管理できない

真実はDBに置く
Source
Beadsの核心は「エージェントが参照・更新すべき情報の真実（source of truth）を、Dolt側に置く」ことです。
データ基盤
Dolt（SQL + Git的ブランチ/マージ/同期）
管理対象
タスク／依存関係／監査ログ／状態遷移
architecture
05 / 16
状態で衝突を減らす
ready→claim→close
Beadsはタスク進行を `ready → claim（引き当て）→ close（完了）` に寄せ、自由記述による混乱を抑えます。
bd ready bd update <id> --claim bd close
ポイント：エージェントが勝手に進捗を書き換えない運用規律（原子的な引き当てを前提）
enumeration
06 / 16
誰の作業か
Claim
claimは「誰がそのタスクを処理するのか」を機械的に固定し、複数エージェント更新時の重複を減らす狙いがあります。
原則 i
claimは引き当て（atomic）として扱う
原則 ii
closeで完了を確定し、再処理を避ける
注意：実運用では「claim残留」等の復旧手順の確認が必要（提示情報では詳細が省略）
distinction
07 / 16
依存で次が決まる
Graph
Beadsはタスクを依存関係グラフで結び、次に何をすべきかを「関係」から導けるようにします。
リンク
`bd dep add` で親子・ブロック・関連などを接続
利点
依存が可視化され、計画が散らばりにくい

08 / 16
衝突しにくい設計
Hash ID
ハッシュベースのID（例：`bd-a1b2`）が、マルチエージェント/マルチブランチでのマージ衝突を抑える設計思想です。
例
`bd-a1b2` のようなハッシュID
狙い
衝突・取り違えの確率を下げる
process
09 / 16
作業→記憶→縮約
Compaction
長期運用では過去タスクが増え、コンテキストが肥大化します。Beadsは閉じたタスクを要約して負担を抑える方針です。
閉じる
closeで完了を確定
要約
compaction / memory decay（記憶の減衰）
効果
エージェントの性能と継続性を維持
注意：要約で失われる情報範囲・復元可能性は導入前に要確認（提示情報では詳細が不足）
comparison
10 / 16
置き方で性格が変わる
Embedded vs Server
Embeddedはプロセス内でDoltを動かす（single writer）前提、Serverは外部SQLサーバに接続し複数書き手（concurrent writers）を想定します。
Embedded
単一書き手：まずは手元で運用しやすい
Server
複数書き手：同時更新を見据える
process
11 / 16
Gitで揃える / Git無しも
Sync
クロスマシン同期はgitリモート経由のプッシュ/プルで扱えます。さらに `BEADS_DIR` 指定と `bd init --stealth` によりGitフック等を無効化する「Git-Free Usage」が示されています。
同期
gitリモートで push/pull（クロスマシン）
回避
stealthモードでGit操作を無効化

13 / 16
要点まとめ
使いどころ
Beadsは「依存グラフ＋監査ログ＋ready→claim→close」により、エージェントが参照・更新すべき真実を保ちます。
強み
衝突回避（ID/状態遷移）と長期文脈維持（要約）
前提
運用規律（誰が何をいつ実行するか）と検証が成否を決める
補足：`.beads/issues.jsonl` はバックアップではなく、source of truthでもないため別のバックアップ手段（例：`bd backup`）を想定。
