Prompt cachingは“設計”だ

壊れやすい最適化を 設計要素へ

コーディングエージェント（LLMに指示・履歴・ツール定義・結果を毎ターン送る構成）では、Prompt Cachingは速度小技ではなく、 コストと体験を左右する設計上の重要要素になり得ます。

日本語メモ：Prompt Caching（プロンプトキャッシュ）＝prefill（事前計算）で作る内部状態（KVキャッシュ）を再利用し、再計算を減らす仕組み。

PROBLEM
03 / 16
効くはずなのに高い

キャッシュが読めない理由

似た会話でも、内部では“同じ前半トークン列”として扱われないとキャッシュ再利用できず、prefillが丸ごと再計算されます。

核心
“意味”ではなく token prefix 依存
症状
キャッシュミス → 再prefill → コスト増・遅延増
ARCHITECTURE
04 / 16
prefill / decode

内部状態の 再利用

LLMの計算は、前半入力をまとめて処理するprefillと、後半を1トークンずつ生成するdecodeに分かれます。

prefill
内部状態計算（KVキャッシュ）
decode
次トークン生成
Prompt Caching
prefill側のKVを再利用
METAPHOR
05 / 16
同じ“前半”だけ

キャッシュは 前置きに紐づく

Prompt Cachingが再利用するのは「完全一致の会話」ではなく、「前半トークン列（プレフィックス）」に対応する計算です。

成立条件
次リクエストが前回と同じ token prefix で開始
注意
セッションIDは補助に過ぎない
ENUMERATION
06 / 16
壊しやすい変更

プレフィックス不一致の 引き金

下のような変更はsystem prompt等へ畳み込まれ、序盤トークン列を別物にして早期にキャッシュを壊しやすくなります。

loadout
ツール追加/削除/スキーマ変更
順序
ツール順の入れ替え
分岐
/tree（分岐）やフォーク的操作
COMPARISON
07 / 16
“遅延ロード”の逆転

短期節約が 逆転する

ツールを遅延ロードする設計は通信量を減らせますが、ツール情報がsystem promptへ畳み込まれると、 その後ろが別プレフィックスになって再prefillが増えることがあります。

狙い
payload（送信）削減
落とし穴
prefill再計算が勝つ（総コスト増）
EVIDENCE
08 / 16
TTLと運用が効く

キャッシュは 時間と運用に負ける

TTL（保持期限）が短いと、待機時間だけで失効し、次のリクエストで過去履歴のprefillを再計算する必要が出ます。

TTL
数分で失効 → 次回が再計算
エビクション
保持枠から追い出し（Eviction）
ルーティング
同入力でも経路違いでミス
PROCESS
09 / 16
pruneを慎重に

途中削除は 境界を動かす

Piはprune（途中削除）をむやみに避けます。削除でプレフィックス境界が変わると、削除以降が再prefill対象になり損になり得るからです。

設計方針
安定した追記指向のトランスクリプト
必要時
compactionは“キャッシュリセット”として扱い、統計へ反映
ARCHITECTURE
10 / 16
一致は“複合条件”

セッションだけでは 決まらない

キャッシュ再利用は「セッションID」だけでなく、モデル/プロバイダ切替、履歴圧縮（compaction）、動的system prompt（タイムスタンプ/乱数）などで崩れます。

プロバイダ要因
モデル切替・保持期限・ルーティング
入力要因
compaction/書き換え・動的system prompt

R/W/CH

/session
状態の確認・要因追跡の入口

結論

最適化ではなく 設計で勝つ

Prompt Cachingは「効けば安い」だけでなく、キャッシュミスが設計・運用判断に跳ね返るため、 前半トークンプレフィックスの安定性と可視化をセットで扱うのが要点です。

要点
同じ会話ではなく“同じ前半トークン列”を作る
行動
prune抑制・ツール安定・TTL前提・R/W/CH等で検証
