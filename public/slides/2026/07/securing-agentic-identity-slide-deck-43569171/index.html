LLMエージェントの認証設計
Securing agentic identity
LLMエージェントが外部サービスへアクセスするには認証情報が必要です。目的は、エージェント環境が安全とは限らない前提で「漏れても致命にならない」構造を作ることです。
トークン流出 × プロキシ介在 × mTLS束縛 Reading deck
Problem
03 / 18
何が危険か
Token-at-Rest
エージェントがメールやカレンダー等へアクセスするには、JWT/opaque tokenのような認証情報が必要になります。ところがエージェント環境は完全には安全でなく、トークンがディスクに残ったり流出したりすると被害が大きくなります。
Failure mode
トークンが保存・混入・漏えい
Impact
機微情報への不正アクセス
設計目標：漏れても「価値がない」状態へ i
Distinction
04 / 18
素朴案の落とし穴
Placeholder Swap
「プレースホルダトークンを渡して、プロキシで本物に差し替える」発想はあります。が、その実装には“プレースホルダ↔実トークン対応表”のような秘密保管が必要になり、複雑化しがちです。
Idea
代理用トークン → プロキシで復元
Cost
対応表（秘密ストア）運用が必要
安全性だけでなく運用リスクも増える ii
Architecture
05 / 18
提案の核
Fake JWT
著者は、ブローカーが認証フローを仲介し、エージェントには“偽JWT”だけを渡す仕組みを提案します。偽JWTには実トークンを暗号化したクレームが含まれ、ブローカー署名があってもプロキシ以外では実質無価値にします。
Broker
認証を仲介し偽JWT発行
Agent
偽JWTを保持（本物は保持しない）
Proxy
署名検証＋復号して本物化
本物トークン：プロキシでのみ登場 iii
Process
06 / 18
どう動くか
Proxy Gate
エージェントはプロキシ経由でAPI要求し、プロキシが署名検証と復号を行って初めて本物トークンが外部APIへ渡ります。結果として「本物トークンがエージェント環境に存在しない」設計上のメリットが得られます。
Step i
エージェント：偽JWTで要求
Step ii
プロキシ：署名検証（限定）
Step iii
プロキシ：復号して外部APIへ送信
状態は暗号鍵中心（対応表の回避を強調） 1/3
Evidence
07 / 18
なぜ価値が落ちるか
Verification Trap
偽JWTは“ブローカーが署名したように見える”一方で、プロキシ側以外では署名検証・有効化ができないように設計されます。加えて実トークンは暗号化クレームとして保持され、プロキシ経由でしか本物になりません。
Agent viewpoint
外部API用の本物へ到達不能
Attacker viewpoint
盗んでも復号・検証の主導権がない
“無価値”を設計で作る iv
Threat
08 / 18
次の攻撃面
Internal Impersonation
攻撃者が組織内の別マシンを侵害し、エージェントに“トークンを使わせる”シナリオが残ります。そこで著者は、mTLSでエージェント実体を束縛し、トークンの流用を抑える考えを示します。
Scenario
別ホスト侵害 → エージェント誘導
Risk
偽JWTが正しい形で使われる可能性
エージェントは“内部”でも完全に安全ではない v
Architecture
09 / 18
証明書で縛る
mTLS Binding
SPIFFEのような発行基盤で、エージェント環境にmTLSクライアント証明書がある前提を置きます。トークン発行時に「その証明書の表現」をクレームに埋め込み、プロキシはmTLSで提示された証明書と照合します。
Claim
証明書の表現（クレーム化）
mTLS
クライアント証明書を提示
Proxy Check
一致しなければ拒否
“正しいインスタンス以外”は復号できない方向へ vi
Distinction
10 / 18
流用耐性の条件
Key Confinement
証明書の秘密鍵がハードウェア/ハイパーバイザに束縛され、インスタンス外に持ち出しにくいなら、トークンの流用耐性が高まると結論づけています。ここでのボトルネックは、鍵管理とライフサイクルです。
Stronger when
秘密鍵が外部に出にくい
Depends on
証明書の発行・失効・ローテーション
万能ではなく“鍵管理の質”に依存 vii
Comparison
11 / 18
標準との距離
RFC 8705 Gap
著者はRFC 8705のような標準バインディングが採用できないIdPでも、ブローカーが別のJWTへ暗号化クレームを内包し、mTLSで制約すれば成立するとしています。一方で、第三者ベンダー相互運用では手間が増えます。
Standard path
IdPがRFC 8705等をサポート
Fallback path
偽JWT＋暗号化クレーム＋mTLS照合
面倒だが“不可能ではない” viii
Distinction
12 / 18
限定事項
Operational Friction
mTLSでの理想的な自己生成・相互継承が現実的でないケースや、ブラウザ対応などの制約が残ります。導入にはベンダー/IdPと環境の統合が必要で、第三者にはやや面倒です。
Integration
IdP・ブローカー・プロキシの連携
Runtime
mTLS証明書の取得と提示
Edge cases
ブラウザ等での制約
理想形は難しい → 現実解へ折り合い ix
Closing
13 / 18
評価と着地点
Design Wins
本提案の強みは、トークン流出を前提に被害を局所化しつつ、「本物トークン非保持」「プロキシでの復号・差し替え」「mTLS束縛」で段階的に下げる点です。成功の鍵は、プロキシ防御・鍵管理・証明書ライフサイクル・監査ログです。
High-value target
ブローカー/プロキシの堅牢化は必須
Operational guardrails
鍵配布・失効・ローテ・検知/停止
脅威モデル（外部漏えい・内部侵害・リプレイ等）で検証 x
