“遅い”より“ズレる”
Software Disfit
典型的なフルスタックでは、DB〜ブラウザ間の変換（SQL→Go→JSON→HTTP→TS→React）で無駄な計算と不整合が増えやすく、AIの生成も“境界が曖昧”だと崩れます。
翻訳コスト
CPU/バグ/整形
契約不足
境界が定義されない
=> API幻覚
検証不足
決定性の欠如
=> 品質退行
用語：摩擦=変換層の増殖＋AI生成の境界不一致＋退行の見逃し。

契約は“翻訳免許”
Single-Contract
Foundationは、変異（変更加筆）やスキーマを“1つの契約”として定義し、Go・TypeScript・ゼロコピーのバイナリ表現へ生成展開します（Protobuf/Cap’n Proto）。
契約（contract）
データ形の一貫定義
protobuf / capnp
生成（生成展開）
Goルート／TS型／バイナリレイアウト
=> 変換層の増殖抑制
効果の方向性：型不整合・変換バグを減らし、AIが参照すべきAPIを明確化。
英語+JP: contract（契約）

DBを“読む形”へ投影
Hermes Hotplane
Hermes Hotplaneは、PostgreSQLの更新を実時間でローカルに投影し、マイクロ秒領域の読み取りモデルを提供することで高速化を狙います。
ベンチマーク主張
述語フィルタで
列挙・ビットマップマージ ~34µs
通常パスとの差
通常 ~7.8ms に対し
229× 高速化
注意：測定条件（データ規模・環境・述語範囲）が不明なため、再現性確認が必要。
日本語：投影（projection）

07 / 16
キャッシュじゃない
Projection（投影）
README要約では、Hermesは単なるキャッシュではなく“投影（projection）”として、DB変更の伝播と整合を前提にした読み取りモデルである点が強調されています。
キャッシュ
ヒット率中心
更新は場当たりになりがち
Hermes（投影）
DBの変更→ノードローカルへ
整合性と更新伝播を設計前提
ただし、この入力だけでは遅延・順序保証・障害時復旧・再同期戦略は特定できません。
