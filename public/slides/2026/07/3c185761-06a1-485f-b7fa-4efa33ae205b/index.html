CI/CD REVIEW
02 / 14
能力で見る

capability diff 第一級

CIの差分が小さく見えても、実は権限境界が動くことがある。 そのためレビュー成果物を「能力（capability）の差分」にするべき、という提案です。
Bilingual: capability diff（能力差分） / authority diff（権限差分）
PROBLEM
03 / 14
デプロイギャップ
Greenが 安全 ではない
「テストが通る（green build）」は挙動の確認であって、「誰がリリースできるか」を証明しません。 CI/CDは実行権限（署名・成果物公開・リリース）も含むため、信頼境界が静かに移動します。
Line diff
変更点は小さく見える
実態
信頼境界（blast radius）が増える

レビューすべき単位は line diff ではなく、capability/authority diffです。 トークンのスコープ、アクションの同一性（SHAか浮動か）、到達可能な秘密、リリース境界を前後比較します。

PRIVILEGE CREEP
05 / 14
小変更でも
権限が増える
たとえばピン留めされた GitHub Actions を固定タグから浮動タグへ変えると、実行されるコードの出自が変わります。 さらに write トークンやシークレット到達が増えると、成果物公開レーンの支配が変わります。
Action identity
pinned SHA → floating ref
Token scope
read ↔ write / deploy ↔ release
FAIL CLOSED
06 / 14
未知は危険
UNKNOWN→ブロック
capability diff を計算できない、または差分が不明なら「承認してしまう」のが最大の事故要因です。 投稿は UNKNOWN をブロックし、見えない拡大を許さない姿勢を強調します。
計算可能
機械が差分を検出
差分あり
ブロック or 承認条件化
UNKNOWN
デフォルトで拒否
STATIC + RUNTIME
07 / 14
計算だけで
抜ける
静的ゲート（事前宣言の比較）だけでは、依存の再解決やトランジティブ到達、実行グラフのズレを完全には塞げません。 だから runtime mirror（実行時観測）で宣言と実行結果の乖離も捕捉すべきです。
Static gate
宣言（capability）を差分化
Runtime mirror
実行（observed capability）を照合
事後監査は安心材料：主目的は実行前の fail closed
GOVERNANCE
08 / 14
承認も
能力で設計
ブレークグラス権限は必要でも、復旧が新しいデプロイ権限になれば別の事故です。 承認は儀式ではなく、revocableで期限付きの「能力証跡」として機械が照合できる形にします。
Break-glass
必要：ただし最小化
Revocation
取り消し可能
Expiry
期限付き委任
OPEN GAP
09 / 14
課題は
表現の標準化
capability グラフを「確実に表現・計算」する標準が弱いと、差分比較の精度が落ちます。 また、reusable workflow、外部連携、トランジティブ、実行時分岐が絡む現実では、完全事前計算は難しい場合があります。
Need
差分の定義の統一
能力（capability）のモデル
Reality
計算不能が残る
UNKNOWNが増える
MY VIEW
10 / 14
賛成だが
段階導入
中心メッセージ（capability/authority diffをゲートの第一級成果物に）は賛成です。 ただし全面 fail closed を即適用するより、最も危険な変更クラス（浮動化、write追加、release lane所有変更）から始めて運用負荷と計算可能性を検証するのが現実的だと考えます。
Step 1
最危険クラスを限定
Step 2
差分モデルと統合を拡張
UNKNOWNが増えすぎると「例外だらけ」or 「結局人が通す」へ戻る
CLOSING
11 / 14
結論
統治を diff へ
green tests は必要条件でも十分条件ではない。 できることの差分（capability/authority diff）をブロッキング入力にし、計算不能は UNKNOWN→拒否、さらに runtime mirrorで乖離も捕捉する—これが「予防文化」への移行です。
Rule
能力差分をレビュー第一級に
Guardrail
UNKNOWN→ブロック
Reality check
runtime mirrorで照合
