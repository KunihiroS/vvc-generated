COVER
02 / 18

AIのための“生きたWiki”

CodeAlmanac

リポジトリ内に、AIコーディングエージェントが参照できる背景知識Wikiを作り、更新までを（基本は）ローカルで完結させます。

PROBLEM
03 / 18

コードの外側

背景が残らない

コードだけでは「なぜその設計か」「過去に何が壊れたか」「守るべき不変条件」が伝わりにくく、AIが文脈を取り落とします。

失われやすい
設計意図 / 意思決定 / 失敗履歴
難しい
不変条件（invariants） / 横断ワークフロー
METAPHOR
04 / 18

Gitの相棒

Wikiをコード化

知識を almanac/ 配下のMarkdownとして管理し、変更をレビュー可能な“運用・意思決定の資産”にします。

保管場所
almanac/（README / topics / decisions など）
価値
AI参照 + チーム合意形成
ARCHITECTURE
05 / 18

Read / Write分離

権限を分ける

検索・閲覧（read）は認証なしで利用でき、更新（write）は認証されたライフサイクル実行（Codex/Claude等）が必要です。

read（閲覧）
search / show / topics / health / validate 等
write（更新）
認証付きハーネス + 実行はガードレールあり
ARCHITECTURE
06 / 18

ローカルのみ

クラウドしない

ホストされたサービスへの接続・アップロード・公開SDK/MCPは契約範囲外で、現状はローカル完結が強調されています。

契約
クラウド連携なし
データ統制
所在が把握しやすい
注意
共有は運用（sync等）次第
PROCESS
07 / 18

取り込み：ingest

素材を統合

ingest はファイル/ディレクトリ/Git差分/PR・issue/URL/エージェントのトランスクリプト等をWikiへ統合します。

入力例
ファイル / Git差分 / PR・issue / URL
出力
背景知識として almanac/ に反映
PROCESS
08 / 18

品質改善：garden

弱点を直す

garden は既存Wikiグラフを更新し、重複や弱いリンク、根拠の弱い主張などを改善することを目指します。

対象
既存グラフ / ページ整合
狙い
重複削減 / 根拠の強化 / 接続の整理
DISTINCTION
09 / 18

“生成”ではない

維持のワークフロー

ingest は統合、garden は保守。どちらも「新規生成」より、既存知識のメンテナンス方針に寄る設計です。

ingest（取り込み）
新しい素材をWikiへ
garden（整形）
壊れやすい点を直して維持
EVIDENCE
10 / 18

守るべき不変条件

編集はalmanac/限定

ライフサイクル実行による更新でも、編集可能領域を almanac/ 配下のソース編集に限定し、誤改変リスクを下げます。

ガードレール
Wiki更新＝ almanac/ のみ
狙い
コード本体・機密領域の改変回避
ENUMERATION
11 / 18

コマンド体系

読む道具がある

Daily Read Surface では、同じコマンド体系で人間・エージェントがWikiを検索・閲覧できます。

検索 / 表示
search / show
話題
topics
状態
health / validate
EVIDENCE
12 / 18

実行ログ：jobs

追跡を残す

実行履歴は ~/.codealmanac/ に jobs として記録され、ソース解決の事実・ハーネスイベント・安全エラー等を含みます。

保存先
~/.codealmanac/
ログに含む
ソース解決 / イベント / 安全エラー / 端末状態
PROCESS
13 / 18

導入：PyPIへ

Python前提運用

Python 3.12+ を前提に PyPI版CLIを導入し、従来の npm CLI は退役（移行が推奨）です。

前提
Python 3.12+
移行要点
古いグローバル実行物/エージェント設定を削除→PyPI CLIへ
COMPARISON
14 / 18

良い点と注意点

強みと落とし穴

意思決定の型（decisions/architecture/guides）や編集制限、jobsログは強み。一方で、品質評価基準や誤更新検知/ロールバックの詳細は別途確認が必要です。

強み
意思決定の“型”/ ingest・garden分離/ 編集領域限定/ 追跡可能
注意
gardenの判定基準 / 誤更新時の扱い / 環境差 / 運用合意
CLOSING
15 / 18

結論

コード資産を運用へ

CodeAlmanacは、リポジトリのコードをAIが扱える「運用・意思決定の資産」へ拡張します。現状はローカルのみという制約が、統制と実装方針を明確にしています。

不変の軸
almanac/ で背景を管理
運用の鍵
read/write分離 + ログ読解 + レビュー体制
