読める防御の設計思想

Context Bombing

プロンプトインジェクション（Prompt Injection）を「検知」ではなく「拒否が固定される形」で止める発想です。文脈爆撃（context bombing / 文脈爆撃）では、攻撃者が参照しがちな場所に“危険に見える仕掛け”を置きます。
攻撃（要点 / 攻撃の狙い）
LLMへ本来させない指示を混ぜる（例：機密の引き出し、危険手順）
防御（要点 / 防御の狙い）
参照した瞬間にガードレール拒否を誘発し、進行を止める
PROBLEM
03 / 16

なぜ今、止めが必要か

Injection 上書き

攻撃（プロンプトインジェクション）は、文章の中に“従わせたい命令”を混ぜて、LLMの本来の指示系をすり替えます。防御が「危険を見つけて警告するだけ」だと、攻撃が先に進む余地が残ります。
典型例（メール）
カレンダー招待・メール本文に命令を混入
典型例（探索）
ネットワーク内でAI支援型の解析を誘導
課題（警告）
検知後も昇格・侵入の完了が間に合う
IDEA
04 / 16

拒否を“勝ち筋”にする

Refusal 固定

context bombingは、ガードレールの“拒否”を攻撃停止のメカニズムとして転用します。攻撃LLMエージェントが参照した文脈で禁止に触れると、拒否モードに入り、その後の進行が難しくなります。
従来（単なる検知）
危険を見つけても、攻撃はそのまま継続しうる
本手法（停止寄り）
拒否が続き、攻撃の“完遂”まで届きにくくする
MECHANISM
05 / 16

デコイで探索を誘導

Decoy 囮

防御側は、AWSの秘密鍵やパスワード“っぽい”文脈にデコイ（囮）文字列を配置します。攻撃者のエージェントが資源を列挙・参照する流れで、その“仕掛け”に遭遇しやすくするのが狙いです。
トリガ場所
秘密鍵 / パスワード周辺の文脈
仕掛け内容
禁止に触れる“危険な命令”の混入
結果
ガードレール拒否が発火・継続
EVIDENCE
06 / 16

複数モデルで統計的に低下

152 runs

5種類のモデル（Opus 4.8、Gemini 3.1 Pro、GLM 5.2、DeepSeek 4 Pro、Kimi 2.6）で、152回の攻撃シミュレーションを評価したと報告されています。デコイ文字列を1つ混入するだけで、管理者権限奪取や持続侵入の成功率が大幅に低下しました。
評価対象（Models）
Opus 4.8 / Gemini 3.1 Pro / GLM 5.2 / DeepSeek 4 Pro / Kimi 2.6
実験規模
5モデル × 152攻撃シミュレーション
RESULT
07 / 16

最強クラスでも崩れる

93% → 0%

特にOpus 4.8では、管理者アクセス達成が93%から0%まで落ちたと報告されています。最上位モデルですら“拒否が進行を止める”効果が出た点が強い主張になります。
Before（なし）
管理者アクセス達成：93%
After（デコイ混入）
管理者アクセス達成：0%

検知だけだと遅い

Minutes ぎりぎり

記事では、初期検知が攻撃開始から8分、昇格まで平均14分という“時間の余裕”がある状況が言及されています。canary（カナリア）系は早期警告に寄る一方、context bombingは進行そのものの停止を狙う位置づけです。
初期検知
平均 8分
昇格（成功の分岐）
平均 14分
狙い（本手法）
拒否で“完遂”まで到達させない
COMPARE
09 / 16

検知 vs 停止

Canary → Bombing

従来のcanary（炭鉱のカナリアのような罠資源）は“早期検知”に主眼がありました。context bombingは、それに拒否誘発を組み合わせ、攻撃プロセスの進行確率を下げる設計です。
canary（検知）
危険の兆候を早めに察知
context bombing（停止）
拒否を踏ませて前進を阻む
FAQ
10 / 16

“置くだけ”ではない

Context トリガ

単に危険文を置いてブロックするのではなく、「攻撃者が参照してしまう文脈」に禁止に触れる命令を埋め込みます。攻撃LLMエージェントが探索・列挙の流れで仕掛けに遭遇すると、拒否が継続して“以後の指示”へ進みにくくなる点が重要です。
ポイント i
参照される“文脈”設計
ポイント ii
禁止に触れて拒否発火
ポイント iii
進行（完遂）を阻害
RISKS
11 / 16

万能ではない

Limitations 注意

記事上、根本的な解決策はないとされます。また、評価は特定のモデル・特定のテスト環境（シミュレートAWS）での結果です。攻撃者が仕掛けを見抜いて回避する、別環境で効果が落ちる、誤って正当作業まで拒否が増える、といった追加検証が必要になります。
回避（適応）
デコイ学習・迂回の可能性
副作用（運用）
誤拒否（正当エージェント停止）のリスク
TAKEAWAY
12 / 16

導入前に見るべき指標

Deploy チェック

防御としては「攻撃成功率を統計的に押し下げられるか」を最優先で評価します。同時に、拒否が“正当な運用”に波及しないよう、ログ・監査と範囲制御の設計が不可欠です。
成功率（効果）
管理者奪取・持続侵入がどれだけ下がるか
誤拒否（影響）
正当タスクまで停止しないか
監査（運用）
検証ログと説明可能性の確保
