企業向け運用基盤

Orchestration

複数のAIエージェントが協調し、タスクを実行→学びをメモリに反映→次の仕事で改善する“仕組み”を提供します。

Lead/Worker × Docker隔離 × 共有メモリ × HITL
Problem
03 / 16
単発チャットの限界

続かない

チャットで解決しても、組織の業務フローに組み込まれず、品質・安全・再現性の担保や学習の蓄積が難しくなります。

品質
誤った実行を止める設計が不足しがち
再現性
環境差で挙動が変わる（隔離不足）
継続性
学びが検索・再利用できない（メモリ不足）
運用
監視・評価・コスト可視化が後回しになりやすい
Core Idea
04 / 16
実行と記憶の循環

Compounding

過去セッションの知見を“共有メモリ”に抽出し、次のタスクで改善に使うことで、知性を複利的に厚くすることを狙います。

入力
Slack / GitHub / Jira / email 等
実行
Leadが分解→WorkerがDocker隔離で実行
学習
成果と学びをメモリへ（検索・再利用）
Architecture
05 / 16
役割分担と隔離

Lead/Worker

Leadは計画・委譲を行い、WorkerはDockerの隔離環境で実行します。これにより安全性と再現性を高めます。

Lead
計画 / 分解 / 委譲（例：Slackやチケットをタスク化）
Worker
Docker隔離実行 / 解決策生成 / アーティファクト作成
隔離により依存関係や実行環境のばらつきを抑え、検証可能性も上がります。
Memory
06 / 16
永続的アイデンティティ

SharedMemory

メモリは“ログ”ではなく、過去セッションの学びを抽出して検索可能な形で保持し、次の成果に効く形で統合します。

検索
ベクトル＋全文検索（リンクされたメモリ展開も）
継続性
人格・専門性・メモが進化し、参照精度を改善
Safety
07 / 16
失敗を制御する

HITL & Judgement

不可逆な工程は停止して人間の承認（HITL）を挟み、さらにLLM-as-judgeで品質ゲートを設ける設計が示されています。

Human-in-the-Loop
重要操作は承認ゲートで事故リスクを低減
LLM-as-judge
評価用（ただし過信は禁物で設計が鍵）
注意：誤判定率や運用基準はREADME抜粋からは定量化できません。
Reliability
08 / 16
止める・戻す・連鎖させる

DrainLoops

Litmus Tests、Drain Loops、No-opなどのパターンで、信頼性と運用の安定性を高める考え方が前面に出ています。

Litmus Tests
LLM-as-judgeで品質を判定（LLMの評価軸）
Drain Loops
チケット1件→複数レビュー可能なPRへ連鎖
No-op
変更がなければ何もしない（無駄な操作を抑制）

OTel & Eval

evalハーネスでシナリオ×設定マトリクスを回し、OpenTelemetry（OTLP）でコスト等も含めて可観測性を確保します。

Evalハーネス
トランスクリプト/アーティファクトを記録し判定
可観測性
OTLPでコスト・挙動を追跡（ダッシュボード想定）
目標は“ブラックボックス化”を避け、運用の改善サイクルを回すことです。
Take
12 / 16
強みと未確定点

Pragmatic

強みは統制（HITL/品質ゲート）・継続性（メモリ）・運用（可観測性/評価）を同時に押さえる点です。一方、README抜粋だけではガバナンスやセキュリティ境界が不明です。

強み
Docker隔離 / DAGワークフロー / mem・eval・OTLP
未確定
メモリ更新のガバナンス、LLM判定の精度、RBAC境界、コスト/遅延
結論：小さく始め、重要操作は人手承認を厚くし、段階的に拡大するのが安全です。

Next Steps

i

ii
メモリ更新の検証/取り消しルール（ガバナンス）
iii
権限最小化と監査（HITL/RBAC/データ取り扱い）
