Goで“動く”基盤へ

go-micro 概要

go-micro は、AI エージェントを“実世界の処理”へ接続する 実行基盤（ハーネス） と、通常の マイクロサービス を同じランタイムとして提供するフレームワークです。
Agent（エージェント）
LLMが計画・実行し、ツール呼び出しで処理を連携する仕組み
Service（サービス）
RPC/REST などのエンドポイントを提供し、ツールとして呼べる部品

“回答”から“実行”へ

ギャップ 課題

LLM はテキストを出せますが、業務処理（外部API、DB、他システム）を 安全に実行 し、運用 まで繋げるのが別問題になります。
接続
ツール/サービス端点をどう“統一”して呼ぶか
継続
長時間処理の再開・障害復旧をどう扱うか
安全
暴走・誤実行・権限過多をどう抑えるか

04 / 16
ツール＝端点

翻訳 モデル

go-micro の核は、マイクロサービスのエンドポイントを ツール（AI呼び出し可能） に見立てる発想です。
統一の流れ
サービスを実装（型・ハンドラ） → エンドポイント情報がツールスキーマ化 → LLM がツール呼び出しとして実行
相互接続
エージェントもサービスとして登録・発見・呼び出し可能 → 分散システムとして組み上げる

二層 全体像

go-micro は「エージェント（推論/実行）」と「サービス（RPC/RESTの実処理）」を同じ基盤に載せ、デプロイ運用まで視野に入れています。
Agent側
LLM / 記憶（メモリ）/ ツール / ガードレール（安全制御）/ 他エージェント連携
Service側
エンドポイント（公開機能）/ レジストリ/ 呼び出し（ロードバランス含む）

06 / 16
中核機能

Plan & Delegate 実行分解

エージェントの実行を、plan（手順化） と delegate（委譲）、さらに durable なワークフロー で分担します。
i / plan
複数ステップを整列し、記憶に残す
ii / delegate
サブタスクを他エージェントへ渡す
iii / durable workflow
checkpointed で再開可能な、イベント駆動の手順

安全制御は“実行箇所”に置かれる、と README 記載から読み取れます。これによりループ抑止や人間承認を組み込みやすくなります。

決まった手順 vs 判断

役割分担 比較

durable なワークフローは「順序立てた確実な処理」を担い、エージェントは「計画・委譲・ツール呼び出しによる意思決定」を担う、という整理ができます。
ワークフロー（durable）
イベント駆動・順序手順・checkpointed再開
エージェント
plan/delegate/ツール実行の意思決定

MCP / A2A 連携

外部連携はゲートウェイとして統合され、MCP でツール公開、A2A でエージェント公開を扱う方針が示されています。
MCP（ツール公開）
外部へ“ツール”として提供し、呼び出しを成立させる
A2A（エージェント公開）
外部へ“エージェント”として提供し、連携を成立させる

12 / 16
強みと不足点

導入判断 要注意

強みは「エージェントを分散システムとして運用する」思想と、ツール/サービス統一・durable・外部ゲートウェイ統合です。一方で、失敗復旧や安全の具体設定は抜粋だけでは確認できません。
強み
ツール＝サービス端点、エージェントもサービス化、durable再開、MCP/A2A/x402の拡張方向
不足（抜粋の限界）
失敗時の整合性/再試行/順序/バックプレッシャ、LLM非決定性の吸収、脅威モデルの前提が不明
