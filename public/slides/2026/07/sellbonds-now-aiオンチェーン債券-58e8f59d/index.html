AIの資金調達を自律化

sellbonds.now

AIエージェントがオンチェーン債券（ボンド）を発行し、USDC等で集めた資金を利息付きで返済する世界観を、仕組み中心に整理します。
オンチェーン債券 / On-chain bond（オンチェーン債券）
ローカル署名 / Local signing（ローカル署名）
非カストディアル / Non-custodial（非保管）
本資料は、提示されたレポート抜粋（sellbonds.nowの説明・画面情報の記述）をもとに要点化しています。
Problem
03 / 16
なぜ“AI×債券”なのか

仲介不要の資金調達

AIが資金を集めるには、通常は口座・仲介・登録（KYC）・契約運用が障壁になります。sellbonds.nowはそれを“チェーン上の契約”に寄せて解く狙いです。
従来の摩擦
口座運用 / KYC / API連携 / 監査の手間
狙う代替
スマートコントラクト債券 / 透明な履歴 / 直接送金
Metaphor
04 / 16
債券＝売買できる“契約”

ERC-20で契約化

sellbonds.nowの“ボンド”はERC-20トークンとして表現され、発行・保有・利払いがスマートコントラクトに記録されます。
発行
AIエージェントが債務を登録 / Issuance（発行）
購入
投資家がUSDC等で拠出 / Funding（資金拠出）
返済
時間とともに利息・返済を分配 / Repayment（返済）
Architecture
05 / 16
口座の代わりはウォレット

非カストディアル設計

説明では、アカウント登録・APIキー・メール・KYCなしで、ウォレット（公開鍵）が本人（発行者/投資家）に相当します。
ユーザー側
ウォレット管理＋ローカル署名（例：CLI/SDK）
サービス側
資金を凍結できない前提 / 取引はチェーンへ直接送信
ただし安全性は、ウォレット管理ミスや署名権限の委譲の影響を受けます。
Architecture
06 / 16
債券は“市場”単位で独立

ボンドごとに条件分離

各ボンドは個別の市場契約として存在し、金利や返済スケジュールがオンチェーンで追跡できます。
金利
APR（年率）がボンドごとに異なる
スケジュール
返済・利払いの条件を契約に内包
投資判断
“発行体全体”より“その債券条件”で比較しやすい
Evidence
07 / 16
リアルタイム追跡

レジストリ直読運用

Base mainnetのレジストリから、発行・購入・アクティビティの状況がリアルタイム更新として見えるとされています。
観測できること
RECENT ACTIVITYのような履歴 / 発行が複数稼働
何が難しいか
テスト債券の混在の可能性を区別する必要
透明性が高い一方で、見えるデータの意味づけは利用者側の確認が要ります。
Distinction
08 / 16
担保なしで市場化

非担保×ペナルティAPR

非担保（担保なし）を前提に、返済遅延が起きればペナルティAPRや評判（reputation）が可視化され、価格付けに反映される設計です。
投資家が信じるもの
返済履歴 / 遅延時の挙動 / 評判の可視化
前提（自己責任）
全自動の安全性保証ではなく、リスク評価が必要
Process
09 / 16
署名→送信→完了

取引はCLI/SDKで直接

CLI/SDK（例：sellbondsの“sbn”コマンド）がローカルで署名し、チェーンへ直接送信する設計で、サーバー連携のAPIは不要と説明されています。
1
エージェントがCLI/SDK実行
2
ローカル署名（秘密鍵は環境側）
3
チェーンへ送信（取引が記録）
安全性は、エージェント実行環境（権限・秘密情報・送信前検証）に左右されます。
Comparison
10 / 16
Webサービスと何が違う

仲介なしで監査可能

sellbonds.nowの主張は「仲介者なし」「手続きはローカル署名中心」「監査可能な契約としてオンチェーン可視化」へ寄せることです。
sellbonds.now
ボンド＝スマートコントラクト / 履歴はオンチェーン / 手数料は主にガス代
一般的な運用
口座・管理者・書類・非公開データの比率が増えがち
プラットフォーム手数料なしとされ、コストは主にガス代（テストでは補給）と説明されています。
Distinction
11 / 16
テスト債券の“見分け”

混在表示の解釈

表示されるボンドには「テスト目的」などの注記が含まれ得るため、実際に貸しているのか（未実資なのか等）を契約情報で確認する必要があります。
起きうること
“Claude Code経由のテスト”のような注記 / 未実資のため実際に貸さない
利用者の確認
ボンドごとの注記・契約情報・用途（発行の性質）
Evidence
12 / 16
筆者の評価：良さと不足

前向きだが要確認

強みは「オンチェーンで評価可能」「非カストディアル」「CLI/SDKで実務に落ちる」こと。一方で、リスクの決定情報が抜粋だけでは不足しています。
不足しがちな論点
評判（reputation）算定の具体 / デフォルト時の回収・清算 / テスト識別の明確化
特に重要
ウォレット管理と署名手順 / AIの権限・誤実行ガードレール
closing
