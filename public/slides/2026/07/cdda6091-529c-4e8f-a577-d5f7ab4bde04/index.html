CODEALMANAC
02 / 16
AIの知識を、リポジトリへ。
CodeAlmanac 概要
CodeAlmanacは、リポジトリ内のMarkdownウィキを“AIエージェント用の知識ベース”として維持するローカル中心の仕組みです。 追加するだけでなく、ingest / garden / buildのライフサイクルで劣化や重複を“メンテ”します。
Bilingual / 二言語
知識基盤 Knowledge Base
設計の芯
検索ではなく 更新されるウィキ
PROBLEM
03 / 16
知識は散逸する。
散逸問題
AIの学び（会話ログ・PR/issue・差分・URLなど）は、そのままだと“どこにも根づかず”劣化します。 CodeAlmanacは、根ざす先をリポジトリのMarkdownウィキに固定し、構造として改善します。
入力（素材）
会話ログ / PR / issue / diff / URL
狙い（状態）
古い情報・重複・弱いリンクを改善
REPO OWNERSHIP
04 / 16
“所有”を明確に。
almanac/がウィキ
`almanac/` 配下のMarkdownがウィキ本体で、コミットされる前提なので“保存場所”が明確です。 クラウドのブラックボックスではなく、リポジトリの履歴で追えます。
保存先
almanac/（Markdownウィキ）
監査の入口
Git差分レビュー前提
LIFECYCLE
05 / 16
ジョブで考える。
ingest / garden / build
CodeAlmanacの中心は、単発の追記ではなくライフサイクルの“ジョブ化”です。 ingestは取り込み、gardenはグラフを整え、buildは成果を構築します。
i ingest / 取り込み
差分・PR/issue・URL等を選別して折り込む
ii garden / 整形
重複・弱いリンク・トピック整合を修正
iii build / 構築
ウィキ成果物を組み立てる（流れの一部）
設計上の価値
劣化メンテまで前提にする
AUTOMATION
06 / 16
クラウドではなく、launchd。
ローカル自動運用
自動運用（sync / garden / update等）はクラウド連携ではなく、macOSのlaunchdローカルジョブで回ります。 実行状態や履歴は `~/.codealmanac/` に保持されます。
実行基盤
macOS launchd
データ保管（ローカル）
~/.codealmanac/
スケジュール例
5時間・24時間等
CLI導線
sync / garden / update
TRUST BOUNDARY
07 / 16
OSサンドボックスではない。
信頼境界
ライフサイクル実行はローカルで動くエージェントで、`almanac/`は“命令とコミット方針”の境界として扱われます。 OSサンドボックスとしての隔離を期待できないため、trust modelを受け入れるリポジトリのみで実行すべきです。
境界の意味
almanac/は隔離ではない
前提
Git差分のレビュー体制
TELEMETRY
08 / 16
送るのは“成否”だけ。
テレメトリ抑制
テレメトリはコード・パス・プロンプト・会話などを送らず、匿名の成否やサニタイズされた未処理クラッシュ概要に限定されます。 それでもゼロでない可能性があるため、必要なら無効化します。
送らないもの
コード / パス / プロンプト / 会話
送る可能性
匿名の成否 / クラッシュ概要
無効化
--no-telemetry や設定
前提
組織ポリシーに合わせる
AUTO COMMIT
09 / 16
自動コミットは慎重に。
auto_commit
デフォルトでは auto_commit = true ですが、設定で変更可能です。 さらにCLIが“分割してステージ”や“内部でコミット粒度調整”を自動で行うわけではないため、差分レビュー前提がより重要になります。
デフォルト
auto_commit = true
注意点
ステージ分割・粒度調整は自動ではない
OPERATIONS
10 / 16
“見える化”がある。
jobs / logs
実行中のジョブは `codealmanac jobs` で確認でき、要約・ログ・追跡（attach）まで用意されています。 実体は `~/.codealmanac/` 配下のローカル記録です。
一覧
codealmanac jobs
要約
jobs show <run-id>
ログ
jobs logs <run-id>
追跡
jobs attach <run-id>
EVALUATION
11 / 16
品質は運用で作る。
実務上の評価軸
良い点は、ジョブ履歴・ログ・差分で“当たり外れ”を検証し、運用で制御できることです。 一方、gardenの誤修正可能性や品質評価指標、更新スキップ条件の詳細などは提示情報だけでは不明です。
Evaluable（評価）
差分・履歴で検証可能
Risk（不明領域）
誤修正率・評価指標・スキップ条件が不明
OPINION
12 / 16
“土台”として有望。
総合評価
CodeAlmanacは、AIの知識散逸を抑え、リポジトリの不変条件を蓄積・保守する土台として筋が良いです。 ただし信頼境界がサンドボックスでない以上、reviewが省略されると危険性が増えるため、慎重な導入が必要です。
強み
ingest/gardenをジョブ分離し、追跡しやすい
最大論点
trust境界：OS隔離ではない
