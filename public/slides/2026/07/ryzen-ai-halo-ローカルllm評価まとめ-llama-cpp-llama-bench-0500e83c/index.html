llm-metrics
04 / 16
pp と tg
速度の分解
LLM推論は、入力を処理して状態を作る prefill（pp）と、出力トークンを1個ずつ生成する decode / token generation（tg）に分かれる。
pp（prefill）
プロンプト読取→計算で状態作成（compute寄り）
tg（token gen）
1トークンずつ生成（一般にメモリ帯域寄り）
速度はコンテキスト長やメモリ帯域の影響を強く受け、GPU/NPUの実行基盤（ROCm/HIP・Vulkan等）も結果を左右する。

bottleneck
06 / 16
帯域が主役
decode はメモリ帯域
結論として、prefillよりも decode（tg）がメモリ帯域依存で、帯域の大きい Mac Studio が全体的に上回る傾向がある。
支配要因
decode（tg）＝メモリ帯域の影響が大きい
観測
特定モデルでは差が 2〜3倍 に広がることがある（例：Gemma 4）
Halo（256GB/s）では、用途・モデル次第で伸びにくい可能性が示唆される。
backend
07 / 16
ROCm vs Vulkan
勝ちは固定しない
ROCm/HIP と Vulkan の優劣は一概に決まらない。モデルの特性、コンテキスト長、ソフト最適化の相性で変動する。
変動要因
モデル特性（密/スパース等）・文脈長・最適化状況
実務結論
ユーザー側でモデル/設定をベンチして決めるのが安全
「どれを選べば常に速い」は成立しにくい、という現実的な不確実性が明示されている。
context
08 / 16
長いほど遅い
コンテキスト増で劣化
コンテキスト長を増やすエージェント風シミュレーションでは、全モデルが速度低下し、最大級コンテキストでは完走しないケースもあった。
なぜ起きる
過去トークン参照の増加で必要処理・参照データが拡大
影響
エージェント用途では実運用の遅延リスクになる
速度劣化はpp/ decode双方で起きやすい。
thermals
09 / 16
120W運用
熱・電力は破綻しにくい
小型筐体でも TDP 120W 近傍の運用が概ね維持され、初期ブースト後に120Wへ落ち着く挙動が観察された。
観測パターン
初期ブースト → その後 TDP 120W へ落ち着く
注意点
条件依存（パラメータ/周辺負荷等）で、全ワークロード保証はできない
（参考）USB-C PDで最大240W級の可能性も述べられている。
value
10 / 16
性能より導入
価値の中心はソフト
Haloの差別化はBKC（Batteries Included 的に“互換性が検証された出発点”）と、AMD AI Playbooksによる段階的な手順案内にある。
BKC/Developer Center
依存関係の沼を避け、検証を止めにくい
Playbooks
LLM導入・微調整の手順を“短く・摩擦少なく”
LM Studio / Lemonade / VSCode（Cline）やPyTorchの周辺手順も整備されている点がメリット。
risks
11 / 16
不確実性の種
プレイブックの失敗
一方で、プレイブックには GitHub issue で失敗している箇所があり、将来の更新や“確実性”にはリスクが残ると述べられている。
互換性リスク
更新で前提が崩れる可能性（導入の再現性が揺れる）
評価の限界
llama-bench中心で、学習・長期稼働・同時実行などへの一般化は難しい
NPUも“動作確認はあるが”、メトリクス不足で定量内訳が弱い。
opinion
12 / 16
購入判断
まず試す価値は高い
私は Haloを「早く試す環境」として有力だが、投資回収はユーザー自身の実モデル・実運用パラメータで検証すべきだと考える。
良い点（要約）
導入体験（BKC/Playbooks）が体験差を作る／熱・電力も破綻しにくい
注意点（要約）
benchの範囲外が多い／decode帯域依存／プレイブック失敗やNPUテレメトリ不足
性能差は“Haloが遅い”というより「支配要因がどこか」を見抜く問題。
closing
13 / 16
Takeaway
指針はベンチ前提
Haloは“始めやすさ”で強く、decodeはメモリ帯域依存、長いコンテキストは遅くなりやすい。最終判断は手元のモデルと設定で検証する。
優先して確認
pp/tg・コンテキスト長・RoCm/HIP vs Vulkan
導入で見る
BKC/Playbooksの再現性（失敗箇所の有無）と更新追従
NPUは動作確認のみなら十分でない可能性があるため、省電力・効率の“内訳”は追加で確かめるのが安全。
