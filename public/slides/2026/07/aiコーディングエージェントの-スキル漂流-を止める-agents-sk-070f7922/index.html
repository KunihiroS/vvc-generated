cover
02 / 16
スキル漂流を止める
.agents/skills
AIコーディングエージェントに渡す「スキル指示」を、エージェントごとに分散させない設計に変える提案を整理します。
problem
03 / 16
別台本が原因
議論ではなく参照先
Claude CodeとCodexのルールが違うのは、AI同士の価値観衝突ではなく、あなたが別のplaybook（台本）を渡しているからです。
metaphor
04 / 16
スキルは台本
配布ミスで挙動が変わる
同じ依頼でも、エージェントが読むスキル文書がズレれば生成方針もズレます。
ズレの発生
更新漏れ（片方だけが最新）
結果
方針・コードが食い違う（skills drift）
comparison
05 / 16
昔の対策
複製 vs symlink
従来は「コピー」か「シンボリックリンク」がよく使われますが、どちらも長期運用で問題が出やすいとされています。
複製（コピー）
更新のたびに分岐が増え、漏れが起きる
symlink
壊れるときが静か／相対パス移動・リネームで破損
distinction
06 / 16
レイアウト問題
内容ではなく参照
解決の中心は「スキル文の正しさ」ではなく、「全エージェントが同じ場所を読める参照設計」にあります。
判断基準
レビュー可能性／再現性／更新の一回性
狙い
どれが正かを推測させない
architecture
07 / 16
一本化の要
.agents/skillsへ集約
スキル指示を .agents/skills/ に一本化し、全エージェントが同じディレクトリを読むよう設定します。
.agents/ skills/ skill-name-1/ SKILL.md skill-name-2/ SKILL.md
process
08 / 16
構造で安定化
1スキル=1ディレクトリ
「skills配下にスキルごとのディレクトリ1つ」「各スキルには SKILL.md 」で、名前と内容の安定性を保ちます。
ルール i
共有する単位はスキル
ルール ii
各スキルは独立フォルダ
ルール iii
内容は必ずSKILL.md
enumeration
09 / 16
Git境界を切る
ユーザー設定を隔離
エージェント別ディレクトリ（例：.claude/ .codex/）はユーザー設定として扱い、ノイズになりやすいので追跡しません。
追跡する
.agents/（ポータブルなスキル指示）
無視する
それ以外のエージェント状態（.gitignoreへ）
evidence
10 / 16
配布を自動化
npx skillsで導入
Vercel CLI npx skills を使い、リポジトリ（またはスキル専用リポジトリ）からインストールする運用が案内されています。
狙い
参照先統一を手作業にしない
効果
導入の再現性を上げる
comparison
11 / 16
壊れ方の差
symlinkより中心化
symlinkは「壊れ方が静か」になり得ますが、中心化は変更が1箇所に集まり、レビューも追跡も容易になります。
symlinkの弱点
相対パス依存で静かに破損／追跡コスト増
中心化の利点
PR差分が単一箇所／全エージェントが同時に最新
problem
12 / 16
落とし穴もある
共有と分岐の線引き
中心化は強い一方で、エージェント固有の要件（出力形式、API差分、ツール制限）をどう吸収するかは別設計が必要になり得ます。
リスク i
固有要件が無視されると別の不整合が生まれる
リスク ii
移行後に配布が本当に統一されたか検証が必要
closing
13 / 16
結論
スキルは中心化せよ
エージェントツールが変わっても、スキル（ドキュメント）は変えずに済むよう設計し、参照の統一でskills driftを防ぎます。
推奨アクション
.agents/skills/ に一本化
運用方針
Git追跡は .agents/ のみ、配布は npx skills
