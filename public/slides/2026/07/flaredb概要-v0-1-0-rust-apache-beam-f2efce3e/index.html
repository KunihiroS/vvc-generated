Rustで動く Beam 実行基盤

FlareDB 概要（v0.1.0）

Apache Beamで書いたバッチ/ストリーミング処理を、Rustベースのランタイムで実行することを目指す「実行エンジン」報告です。
狙い

現状（v0.1.0）
単一ノード、Global Window＋Bounded sources、限定的ネイティブ変換、Portable DoFn 等

03 / 15
ストリームだけでも表だけでも

課題は 統合

ストリーミング処理（動いているデータ）と、テーブル化した状態（問い合わせ可能なデータ）は、別々の仕組みに分かれがちです。
分断の結果
処理と保存/参照のつながりが弱く、状態管理・再処理・運用の手間が増える
理想の姿
同一エンジン内で、状態を作りながら処理とクエリへ到達する
metaphor
04 / 15
移り変わるPCollection

Streams → Tables

PCollectionがパイプラインの進行とともに、ストリーム的表現から材料化テーブル状態へ遷移するイメージです。
Streams
流れている途中のデータ状態
遷移点
処理の積み上げで「材料化」が生まれる
Tables
同じデータを材料化した状態（保存/参照へ）
architecture

実行（動かす側）
Beam SDK Harness / Portable DoFn / Portability Framework を扱うランタイム

10 / 15
リスクは“状態”に集まる

運用の 事故要因

ストリーミングでは遅延・重複・再処理・状態整合性が事故の中心になりやすいです。
整合性（Consistency）
Stateful Processingや再処理の仕様が未充足なら要検証
スケール（Scale）
単一ノードではボトルネックが顕在化しにくい可能性
運用（Ops）
ログ/監視/メトリクス、状態バックアップ、アップグレード手順の記載が不足
互換性（Compatibility）
Portable DoFn最適化・互換の詳細は情報不足
