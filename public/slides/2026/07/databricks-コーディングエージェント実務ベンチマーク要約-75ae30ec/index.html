Databricks / Benchmark
02 / 16
タスク総コストが鍵
Coding Agents
Databricksの社内ベンチマーク要約。モデル性能だけでなく、ハーネス（エージェントの呼び出し・手順）でコストと品質が大きく変わることを示します。（ハーネス/ハーネシング：日本語補足）
目的：導入判断のために「何を測るべきか」を短く整理する。
Problem
03 / 16
モデル選びだけでは足りない
Same Model, Different Result
同じLLMでも、エージェントをどう呼び、何をどれだけ渡し、どんな手順で作業させるか（ハーネス）で結果が変わります。報告では、品質が近いのにコストが2倍以上変わり得ることが示唆されています。
ハーネス例
文脈量（context）／実行回数／探索・停止条件（作業長）
帰結
性能差ではなく「システム設計差」が勝つ局面がある
Core Idea
04 / 16
何を買うべきか
Total Cost / Task
トークン単価（1トークンいくら）は、実タスクの完了までに必要な試行・読み返し・実行ステップの揺れを反映しにくいです。だから意思決定はタスクあたりの総コストで見るべきだと強調しています。
なぜ無理？
モデルごとに推論効率・作業長が異なる
現象
“安い”のに完了まで長くなると逆転する
結論
同品質でも費用が変動するので測定が必要
Cost vs Quality
05 / 16
パレートを見ろ
Pareto Frontiers
コーディングタスクでは、同一コストでの最良品質（または同一品質での最小コスト）＝パレートフロンティアが重要です。結論として、OpenAIやAnthropicだけでなくオープンソースも含まれると述べています。
見方
品質 × 費用 のトレードオフを同時に評価
含意
“最上位モデル固定”が最適とは限らない
Evidence
06 / 16
GLM 5.2の位置づけ
GLM 5.2: Daily Driver
報告では、オープンモデル、とりわけGLM 5.2が高い品質帯に入りつつ、価格面でも優位とされています（例：$1.28/task 対 $1.94）。つまり日常の開発ワークフローで現実的に使える、という位置づけです。
品質
上位能力帯（報告上の位置づけ）
費用
$1.28/task（例）でコスト効率が高いと示唆
Harness Effect
07 / 16
呼び方が支配する
Claude / Pi / Harness
ハーネス（エージェントの枠組み）を変えると、同じモデルでも品質が近くてもコストが2倍以上になる可能性がある、と報告されています。例としてClaude Code/CodexとPiのような比較で示唆されています。
変化
文脈供給（context）と反復回数
結果
同品質でも作業時間・消費が変わる
教訓
モデル比較の前に設計を揃える
Benchmark Design
08 / 16
自前ベンチの理由
Why Not Public?
公開ベンチマークは有用ですが、時間経過で学習データに漏れてしまう可能性や、自社コードと非代表性の問題があります。さらに自社PRベースなら、開発者への悪影響を抑えつつ実テストで評価できるため、自前ベンチを採用したと説明されています。
限界(a)
漏えい・既知化のリスク（学習データへの混入）
限界(b)
言語・ビルド基盤・サービス構成が違う
利点
自社PRに対する実テスト合否で現場に近づく
設計思想
研究ではなく意思決定の基盤を作る
Evaluation Method
09 / 16
合否は“テスト”で
No LLM Judge
LLM judge（もっともらしいが誤り）に報酬が偏るリスクを避け、テスト評価で合否判定する方針が説明されています。報告では、テストで測れることを中心に置くことで解釈のズレを抑えます。
狙い
“尤度の高い説明”ではなく“動く変更”を評価
効果
実装品質の観点でブレを減らす
英語用語：LLM judge / test-based evaluation（テストベース評価） （LLM judge：LLMで採点する方式）
Decision Framework
10 / 16
運用に落とす
Route Models by Data
「最適モデルを固定」するより、ベンチに基づいてモデルとハーネスを切り替える運用思想が合理的だと読み取れます。実現にはログ・評価・ガードレール（安全柵）を含む計測基盤が前提になります。
設計
ハーネス（手順）をタスク形状に合わせる
計測
タスク単位の総コストと品質を継続評価
ガバナンス
モデル更新・再評価で陳腐化を防ぐ
Critical Notes
11 / 16
そのまま一般化は危険
What Might Not Transfer?
結果は社内コードベース前提なので、タスク選びや難易度分布、カテゴリ構成が他社に一致する保証は本文だけでは不十分です。さらに、テストが測れない非機能要件（性能・セキュリティ・保守性・可読性）へのカバーが限定的な可能性があります。
一般化
タスク分布・カテゴリが違うと順位が変わり得る
品質軸
テスト以外（セキュリティ/保守性等）を別途評価
Missing Analyses
12 / 16
因果まで見たい
Harness: Which Knobs?
ハーネスが効くことは示されても、どのパラメータ（文脈量、探索戦略、停止条件など）が最も効くかの因果分析までは読み取れません。導入時は、あなたの環境で同等のログ計測と反復設計を行う必要があります。
不明点
文脈量（context size）が支配するのか
不明点
探索・試行回数・停止条件の寄与
要件
ログ/評価で自社向けにチューニング
Closing
13 / 16
結論：測ってルーティング
Don’t Guess. Benchmark.
モデル単体ではなく、タスク総コストとハーネス設計でコーディングエージェントを評価すべきです。公開ベンチは補助で、自社PR＋実テストで意思決定の再現性を高める――その思想が中心結論です。
必須
タスクあたりの総コスト × 合否（テスト）
推奨
モデル固定より、ベンチに基づく切替運用
注意：ここでは安全・法務の網羅はしていないため、導入時は社内ガバナンス（データ取り扱い、権限、ログ、依存関係、ライセンス）を別途確認。
