Undocumentedの要点
Claude Code READ
Claude Codeを「公式に十分書かれていない設定・挙動」まで含めて、ソースコードから読み解いた内容を一気に把握します。
Focus / 主題
hook・auto-mode・skill拡張・memory/dream・Magic Docs
Risk / 前提
バージョン依存（未公開は将来変更の可能性）
読み方の前提：入力データは「特定バージョンで観測できたスナップショット」です。

03 / 14
強力だが境界が曖昧
Safetyの勘所
未記載機能を使うと自動化の到達範囲が広がりますが、誤許可・誤変換・将来の互換性断絶のリスクも増えます。
Flexible / 柔軟性
イベント単位で挙動・権限・入力を組み替え可能
Danger / 危険
条件ミスで意図しないコマンド変形が起こり得る
Version drift
Undocumented は将来挙動が変わる（止まる/暴走の差が不明）
Operational load
ログ・監査・再現性がないとデバッグが急に難しくなる

更新範囲を“境界設計”
Magic Docs
「# MAGIC DOC: ...」の見出しと、その直後の指示で更新対象をスコープできる、と説明されています。
How
特定書式の見出し → 更新エージェントの対象を制限
Stop rule
Magic DOCのヘッダを削除すると追跡が止まる（とされる）
Limit
書式不備で意図しない更新の可能性
Human check
更新結果の整合性までは自動保証されない可能性

13 / 14
魔法ではなく、設計で勝つ
Takeaway / 要点
Claude Codeの未記載拡張は実務的に強い一方、監査・テスト・権限最小化・回帰で“事故率”を下げるのが必須です。
Hook（フック）
stdout JSONで制御できる → 境界条件を厳密に
Auto-mode（自動許可）
YOLO Classifierが環境文章も参照 → 検証可能な条件を書く
Memory（記憶）
dreamで統合 → 誤りの残存に備え監査と取り消し
Magic Docs（更新境界）
見出し書式でスコープ → ヒューマン確認フローを作る
“Undocumented＝不安定/将来変更され得る”前提で、導入後の観測と改善を組み込む。
