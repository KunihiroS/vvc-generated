公開を最小化

Secure MCP Tunnel安全トンネル

プライベートなオンプレ/社内MCPサーバを、外部インターネットへ公開せずに ChatGPT・Codex・Responses API 等から使う方法です。

Key idea: tunnel-client が OpenAI 側のトンネル端点へアウトバウンドHTTPS接続し、必要なMCPだけを社内へ転送します。

境界の壁

公開できない制約

ファイアウォールやガバナンスの都合で、社内（オンプレ/開発マシン内）のMCPサーバを公衆向けに公開できないことがあります。

必要
対応OpenAI製品からMCP経路で呼び出せること
困る
インバウンドのポート解放や外部公開リスナーが無理なこと

03 / 15

出口は1つ

アウトバウンド一本釣り

社内に置く tunnel-client が OpenAI ホストのトンネル端点へ outbound HTTPS で接続し、MCP要求を社内へ安全に渡します。

OpenAI側
トンネル端点が「通常のMCP経路」として振る舞う
社内側
tunnel-client が JSON-RPC形式で転送し、レスポンスを返す
architecture
04 / 15

転送の流れ

Synchronousキュー/待ち

OpenAI側がキューにあるMCP作業をクライアントへ渡し、tunnel-client が社内MCPサーバへ要求・応答を同じトンネル経路で中継します。

1
OpenAIプロダクト → トンネル端点へMCP要求
2
トンネル端点 → tunnel-client が社内へ転送
3
社内MCPサーバ → 応答をJSON-RPCで返送
4
結果がトンネル経路でクライアントへ戻る

05 / 15

権限で制御

Tunnels運用モデル

トンネルIDと実行用APIキー（Tunnels Read + Use）、必要に応じてメタデータ編集権限（Tunnels Read + Manage）でアクセスを絞ります。

トンネルID
どのトンネルを使うかを一意に特定
Tunnels Read + Use
読み取りと利用（実行）を許可
Tunnels Read + Manage
必要な場合にのみメタデータ編集を許可
ワークスペース
トンネルが関連付く場所（scope）を誤ると失敗

まず健全性

/healthz/readyz

Secure MCP Tunnelは、tunnel-client の稼働状態（/healthz, /readyz, /metrics, /ui）が運用の鍵です。

接続失敗
クライアントからのMCP要求が失敗し、再接続まで復旧しない
切り分け
「クライアントが健全か」を先に確認できる
ヒント: /ui はループバック前提のため、確認環境に注意します。

何が変わるか

公開不要へ転換

MCPサーバは外部から到達可能な公開リスナーを持つ必要がなく、受信側ポートを開けない運用が可能になります。

従来
オンプレMCPをインターネットへ公開 or ポート解放が必要になりがち
Secure MCP Tunnel
tunnel-client が社内境界を維持し、外部公開を最小化

限定アウトコール

Harpoon必要最小限

Harpoonは内蔵MCPサーバにより、特定ラベル/設定に限定されたHTTPコールアウトを社内へ行えます。

安全側
汎用プロキシではなく、設定したターゲットとメソッドに範囲制限
注意
事前登録されていないREST APIは呼べず、設計不足が障害になる

よくある詰まり

FAQ失敗理由

つまずきは「公開」よりも「関連付け/権限/到達性」に出やすいです。

Q: トンネルがChatGPTに出ない
ワークスペーススコープ未関連付、または Tunnels Read + Use 権限不足
Q: OAuthが失敗することは？
MCPサーバが到達可能でも、authorization server の到達性が別問題になり得る
Q: tunnel-client run が失敗した場合は、再接続までMCPフローが正常化しないことがあります。

導入の見方

接続＋運用セット

私は「外部公開を最小化しつつ、必要な連携だけを成立させる」設計として非常に実務的だと考えます。

良い点
権限（Tunnels Read/Use/Manage）と健全性確認（/healthz等）で、導入後の切り分けがしやすい
注意点
レイテンシ/再送/ストリーミング等の挙動、鍵ローテーションや監視など「周辺設計」が必要

要点再掲

Checklistまとめ

次の5点を押さえると、技術・運用・統制の観点で導入判断が進めやすくなります。

i
インバウンド不要：アウトバウンドHTTPSで tunnel-client が接続
ii
経路：OpenAI側トンネル端点が通常のMCP経路として機能
iii
権限：Tunnels Read + Use（必要なら Manage）とワークスペース
iv
運用：/healthz, /readyz, /metrics, /ui を監視
v
限界：OAuthの authorization server 到達性、Harpoonの範囲制限、周辺の監視/鍵管理
