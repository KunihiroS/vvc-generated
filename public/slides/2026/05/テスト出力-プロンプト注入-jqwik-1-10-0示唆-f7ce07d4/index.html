Secure Supply Chain
02 / 16
stdoutが“プロンプト”になる

Test Output 起点の注入リスク

テスト用ライブラリの標準出力（stdout）が、CIログやIDE表示を介してコーディングエージェントの入力文脈に混ざる—という新しいサプライチェーン視点を扱います。

Case
jqwik 1.10.0（主張ベース）
Core risk
“攻撃挙動”ではなく“入力混入”が問題
Problem
03 / 16
どこが危険？

Inputs 混ざる場所

典型的なサプライチェーン対策は、インストールフックやネットワーク通信など“攻撃っぽい挙動”を見ます。

しかし今回は、実行時にstdoutへ出る“指示文テキスト”が、ログ/表示を介してAIの入力文脈へ回り込むことが論点です。

Metaphor
04 / 16
見えない紙片

ANSI 消す

ANSIエスケープで人の端末では文字が消えやすくても、CIログや記録経路ではテキストが残り得ます。

Human view
ESC[2K / \r で“見えにくい”
Machine capture
ログに残る → 要約/解析で取り込まれる
Architecture
05 / 16
回り込みの流れ

CI → Agent 文脈へ

テスト実行の失敗ログが、コーディングエージェントの“プロンプト生成”に使われると、stdout由来の文字が混入し得ます。

Step i
jqwik等がstdoutに指示文を出す（主張）
Step ii
CI/IDEが表示・保存（消去は“描画”に依存）
Step iii
エージェントがログを要約し、プロンプトへ組み込む
Why risky
06 / 16
リスクは“場所”

Prompt-like テキスト

今回の本質は、破壊的なコード実行そのものよりも、AIが読むべきでない文脈に“命令っぽい英語”が混ざる点です。

i. Not malware
ファイル改変・通信などは強調されない（提示文）
ii. But injection
混入 → AIが“指示”として解釈する可能性
Evidence
07 / 16
提示された要点

jqwik 1.10.0示唆

jqwik 1.10.0リリース時のテスト実行器に、指示文とANSI消去の組み合わせが含まれていた—という主張が中心です。

Mechanism
“Disregard previous instructions…”系の英語テキスト
Visibility gap
端末では消えるが、CIログでは残り得る
Response (thread)
挙動説明が追加され、依存が外され、文字列は残った（経緯）
Why it matters
差分を追わないと気づきにくい
SLSA
08 / 16
来歴は通るかも

Provenance だけでは足りない

SLSA的な“来歴（provenance）”が正しくても、内容がエージェントの安全性にとって有害な入力になり得ます。

What SLSA checks
どこから来たか／改変されていないか
What this attacks
“AIが読む入力”という意味での安全性
Comparison
09 / 16
従来と違う盲点

Install vs Test 観測範囲

インストールフックや難読化・外部通信を主に見るスキャナでは、テスト出力（平文ASCII＋消去制御）の見落としが起きやすい、とされています。

Classic signals
hook / network / file rewrite / obfuscation
This category
stdout指示文（描画隠蔽＋ログ残存）
Threat Model
10 / 16
“実害”の確度は条件次第

Not proved 即悪用

提示文は「確実にAIが従った」ことを実証しきれておらず、依存関係やエージェントの組み込み方次第で結果が変わる“仮説”として扱う必要があります。

Condition i
エージェントがstdout/ログをどうプロンプト化するか
Condition ii
AIが英語の“命令文”を文脈指示として扱う度合い
Mitigation
11 / 16
現場での対策観点

Harden ログ→プロンプト

具体手順は提示文に十分に書かれていませんが、現実的には「ログの取り扱い」「プロンプト混入」「レビュー観点」の拡張が必要になります。

i. Review
依存更新差分で“テスト出力”も確認
ii. Sanitize
CI/エージェントへ渡すログをフィルタ/サニタイズ
iii. Isolate
命令文らしい文字列をプロンプトへ混ぜない設計（prompt isolation）
Bilingual note
prompt isolation（プロンプト隔離）＝入力の混ざりを遮断
Bilingual note
sanitization（サニタイズ）＝危険な形式を除去
Ethics
12 / 16
意図と影響がズレる

Intent vs Impact 緊張関係

メンテナの意図が倫理的反対であっても、第三者のCIや開発フローに“安全設計として遮断すべき入力”を注入してしまうと、被害が生まれ得ます。

Ethical intent
反対運動の主張（提示文に基づく文脈）
Safety impact
第三者の作業効率・安全に介入（ログ経由で起きうる）
Governance
どこまで“許容”し、どこから“禁止/注意”するか
Need evidence
再現手順・頻度・統計が不足 → 追加調査が望ましい
Takeaway
13 / 16
結論

Detect 入力混入

サプライチェーンの検査範囲を、インストール時だけでなく「テスト出力→AI入力」の経路まで広げないと、来歴が正しくてもリスクを見落とします。

Key message
“攻撃コード”より“prompt-like混入”を脅威として扱う
Next step
ログのサニタイズ／プロンプト隔離／テスト依存レビューを試す

参考：本デッキは提示文に基づく要約であり、AIが実際に従ったことの“確証”ではありません。
