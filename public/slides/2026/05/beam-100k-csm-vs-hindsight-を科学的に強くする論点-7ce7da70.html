CSM 科学性の論点
Stronger

BEAM 100Kで Context Swarm Memory（CSM）を Hindsight artifact と比較する話を、「科学的に強くする」ための不足点と追加実験を整理します。

PROBLEM
02 / 16
本質
Evidence

数値（AMB・正答・トークン削減）だけでは因果が分からず、再現性情報が欠けると「比較としての妥当性」が弱くなります。

主張の形
性能向上＋（遅延という）制約の同時提示
最大の弱点
reproducibility と要素分解の不足

03 / 16
結果
Scores

ローカル設定ではCSMが Hindsight を上回り、AMBスコアと正答数、さらに「見えるコンテキスト・トークン」の削減が示されています。

CSM

Latency

CSMは性能が良い一方、平均リトリーバル遅延が大きく、低レイテンシ用途では本番投入が難しいと結論づけられています。

アブレーション候補
Elements

投稿者が整理したアブレーション案は、書き込み・読み出し双方の失敗モードを検査するための要点になっています。

read-path
probe gateなし / トップルータシャードの安全リコールなし
synthesis
合成ステップなし / context窓のキャップ変更
write-path
event-IDヒントなし / embeddingリコール床なし / return-K変更

07 / 16
頑健性
Thresholds

Committer thresholdsの微妙な違いで結果が変わるなら、再現性と一般化性が弱くなります。感度曲線（スイープ）が必要です。

懸念
AMBが gate閾値への依存度を持つ可能性
追加
「書き込み側のドリフト」別ベンチマークで閾値スイープ

不十分

外挿
Window

context窓分布やキャップを変えるテストは、メモリ効率（削減）と品質維持（正答率）が本当に両立しているかを確認します。

提案
context窓分布の変更テスト
見るべき
AMB・正答・トークン削減・失敗モードの同時変化
SYSTEM
11 / 16
ボトルネック
Breakdown

遅延が4.5倍規模で悪化しているのに内訳がないため、どこを最適化すべきかが読めません。段階別の計測が必要です。

計測
probe / recall / synthesis それぞれの時間
比較
read-path vs write-path の遅延分離
対策
並列化・キャッシュ・ルーティング改善の優先度決定
CLOSING
12 / 16
結論
Next

CSMの優位は「有望な兆候」です。科学的に確かにするには、アブレーション・閾値感度・再現性情報・段階別遅延と外挿テストを追加して、因果と頑健性を固める必要があります。
