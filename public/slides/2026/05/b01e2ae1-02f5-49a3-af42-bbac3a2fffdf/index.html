最小モデルの境目

ReAct オーケストレーション

「考える→ツール選択→呼び出し→観測」を回すエージェントで、
推論力よりツール呼び出しの規律（契約遵守）が先に壊れる。

読み物（16:9）
ReActループ / tool discipline
PROBLEM
03 / 16

何が信頼性を落とす？

壊れるのは「推論」より先

オーケストレーション役の出力は文章ではなく「ツール名・引数・JSON形式」で検証される。
小型化で落ちるのは、意味よりAPI契約（スキーマ）の遵守率。

FAILURE MODES
04 / 16

典型的な破綻例

引数の“創作”が致命傷

未知パラメータ
存在しない引数を付与：例 overwrite=true（署名に無い）
形式逸脱
宣言されない output_mode や出力フォーマット違反でツールコール不能
ツール名の誤り
“あるはずのツール”を呼ぼうとして tool not found 方向に崩れる
結論用ツール
宣言されていない conclusion tool を発明的に呼ぶ
MINIMUM MODEL
05 / 16

下限の目安

Qwen3.6-35Bが近い

投稿者は、ループ部分だけに小型モデルを割り当てる運用で、Qwen3.6-35B-A3B（MoEで約3B active）が下限に近いと述べる。
ただしタスク・ツール設計・評価条件次第で前後する。

PRINCIPLE
06 / 16

設計で勝つ

「作ってはいけない」を潰す

小型モデルでも成立させる鍵は、自由生成の余地を減らし、出力空間を物理的に制約して“捏造”を出にくくすること。
プロンプトよりデコーディング制約・検証が効きやすい。

PROMPTING
07 / 16

ツール署名を明示

改善するが万能ではない

狙い
システムプロンプトにツールの引数一覧（署名）を載せて選択と整合を促す
限界
プロンプトだけでは「宣言されていないtoolを呼ぶ」や完全な形式逸脱は止まりきらない

つまり、署名提示は主観的には効くが、構造的破綻まで封じる手段ではない。 （制約デコード／スキーマ検証が本命になりやすい）

CONSTRAINED DECODING
08 / 16

文法制約デコード

JSONスキーマ/BNFで抑止

JSONスキーマやBNF（llama.cppのgrammar等）で、スキーマに反する引数の追加をデコーダ段階で抑える。
その結果、未知パラメータ（例 overwrite=true）が「生成されても通らない」になる。

CLASSIFICATION
09 / 16

生成→分類へ

自由度を捨てて正確性を上げる

オーケストレーションでは、自由記述ではなく テンプレ選択（分類）に寄せる。
例：次に呼ぶ（tool,args）をN個のテンプレから1つ選ぶ形にする（/テンプレ選択：テンプレ選択〈分類〉）。

TOOL COUNT
10 / 16

ツール数を絞る

混乱源を減らす

ルール
小型モデルは選択肢が多いほど混乱しやすい
設計
ツールルーターやスキル分割で、メインが見て決める選択肢を最小化

“何を呼ぶか”の自由度を構造的に圧縮すると、壊れ方（捏造）が起きにくくなる。

GUARDRAILS
11 / 16

ウォッチドッグ

失敗反復を検知して戦略変更

ツールコール失敗の繰り返しを検知して、戦略変更を促す“周辺ガードレール”も提案される。
ただし位置付けは応急処置で、根本は生成空間の制約にある。

EVALUATION
12 / 16

断言が難しい

最小モデルは一般解にできない

左右する要因
ツール数 / 引数の複雑さ / タスク分布（ファイル操作・コード検索比率）
評価設計
成功判定（何ステップまで許容？）/ 最大手数 / 致命失敗の扱い / ガードレール有無

そのため「このスレッドだけで何Bまで」とは言えない。

TAKEAWAY
13 / 16

結論

最小モデルより、契約設計

最も確からしい戦略は「小型モデルを無理に選ぶ」より、(tool,args)をスキーマ制約・選択肢削減・テンプレ化で誤りの形を限定すること。
結果として、モデルサイズより“設計の強さ”が小型化耐性を決めやすい。

次の実務ステップ例：JSON/BNF制約 → ツール数圧縮（ルーター/分割） → 失敗時の戦略変更（ただし本命は制約）。
