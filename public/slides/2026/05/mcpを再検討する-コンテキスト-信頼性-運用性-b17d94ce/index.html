02 / 16
cover
万能ではない
MCP再検討
MCPは“標準接続”として語られる一方、実運用ではコンテキスト・信頼性・デバッグ性の論点が出やすい。追実測と代替案を、批判的に整理する。
要点：いまは CLI / Skills が勝ちやすい場面が多い。必要な場面だけ MCP を使う。
03 / 16
problem
コストは常時発生
コンテキスト圧迫
MCPはツール定義（名前・説明・引数スキーマなど）をLLM側が読む必要があるため、コンテキストウィンドウを恒常的に消費しやすい。使わないツールまで“定義として保持”される構造が効く。
実測の示唆
Quandriで Linear/Notion/Slack/Postgres の4サーバ連携：ツール定義だけで 約10.5%〜16.5% を消費（モデル：Claude 200K / GPT-4o 128K）。
例（厳しさ）
Linearだけでも「42ツール分」が常時ロード。さらにLinearで12,800トークン超の言及もあり、説明・根拠が“作業領域”を削る。
04 / 16
evidence
遅延＋失敗モード
信頼性の追加層
MCPでは、LLMと外部APIの間に“サーバプロセス層”が入る。初期化・認証・ラウンドトリップが増え、初回遅延や停止（クラッシュ）などの障害が問題化しやすい。
発生しやすい
初回の初期化遅延（立ち上げ・準備）
運用負担
再認証・初期化失敗・セッション中停止
構造的理由
往復回数増（呼び出しのたびにプロセス層を経由）
05 / 16
distinction
“会話内”の閉域
デバッグ性
CLI/APIは、人間とAIで同じコマンド体系を共有し、パイプで合成し、端末で即時デバッグできる。MCPは会話コンテキスト内に閉じるため、再現性や合成の導線で不利になり得る。
CLI/APIの強み
人間とAIで同じコマンド体験 / パイプ合成 / 即時デバッグ / 既存ドキュメント（man pages等）との親和性。
MCPの弱点
“会話内にしか存在しない”設計になりやすく、学習・検証・再現の手触りが弱くなりがち。
06 / 16
enumeration
主張の地図
論点5点
記事が提示する中心メッセージは「MCPは今のままだと効率と運用で不利」だが、更新で一部が改善し得る、という折衷です。
i コンテキスト
ツール定義が常時ロードされると入力余地が減る（コストと推論制約）。
ii 信頼性
初期化・認証・クラッシュなど“プロセス層”の失敗が増える。
iii 役割重複
CLI/APIの強み（コマンド整合・合成・デバッグ・知識学習）が既にある。
iv 改善余地
Deferred Loading等でコンテキスト問題は軽減し得る。
v 代替策
CLI-first と Skills（必要時だけ載せる）で実務を回す。
結論
MCPは万能の標準ではなく、“向いている場面で採用”。
07 / 16
process
Deferred Loading
改善の方向性
アップデートとして Claude Code の Tool Search with Deferred Loading が挙げられ、MCPツールスキーマのコンテキスト使用が 85%+ 削減されたとされる。つまり“問題1は本質的に不変”とは限らない。
期待される効果
必要なツールだけを段階的にロードし、常時保持の割合を下げる。
残る論点
性能・デバッグ・アーキテクチャ層（プロセス層）は依然として残り得る。
08 / 16
metaphor
工具箱 vs 指示書
Skills/CLIの感覚
Skillsは“必要な工具だけをその場に出す”発想。MCPは“工具箱の説明書を常に卓に置く”構造になりやすく、使わない分もコストになる。
Skills（必要時）
手順・コマンド（例：curl、jq）を“必要になった瞬間”にコンテキストへ。
MCP（定義常時）
ツールスキーマを読むため、会話内に置く情報量が増えやすい。
09 / 16
comparison
どっちが得？
CLI-first vs MCP
現実解は「常にMCP」でも「完全に置き換え」でもなく、ワークフロー適合で選ぶこと。特に一般的な開発の導線では、CLI-firstが有利になりやすい。
CLI-firstの利点
端末デバッグの即応 / パイプ合成 / 既存ドキュメントと学習済み知識への親和性。
MCPの採用条件
CLIが存在しないサービス / 非開発者向け / 双方向のリアルタイム通信などで価値が出やすい。
10 / 16
evidence
“強い結論”への警戒
一般化の罠
ただし、記事の“強い言い切り”は注意が必要。計測条件（モデル、環境、ロード方式、ツール数）が特定状況に依存し、更新で結果が変わる可能性がある。
依存要因
Claude Code上の実装 / Quandriスタック / ツールロード戦略
変動要因
ネットワーク条件 / サーバ実装品質 / 並列性 / キャッシュ
含意
“MCPが無価値”ではなく“現状の運用では不利になりやすい”。
11 / 16
architecture
復旧性まで設計する
導入の検証項目
MCPを導入するなら、(1)ツール定義ロード戦略、(2)認証・権限運用、(3)障害時の復旧性、(4)端末でのデバッグ導線 を個別に検証すべきだ。
運用設計
認証の再試行 / 初期化失敗時の挙動 / セッション途中停止からの復帰。
開発体験
再現可能な実行ログ / 端末で検証できる経路 / “どのツールをいつ載せたか”の可視化。
12 / 16
comparison
“安全”は実装次第
セキュリティ観点
記事では、DB領域のようにサーバ側で read-only や危険クエリ遮断を行える可能性がある点が言及される。ただし最終的には権限設計・検証手順が決め手になる。
ガードレール例
read-only / 危険クエリ遮断をサーバ側ポリシーで実装可能。
注意点
魔法ではない：資格情報の扱い・権限境界・監査（audit）が不可欠。
13 / 16
closing
結論：併用で勝つ
MCPは用途限定
MCPは万能の“標準接続”ではなく、状況によって残るべき手段。今は Skills / CLI-first が実務で有利になりやすく、導入判断ではツールロード・運用・復旧・デバッグを必ず検証する。
JP gloss: MCP（Model Context Protocol）= LLMを外部ツールへ接続する枠組み / Skills = 必要時だけ手順・コマンドを載せる運用 / CLI-first = まず端末手順を基準にする設計。
