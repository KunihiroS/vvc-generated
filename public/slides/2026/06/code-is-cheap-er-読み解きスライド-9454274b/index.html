Code is Cheap(er)
02 / 16
AIで“作る”が安くなる

読めば制御できる

大量のコード生成が当たり前になるほど、開発者の仕事は「理解」と「保守性の統制」に寄っていく——という論考の要点を整理します。

Problem
03 / 16
中心が移動する

書く→理解

AIで“コーディングそのもの”は速く・多くできるようになった。だが、理解・保守・複雑性制御は別コストとして残り、むしろ人間側に集まります。

安くなる
生成コスト（速度・量・作成容易性）
高くなる
読解コスト（後から理解し直す負担）
Metaphor
04 / 16
魔法使いの弟子

制御

魔法（AI）で手作業は楽になるが、制御を失えば暴走して混乱が起きる。AI時代は「魔法使い側（制御できる側）」であり続けるべきだ、という警告です。

弟子：outputに飲まれる
魔法使い：制約で形を保つ
Shift
05 / 16
逆転の構造

理解が主戦場

LLMが大量コードを生成すると、そのコードは「作った本人が理解している」前提に頼れない。人間は“後から読むことで理解する”費用を払い続けることになります。

従来
書いた人が概ね理解
LLM後
人間が大量成果を追う
結果
理解・レビュー負担が増える
Dispute
06 / 16
コンパイラは別物

“理解不要”は誤り

「コンパイラの出力を理解しなくても良い」ことをLLM擁護に使うのは誤りだとします。コンパイラは決定的で、ソースやワークフローも保たれるなど性質が違います。

コンパイラ
決定論的／範囲が限定／前提が追跡しやすい
LLM生成
前提・副作用を検証しないと不確実性が残る
Evidence
07 / 16
危険の形

大きな差分

LLM出力は人間の理解より速く増え得るため、「理解不能な差分（大きなチェンジリスト）」が危険です。特に新しいセマンティクス（意味・振る舞い）導入の局面で目立ちます。

何が起きる
追跡・検証が間に合わなくなる
なぜ危険
意図や副作用がレビューで固定されない
Process
08 / 16
段階化して追跡

小さく回す

大規模一括生成ではなく、小さな単位で提案→確認→反映を繰り返します。変更の意味が追跡できる状態を維持するのが狙いです。

i
小さく提案
ii
検証・確認
iii
反映して追跡
Risks
09 / 16
複雑性は敵

恐れがない

中核の懸念は「複雑性が増えるのを、人間ほど恐れない」ことです。過剰な積み上げはシステムを不変に押し込み、変更コストとバグ率を跳ね上げます。

LLMの性質
複雑性への心理的ブレーキが弱い
現場の結果
変更困難化→障害連鎖
Practice
10 / 16
減算的に整える

削るが設計

提案される態度は「減算的（Subtractive）で制約するエンジニア」です。出力を鵜呑みにせず、不要なレイヤやコンポーネントを“入れない／削る”ことに誇りを持ちます。

設計
境界・構成を増やしすぎない
運用
デプロイ相互作用を単純化
Builder
11 / 16
作るだけではない

Sculptor

責任は「builder（作る）」だけでなく「sculptor（彫る）」にある、という強調です。AIで増えた分を、形が壊れないよう削り、整えます。

builder
速度で作る
sculptor
制約で削る
Takeaway
12 / 16
最短の結論

作るより制御

コード生成が安くなるほど、「理解不能な差分」を減らし、複雑性の増え方を管理する価値が上がります。人間は生産性より、システムの形を壊さない判断に重心を置くべきです。

i
理解コストの増大を前提にする
ii
差分は小さくし、段階化する
iii
削る・入れないで複雑性を抑える
Closing
13 / 16
実務への問い

あなたは今どちら？

AIの出力を受け取るだけで終わっていないか。次の変更で「誰が理解し、何を制約し、どこを削るか」を先に決められているか。

チェック
大きな差分になっていないか（特に新しい振る舞い）
方針
減算で単純化できる箇所を先に探す
