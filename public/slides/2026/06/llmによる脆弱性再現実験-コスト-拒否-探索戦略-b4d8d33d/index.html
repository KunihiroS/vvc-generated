LLMで脆弱性再現は可能か

実験記録

LLMが模擬チャレンジ（脆弱な書評アプリ）でフラグを取れるかを、コスト・拒否・探索の作法まで含めて整理します。

対象は実サービス侵入ではなく、研究用に作られた環境で「解ける経路を探索できるか」を測る設計です。

PROBLEM
03 / 16
勝敗の外側にある鍵

3つの軸

本実験の成否は「コード/挙動から攻撃経路を組めるか」「拒否（refusal）やガードレールで止まるか」「予算上限で探索が途切れるか」の相互作用で変わります。

経路構築
攻撃手順を“どこへアクセスするか”まで成立させる
拒否・ガード
即時/遅延の拒否で探索が完了しない
資源制約
トークン/費用/試行上限で“解ける手前”で打ち切り
ARCHITECTURE
04 / 16
チャレンジの形

書評アプリ

LLMには「偽のReact Native（Expo）アプリ＋Pythonバックエンド」とチャレンジ説明（Markdown、APK相当）が渡され、成功すればフラグ取得です。

入力
APK（模擬）/チャレンジ説明（Markdown）/挙動・コード断片
ゴール
Firebaseに到達し、正しい操作経路でフラグ条件を満たす
EVIDENCE
05 / 16
成功はモデルに偏る

solve

成功（solve）は一部モデルに偏り、失敗は「主戦場に辿り着けない／辿り着いても手順がズレる／拒否で停止／予算で打ち切り」が中心でした。

GPT-5.5
10回中 7/10（最も高い傾向）
Deepseek系
拒否・探索停止で 0/10 の例（報告）
Claude
成功率は高くないが停止が少ない設定で進行（報告）
DISTINCTION
06 / 16
中心はFirebase到達

到達×正解経路

多くのモデルはAPIやRNアプリ側に固執し、Firebaseへ切り替えられないか、Firebaseに着地しても誤った使い方を選んで失敗しました。

失敗パターン
Firebaseに“行かない”
失敗パターン
Firebaseに“行くが”経路選択ミス（例：Firebase AuthをAPI経由で使う等）

つまり「脆弱性発見」だけでなく「攻撃の座標系（どこからどう進むか）」の一致が必要です。

ENUMERATION
07 / 16
拒否が探索を削る

refusal

拒否（refusal）が即時だと探索が止まり、遅延拒否でも“肝心の手順が終わる前”に打ち切られます。

即時拒否
Gemini 3.1 Pro Previewの例：早期停止
遅延拒否
Claude Opusの例：成功直前で効いた可能性
実務含意
安全制御は“攻撃可能性”だけでなく探索完了率そのものを下げる
PROCESS
08 / 16
探索は途中で終わる

予算上限

試行回数や上限（費用・時間）があるため、LLMが“まだ届いていない”局面で打ち切られ、結果が変わります。

例
10回走らせようとして $1,500 かかった（投稿者）
観測
Claude Sonnet/Opusで上限予算による途中停止が記載
ARCHITECTURE
09 / 16
モデル差だけではない

運用方式

エージェント/ハーネス/再試行（継続）など“推論フロー制御”が、到達可否や停止のされ方に影響し、結果差を作ります。

pi-goal-x
探索継続の試み（ハーネス差が出る）
Claude Code -p
中断が少なく計算途中で進みやすい可能性
含意
同じモデルでも“やり方”で成功率が変わり得る
COMPARISON
10 / 16
失敗の理由は複合

なぜ負ける

失敗は能力不足に限らず、(i) Firebaseに到達しない、(ii) 到達後に正しい操作経路を選べない、(iii) ガードレールで拒否される、(iv) 予算/時間で止まる、の組合せで起きます。

探索失敗
座標系のズレ：Firebaseへ寄らない
完了失敗
拒否・上限：解ける前に停止する
DISTINCTION
11 / 16
万能ハッカーではない

注意

成功しても“攻撃手順を自動で常に再現できる”わけではなく、探索対象（Firebase）への寄せ方を誤ると、費用だけ増えて成果が出にくいという限界が示唆されます。

誤経路
それっぽい別経路を選ぶ
打ち切り
“解ける局面”で上限に当たる
一般化
制約付き実験のため結論は限定的
EVIDENCE
12 / 16
読んで得る実務示唆

設計ポイント

セキュリティ研究/レッドチーミング文脈では、(a)探索継続、(b)誤経路への吸い込み防止、(c)コスト上限を前提にした段階評価が重要だと分かります。

i. 継続
探索を止めずに最後まで到達させる（探索戦略）継続/continuous
ii. 回避
誤経路に迷い込む評価・導線を抑える回避/avoid
iii. 段階
予算/時間で区切り、到達度を段階評価する段階/phase
CLOSING
13 / 16
結論

性能より探索

この記録は、LLMの賢さより「到達までの探索設計（場所の正解・手順の正解）」「拒否挙動」「コスト上限」が結果を決めることを示します。

“解けるか”はモデル単体ではなく、探索の運用方式と制約条件の束として現れる——それが最大の示唆です。
