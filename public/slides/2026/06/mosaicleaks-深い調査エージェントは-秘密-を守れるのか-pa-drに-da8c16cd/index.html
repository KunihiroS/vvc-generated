秘密は“順序”で漏れる
MosaicLeaks
深い調査エージェントがWeb検索を使うとき、検索クエリのログがモザイク効果（mosaic effect）で機密を復元し得ることを示す研究です。
核心テーマ / Core
検索ログ × 複数クエリの組合せ
対策 / Mitigation
PA-DR（プライバシー配慮学習）
Problem
03 / 16
単発は無害でも…
モザイク漏えい
単一の検索が機密を直接含まなくても、複数回の検索クエリが“断片”として外部から観測されると、観測者は組み立てて内部情報を復元できます。
何が危険？
クエリ内容だけでなく、順序・組合せ
日本語補足
断片を寄せ集めて真実が見える → モザイク効果
Metaphor
04 / 16
パズルのピース
ログは断片
各クエリは薄いヒントに見えても、深い調査の多段推論では“必要な手がかり”が交互に混ざりやすく、ピースが揃うほど復元されます。
Step A
社内文書の一部 → クエリ化
Step B
Web情報で次の推論 → 次クエリ
観測者視点
複数断片を再構成 → 機密が浮上
Enumeration
05 / 16
漏えいは3段階
intent / answer / full
漏えいの強さを、観測者がどこまで復元できるかで3種類に整理します。
i) intent（意図）
ログから「何を調べているか」が推測される
ii) answer（答え）
特定の質問があれば、その答えに到達できる
iii) full-information（完全情報）
質問なしでも検証可能な私的事実が言い当てられる
Evidence
06 / 16
性能最適化が逆効果
学習すると漏れる
単にタスク達成率を上げるよう最適化すると、むしろクエリに具体的な数値や日付など私的断片を入れがちになり、漏えいが増えることが確認されています。
見えた改善
厳格チェーン成功率（全ホップ成功）が上がる
見えた悪化
answer / full-information漏えいが増え得る
Architecture
07 / 16
対策は報酬で矯正
PA-DR
PA-DR（Privacy-Aware Deep Research）は、検索クエリ生成を“漏えいしない方向”へ学習で矯正するプライバシー配慮型の強化学習手法です。
(1) 計画/選択の報酬
各ステップの意思決定に状況別報酬を与える
(2) モザイク成立の評価
新クエリが「新たな漏えい」を作るかを学習的に罰する
Process
08 / 16
“次の一手”で判定
漏えいペナルティ
PA-DRは「クエリを出すたび」に、直接漏えい（direct）だけでなく、過去ログと組み合わさる新しいモザイク漏えいも評価して意思決定へ紐づけます。
i) クエリ生成
Plan / Chooseなどで候補を作る
ii) 直接影響を判定
そのクエリ単独で漏えるかを見る
iii) 組合せ影響を判定
既存ログに追加して新しいモザイクが成立するかを見る
Comparison
09 / 16
注意喚起は不十分
プロンプトだけ
「漏れないように検索しろ」といった後付けの注意喚起は、一貫して効かず、状況によっては性能低下や漏えい残存が起きました。
なぜ効きにくい？
“行動の最適化対象”が変わらず、クエリ生成の癖が残りやすい
対照
PA-DRはクエリ生成を学習で矯正する
Evidence
10 / 16
成否と漏えいを両立
数値で見る
PA-DRにより、厳格チェーン成功率は上がり、同時に答え/完全情報漏えいが大幅に下がったと報告されています。
厳格チェーン成功率
48.7% → 58.7%
answer / full-information漏えい
34.0% → 9.9%
Distinction
11 / 16
“どこが違うか”
漏えい削減の設計
重要なのは最終成否だけを見ることではなく、プランニングの各意思決定に「漏えい成立」を誤差として帰属させる点です。
従来寄り
性能報酬中心 → より有用な（具体的な）クエリへ
注意喚起
文章の規範 → 行動の学習対象になりにくい
PA-DR
クエリ段階で漏えいを評価しペナルティ → 行動が変わる
Closing
12 / 16
実務への示唆
再テストが必須
PA-DRは有力な方向性ですが、漏えい評価は分類器やベンチ設計に依存します。導入前に、自社の観測者モデル・ログ粒度・ログ対象範囲で intent/answer/full に相当する指標を再検証するのが安全です。
リスク点
分類器の推定が現実の攻撃者能力と一致する保証はない
別観点
検索以外（引用・要約・キャッシュ・頻度など）でも同様のモザイクが起き得る
