COVER
02 / 16
ボタンからコードへ / drift対策

NixOS × Incus 移行

Proxmoxクラスタをやめ、宣言的（declarative）な管理とCLI運用に寄せたホームラボ移行の要点です。
PROBLEM
03 / 16
state drift が増える

Proxmox 摩擦

UI中心の運用は、変更が“実運用”にだけ残り、宣言された状態と乖離（state drift）しやすいです。 トラブル時に「何をしたか」を思い出して再現する必要が出ます。
I. Drift
手作業の変更がコード側に反映されないまま蓄積しやすい。
II. 再現性
復旧手順が“思い出依存”になり、再現不能になる。
METAPHOR
04 / 16
状態はコードで“固定”

宣言的（declarative）運用

宣言的設定とは、状態を“テキスト（コード）”として管理し、再現性を高める考え方です（再現＝同じ構成が同じように立ち上がる）。
i. 管理
設定がGit等に集約される。
ii. 検証
git diffで差分・意図を追える。
iii. 再適用
別マシン／再インストールでも適用し直せる。
ARCHITECTURE
05 / 16
NixOS：ホストも宣言的

NixOS が土台

ホストOSの設定から運用の要点までを宣言的テキストで管理し、構成適用の再現性を高めます（“バイト単位”の感覚に近づく）。
I. 構成
ネットワークやサービス定義が説明付きで残る。
II. 回復
再インストール時に“コードを戻して実行”できる。
ARCHITECTURE
06 / 16
Incus：CLIで仮想化運用

Incus が器

NixOS上でコンテナ/VMをCLI（テキスト）で扱えるため、仮想化運用を“読み書き可能”に寄せられます。 永続サービスを安定運用しつつ、従来の“アプライアンス的制約”を減らせる点も評価です。
i. コンテナ
LXC系をCLIで運用。
ii. VM
必要な“レガシー用途”はVMで残す。
iii. CLI
手順がテキストとして追跡できる。
EVIDENCE
07 / 16
移行：LXCとVMで持ち物が違う

手順の“自然さ”

対象（LXCかVMか）で移行の持ち物が違うため、手順も分かれます。 それでも“追試可能”な形（具体コマンド）で移せたのが強みです。
i. LXC
vzdumpでダンプ→Incusへストリーム投入。
ii. VM
ZFS等→qemu-imgでQCOW2へ変換→Incusへ取り込み。
iii. 履歴
7年以上のデータ履歴を失わず移せた。
PROCESS
08 / 16
段階的に“恐怖”を消す

Incusで先に確認

物理切替の前に、物理機と同じ構成をIncus上のVMで動かして確認する戦略で、移行の不安を下げました。 再現性（検証可能性）を先に獲得する考え方です。
I. シミュレーション
物理構成を“VM化”して事前検証。
II. 切替
問題が出る前提で、切替のリスクを縮小。
COMPARISON
09 / 16
“実運用”をテキスト化

UI中心 vs CLI中心

UI中心では変更がブラックボックス化しやすい一方、CLI/設定テキスト中心では差分・検証・ロールバックがしやすくなります。 同じ変更でも“追跡できる形”に寄せられるのが決定的です。
I. UI中心
状態のずれ、手順の思い出し依存。
II. CLI中心
設定差分・検証・再適用がテキストで可能。
EVIDENCE
10 / 16
AIエージェントと相性

“読み書き可能”が鍵

AIエージェントが運用に入ると、テキストでの入力・変更・検証（例：git diff）が中心になりやすいです。 UI操作はブラックボックス化しがちですが、NixOS/Incusはテキストとして扱えます。
i. 入力
設定が文章として与えられる。
ii. 検証
差分・有効化の確認がしやすい。
iii. 安全
YOLO操作の暴走リスクを抑える方向に働く。
DISTINCTION
11 / 16
“Nixだけ”では完結しない

Incusの実務的な位置

理想はNix由来でコンテナ/VMをすべて揃えることですが、成熟度や機能面の事情で完全移行は難しい場合があります。 その代替としてIncusを採り、運用を堅牢に保ったという納得感があります。
I. 理想
Nixの宣言的管理で統一したい。
II. 現実
成熟度不足を補うためIncusを併用。
EVALUATION
12 / 16
強みと注意点

価値と限界

強みは“再現性・デバッグ可能性・透明性”が運用で効いてくる点で、移行手順まで具体的に追えることです。 一方で、効果が環境依存になりうること、AI運用のガバナンス設計が別途必要なことが注意点です。
i. 強み
評価可能：Git集約・差分確認。
ii. 実務
vzdump/qemu-imgで移行が手順化。
iii. 注意
ガードレール（権限・承認等）の具体は別途要確認。
CLOSING
13 / 16
今日の持ち帰り

宣言が運用を強くする

“ボタン操作中心”から“コード中心”へ移す理由は、state driftと再現不能を減らし、AI時代に必要なテキスト運用へ寄せるためです。 自分の環境でも「ドリフト原因」「安全設計」「復旧検証頻度」を確認すると判断しやすくなります。
