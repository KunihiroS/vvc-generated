LOOPER / ROUTER
02 / 15
モデルの背後にチーム
サービング層の協調
vLLM Semantic Router の Looper は、ユーザに「1つのモデル呼び出し」に見せたまま、内部で micro-agents 協調を実行し、最終的に OpenAI 互換の 1 応答として返します。
PROBLEM
03 / 15
受動→制御へ
ルータが“推論盤面”になる
従来は「アプリがモデル名を指定し、バックエンドがそのまま返す」が中心でしたが、今はルータ（Semantic Router）が control plane となり、どのモデルへいつ切り替えるかが品質とコストを左右します。
CORE IDEA
04 / 15
単なる配線ではない
能力構築（capability）
ルータはモデル選択だけでなく、呼び出し内部で協調パターン（Looper）を選び、モデルの重みを変えずに推論品質を上げる余地になる、と主張します。
UX PROMISE
05 / 15
外は 1 回、内は多段
1 API に協調を隠す
ユーザは `vllm-sr/auto` のように 1 つのモデル名だけ指定します。裏側ではファンアウト、quorum、検証、不一致統合、契約修復まで行い、最終的には通常の 1 応答を返します。
LOOPER RUNTIME
06 / 15
スローガンでなく制御
予算・失敗・トレース
Looper は実行ランタイムであり、停止条件・並列上限・合成・フォールバック・トレース（観測ラベル）などの統治が前提です。無制限のエージェントグラフではありません。
停止 / コスト（予算）
Confidence 閾値・並列上限・最大ステップ等で制御
運用 / デバッグ
trace を保持し、どこで失敗しなぜ統合したかを追跡
RECIPES
07 / 15
代表パターン 5 つ
Looper のレシピ
協調パターンは 5 系統が中心です（Confidence / Ratings / ReMoM / Fusion / Workflows）。それぞれ停止・合成・契約順守・役割分担の仕方が違います。
i
Confidence
閾値でエスカレーション停止
ii
Ratings
並列アンサンブル（max_concurrent）
iii
ReMoM
合成を contract 形式へ統合
iv
Fusion
不一致を証拠化して統合
v
Workflows
planner 選択の worker を許可制で実行（境界あり）
DECISION
08 / 15
万能ループはない
タスク形状に合わせる
最適なループは普遍ではなく、benchmark が求める出力形式や安定性の性質に応じてレシピを切り替えるべき、という評価観点が提示されています。
例
GPQA-Diamond / LiveCodeBench / Humanity’s Last Exam / SWE 相当で要求が異なる
結論
信号から推定し、レシピを変えて勝ち筋を作る
EVIDENCE
09 / 15
scorecard は補助線
閉モデルで良スコア
記事では scorecard（例：92.6 など）を「タスクに応じたレシピ選択」の有効性の一部として紹介します。ただし、解釈には注意が必要です。
示唆
router-owned collaboration が“より強いモデル同一性”になり得る
注意
全リクエストで全閉モデルを回すべき、という意味ではない
OUTPUT CONTRACT
10 / 15
“文”だけじゃない
出力契約（output contract）
Fusion / ReMoM の文脈では、自然文の良さだけでなく「特定フォーマット」「実装可能性」「誤差耐性」などの契約を守ることが狙いです。
例
ANSWER: X のような厳密フォーマット（想定）
手段
協調過程で合成・修復し、最終的に契約順守へ収束
DESIGN CHECK
11 / 15
導入前の論点
レシピ設計が鍵
自社導入では、信号設計・閾値・モデルプール・失敗時 UX・監視指標まで整備が不可欠です。特に「ループ寄与」と「周辺選択寄与」を切り分けられる設計が重要になります。
i
信号設計（難しさ / リスク / 契約圧力等）
ii
制御パラメータ（閾値 / 並列上限 / 最大ステップ）
iii
観測（trace / ラベル）と品質・コストの監視
