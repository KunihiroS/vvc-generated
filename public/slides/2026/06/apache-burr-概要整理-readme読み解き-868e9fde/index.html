Apache Burr
02 / 16
LLMより先に“状態”
状態機械で 運用 する
Apache Burr は、LLMを含む意思決定アプリを「状態・追跡・永続化」込みで作りやすくするフレームワークです。
本整理は README に相当する内容（提供テキスト）を読み解き、何ができ、他と何が違い、導入で何に注意するかを短くまとめます。
Problem
03 / 16
ブラックボックス化
なぜ 分からない のか
LLMアプリは「状態がどこでどう変わったか」が追えず、中間判断や再実行が難しくなりがちです。
現象
分岐の理由（なぜその出力？）が曖昧
影響
デバッグ・評価・再現性・改善が遅れる
Burrはこのギャップを「状態機械＋可視化＋永続化」で埋める方向性です。
Metaphor
04 / 16
道案内
実行を 地図化 する
Burrはアプリをグラフ/フローチャートとして組み、各ステップを宣言的な関数で繋ぎます。
node
処理（action）
edge
遷移（transition）
state
変数（例：履歴）
これにより「循環（人↔AI）」のような複雑な流れを構造として管理できます。
Architecture
05 / 16
中心は“状態機械”
@action で組む
各ステップを `@action` のように宣言し、状態の読み取り/書き込みと遷移で制御します。
reads
参照する状態を明示
writes
更新する状態を明示
状態が絡む判断ロジックを、LLM呼び出し以上の“データフロー”として扱うのが要点です。
Evidence
06 / 16
何が起きたか
UIで トレース
READMEではUIサーバを起動し、実行の様子やデモチャットの変化をリアルタイムに追えるとされています。
見るもの
入力 → 中間状態 → 出力
効く場面
デバッグ、品質改善、再現性
「なぜそうなったか」を、ログではなく実行の流れとして可視化する狙いです。
Core
07 / 16
Burrの要点
3つの 統合
i
状態機械でアプリを表現（action＋transition）
ii
UIによる実行テレメトリ可視化（監視・デバッグ・トレース）
iii
プラガブルな永続化（persisters）で保存/復元を扱う
単なる“モデル呼び出し支援”ではなく、運用上の穴（追跡できない/再実行できない）を埋めに行く設計です。
Process
08 / 16
状態の流れ
書くのは “更新”
READMEの雰囲気では、初期状態を与え、更新し、遷移で次の処理へ進めます。
初期化
with_state(chat_history=[])
更新
state.update(...) / append(...)
遷移
分岐条件で次の node へ
例として `chat_history`、`prompt`、`response` のようなアプリ内変数が“状態”として扱われます。
Comparison
09 / 16
分類の軸
“LLM以外”も 同じ
Burrはチャットボット以外にも時系列シミュレーションやハイパラチューニング等への適用を明記しています。
LLM中心
意思決定・エージェントの制御
非LLM中心
状態を持つ一般処理（シミュ/最適化）
フレームワーク依存の“モデルの作法”より、状態機械で組み立てる思想が前に出ています。
Distinction
10 / 16
できること/できないこと
結び付けるのは “運用”
Burrはモデルの作り方やAPI呼び出しを固定せず、状態管理・追跡・永続化の枠組みに重点があります。
Burrが主に担う
状態機械、reads/writes、UIトレース、persisters
Burrが決めない
モデル実装、API設計、データ管理の中身
その分、既存スタックへの統合余地はある一方、“正しい統合の仕方”はチームの力量に依存します。
Operational
11 / 16
運用へ寄せる
ロードマップは 実務寄り
FastAPI統合、ホステッド実行、リトライ/例外管理、ストレージ連携、Ray/modal等との統合が挙げられています。
開発→実運用
デプロイ/保守を見据えた拡張
品質要素
再実行、例外、追加ストレージを段階的に強化
ただし、性能や安全性、永続化の詳細、権限設計などは今回の情報範囲では十分に確定できません。
Caution
12 / 16
導入判断の注意
“ログ/状態”は 増える
状態やテレメトリが増えると、機密情報の取り扱い・ログ保持・コストがリスクになり得ます。
性能
UI/トレース/永続化でレイテンシ増の可能性
安全
PII/機密のログ出力・保存ポリシーが必要
運用
ストレージ容量、整合性、障害時の振る舞い要検討
結論：PoCで非機能要件（性能・安全・運用設計）を先に検証するのが重要です。
Closing
13 / 16
まとめ
状態機械×可視化×永続化
Apache Burrは、LLMアプリを“状態を持つ処理”としてモデル化し、追跡と保存/復元までを運用可能な形に寄せます。
向くケース
状態遷移が複雑／デバッグ・評価の再現性が必要
確認したい点
性能・コスト・安全性・永続化の整合性・権限制御
（参考：類似カテゴリは比較表で方向性理解に有用。ただし最終判断は用途・運用体制で決まります。）
