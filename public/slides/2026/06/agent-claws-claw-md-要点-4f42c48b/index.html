COVER
01 / 10
定期実行を“標準化”
Agent Claws
（CLAW.md）
AIエージェントが無人で定期的に行う仕事を、1つのCLAW.mdで記述し、runnerがその順序で実行できるオープン標準です。
（日本語：エージェントクロー／クロー）
PROBLEM
02 / 10
課題は“個別実装”
定常業務が増える
メール整理・レポート生成・監視/復旧など、毎回ほぼ同じ手順を「定期実行＋判断つき」で回したいのに、作り方が個別最適になりがちです。
その結果、運用が属人化し、移植も難しくなります。
欲しいこと
同じ手順を
どこでも動かす
困ること
設定分散で差分が崩れる
（属人化・破壊的更新）
METAPHOR
03 / 10
“指示書”としてのCLAW
MD＋YAML
CLAW.mdは、YAMLフロントマター（メタ情報）とMarkdown本文（タスク定義）に分かれます。
runnerはファイルを読み、本文のタスクを上から順に実行します。
frontmatter
version / name / description
schedule / runtime など
本文
見出し＝タスク章
実行手順を記述
runner
順序どおり実行
必要なツール/CLIへ接続
ARCHITECTURE
04 / 10
順序実行が効く
出現順に実行
タスクはMarkdownの記載順で実行されます。
前工程の結果が後工程へ影響する運用（救出→可視化→返信など）を安全に組み立てやすいです。
例：inbox-tidy
①迷惑判定の補正
②誤って隔離された正規の救出
③返信が必要なスレッドの可視化
狙い
順序が担保されることで
“文脈の連鎖”が壊れない
DISTINCTION
05 / 10
cronではない
推論つきの手順
cronは基本的に固定コマンドを定期実行しますが、CLAWは推論するエージェントの手順をジョブ定義として書きます。
状況（コンテキスト）を見て判断し、その結果で行動します。
cron
毎回同じ実行（例：コマンド1本）
Agent Claws
状況→判断→実行（agentの手順）
（日本語：推論／判断）
EVIDENCE
06 / 10
system_promptで安全運転
ガードレール
無人実行で最重要な誤操作を抑えるため、system_promptに行動制約を書けます。
例では「確信がない場合は動かさない」「削除しない」などが方針化されています。
例：inbox-tidy
不確実なら動かさない
削除しない
誤分類リスクを下げる
価値
“無人稼働”の最大の懸念（誤操作）に直接効く
ENUMERATION
07 / 10
書き方の要点
最小要素
CLAW.mdには最低限のメタ情報に加え、タスクを示す見出し（章）を複数置きます。実行制御として schedule / runtime / system_promptなどが使われます。
必須
name / description
（version 前提）
実行制御
schedule（例：every 5m、daily @ 07:00）
タスク
見出し単位で複数定義
agent or bash
PROCESS
08 / 10
runtimeの分岐
agent / bash
runtimeは主に2系統で考えます：runtime: agentは推論してツールで実行し、plain bash stepは定型処理を行います。実務では「判断が必要→agent、単純変換→bash」が堅牢です。
runtime: agent
検索・判断・実行
（日本語：推論型）
bash step
取得/変換など定型作業
推論を介さない
DISTINCTION
09 / 10
懸念は仕様の“穴”
安全性の検証が必要
提示情報だけでは、確信判定の具体（閾値/アルゴリズム）、失敗時のリトライ/ロールバック、ログ・監査、権限管理が明確ではありません。
そのため、同じCLAW.mdでもrunnerとホスト設計でリスクが変わり得ます。
不明/不足になりやすい
確信判定の原理
失敗時の整合性
監査・ログ仕様
権限（操作可能範囲）
運用で必要
段階導入（初期は手動確認）
ガードレール（制御/通知）
権限制御の事前確認
CLOSING
10 / 10
結論：方向性は良い
“言語”としての価値
Agent Clawsの魅力は、定期実行を単一のCLAW.mdで標準化し、ポータブルにすることです。
一方で、期待した信頼性は「周辺の安全設計（runner/ホスト）」の確認次第になります。
強み
Markdown＋YAMLで読みやすい
diffしやすい運用
ジョブ資産化の方向性
次に確認
確信判定・失敗時運用
ログ監査・権限設計
段階導入と復旧導線
