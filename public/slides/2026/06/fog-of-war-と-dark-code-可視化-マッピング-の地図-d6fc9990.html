霧の中で迷う
Fog of War で見る 暗黙 のコード
ソフトウェア組織では、見えている間は速く進むのに、離れると霧に戻る。 その“見えない領域”が dark code を増やし、意思決定と復旧を壊します。
problem
02 / 15
状況認識の喪失
Fog が増える 理由
戦場では偵察が止まれば霧に戻り、敵の動きは推測になります。 組織でもコードベース、依存関係、技術負債、知識の属人化が“見えない霧”として残ります。
見えている間
変更の理由・影響・復旧手順が理解可能
離れた後
全体像が消え、説明と修正が難しくなる
architecture
03 / 15
定義から
dark code とは
人間が読んだ・レビューしたことがないコード、またはレガシーで全体像が把握できない“灰色の箱”です。 AIで加速しやすいだけで、本質は理解可能性（epistemic control）の低下です。
AI以前から
レガシーで追えない領域＝不可視の領域
AIで顕在化
速度上昇が“理解の追いつき”を物理的に破る
evidence
04 / 15
レビュー可能性が崩れる
大量PRが作る 霧
例：フロントエンド改修で 211ファイル、7000行超が 1つのPRとして出ると、差分追跡が成立しにくくなります。 これは“面倒”ではなく、後から説明・修正できる状態を壊します。
従来の前提
人間が差分を追い、行単位で理解する
崩れる結果
復旧時に判断できず、epistemic debt が増える
comparison
05 / 15
外と内で違う
Fog は武器にも 毒にも
外側では交渉力になりますが、内側では危険です。 CEO/CPO/VPが同じ技術スタックに別の前提を置くと、制約を理解しない方針変更や統合が起きます。
外部
見えないことで有利に交渉できる
内部
可視性不足が技術失敗だけでなく損失・政治にも連鎖
enumeration

必要な観点
“見る”ための軸 5つ
単なるダッシュボードではなく、意思決定に使える可視化へ落とす必要があります。 見るべき観点を 2層で整理します。
コード健全性
複雑さ／健康状態／変更頻度（生きている場所）
組織リスク
バスファクター（属人化）／どこが復旧不能になり得るか
metaphor
07 / 15
見張り（watchtowers）
Scouting と 地図化
トップ企業が“速さ・主導権・効率”を両立するには、内部のフィードバックループに加え調査（scouting）と見張りが要ります。 その比喩が、コミットと依存関係を統合して“地図”にする発想です。
Scouting
Gitデータから全体を再構成する
Map
複雑さ・属人化・変更偏りを重ね表示
Action
意思決定の言語へ翻訳して実行へ接続
process
08 / 15
地図ができるまで
統合 → 重ねる → 翻訳
Comperは Git データからソフトウェア全体を統合し、複雑さ・健康状態・バスファクター・変更頻度などを地図として可視化します。 その上で CEO に届く形へ翻訳します。
i. 統合
リポジトリや情報を“全体”として接続
ii. 重ね表示
リスク地帯と変更の偏りを同時に見る
iii. 翻訳
ロードマップと依存関係に結びつけ説明する
evidence
09 / 15
赤い領域（Red zone）
“なぜ遅いか”を 実証
赤い技術的リスク地帯を、機能開発のロードマップや依存関係と結びつけます。 例：どこを切り抜ける必要があり、時間とコストがなぜ増えるのかを示します。
Before
「難しい」で終わり、説明責任が曖昧
After
機能単位に紐づく負債と依存で根拠を作る
distinction
10 / 15
KPIの組み替え
LOC より サイクルタイム
LLMが瞬時に大量コードを生成すると、LOCや“生産時間”は Goodhart の法則で崩れます。 視覚的システムマップ上でフィーチャーの実行サイクルを追い、実効的な効率を捉えます。
危険な指標
LOC（コード量）＝生成が増えるだけで改善を反映しない
効く指標
フィーチャーのサイクルタイム（Feature execution loop）
closing
11 / 15
万能ではない
地図は真実を 置換しない
Git 可視化は強い一方、実行時設定・インフラ状態・外部依存・手動運用・非Git管理の変更までは完全に追えません。 だから「地図を前提に、運用とガバナンスも設計する」必要があります。
前提
地図が“正確な真実”にならない可能性を織り込む（誤検知・欠落）
必須
レビュー権限・承認プロセス・障害時責任範囲・教育とKPIを一体化
references

参照
この講演での 要点
下記は、今回の要約で扱った概念と論点です（出典詳細は原講演資料に従ってください）。 AI時代の可視化は、dark code（理解の喪失）とepistemic debt（説明不能の負債）を抑えるための“地図”です。
Key terms
fog of war（戦場の霧）／dark code（暗黙のブラックボックス）
epistemic debt（説明不能の負債）／bus factor（属人化）
Concrete ideas
Git統合マップ化（Comper）／Red zone をロードマップへ接続／
LOCではなくFeature cycle time（サイクル）で評価
CEO/役員向けには、リスク地帯を“機能開発のために切り抜けるべき領域”へ翻訳し、時間と費用の増加理由を実証する。 それが可視化の価値（マップの役割）です。
