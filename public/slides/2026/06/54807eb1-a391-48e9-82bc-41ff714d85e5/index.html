Cognitive Debt / ROI
02 / 16
CTOが見た現場の転換
ROI
AI導入は「使うほど得」から「価値を測り、検証し、管理する」へ移った。鍵は認知的負債（cognitive debt）が“新しい技術的負債”として膨らむ構造を理解することです。
Problem Shift
03 / 16
魔法は終わった
投資の説明
2年前は「AIに投資」で通ったが、今はCFOを巻き込んでROIを説明する必要がある。 しかしAIコストはトークン（利用量）で変動し、固定の投資額として扱いにくいのが現実です。
Core Concept
04 / 16
速さの裏側
cognitive debt
コードを書く速度は上がっても、品質・保守性・検証の甘さが後から問題化する。結果として「書いたのに出荷していない」「恒久運用でない前提が残る」負債が積み上がります。
Structural Bottleneck
05 / 16
ボトルネック移動
レビュー
ボトルネックはコード生成ではなく、レビューと判断にある。CI/CDは加速できても、正確性・安全性・設計整合性の確認は同じ速度で増やせません。
Strategy
AIでPRのリスク評価 → 人間は重要箇所へ集中
Mechanism
レビューの優先順位付け（トリアージ）で検証を最適化
Foundation First
06 / 16
測れる土台
ベースライン
ROIを追い過ぎる前に、現状の使用量・生産性・品質のベースラインを作る。標準化（同じものをより賢く使う）へ切り替えるのが、再現性を持つ近道です。
Usage
現状の使用量
Output
生産性（例：リードタイム）
Quality
品質（例：不具合率/手戻り）
Verification Infrastructure
07 / 16
信頼の積み上げ
検証基盤
AIコードへの信頼は「コンパイラを信じる」ように積み上げる必要がある。feature flag（段階的有効化）、観測性（observability）、受け入れテストが前提です。
Feature gates
feature flagで影響範囲を制御
Evidence
観測性・受け入れテストで挙動を確かめる
evals
08 / 16
評価を先に
evals
evals（評価）を用意しておくと、モデル変更が起きても同じ基準で品質を確かめられる。特に“導入”より“検証基盤”のコストが高い点が、今から作る理由です。
Stability
モデル差し替えでも品質を比較可能に
Governance
品質・安全を意思決定の共通言語にする
Organization Design
09 / 16
役割の再定義
採用/評価
AI時代の“エンジニア”は一枚岩ではない。プロダクトを出す人と、可用性・予算・設計責任を持つ人は別で、採用・評価制度もそれに合わせて再設計が必要です。
Decompose
曖昧な課題の分解
Review
コードレビュー能力
Value
チームで価値創出を設計
Quality / Safety
10 / 16
速さが危ない
出荷と保守
生成が速くても、出荷・保守・検証が同じ速度で速くならない。だからこそ“安全対策は後回しにしない”設計が要点になります。
CI/CD
検証設計（テスト/観測）を本丸に
Risk
AI出力の増加ほどレビュー価値が上がる
Risk / Lock-in
11 / 16
依存の現実
ベンダーロックイン
価格改定、障害、供給停止、モデル提供終了などは事業継続リスクになる。対策はモデルだけでなく周辺依存まで見て、差し替え可能性を用意することです。
Abstraction
汎用モデルAPI上に社内ラッパーを構築
Continuity
移行難所（スキル/ツール/ワークフロー）も先に設計
Human Factors
12 / 16
人の負荷
心理・動機
AIでフィードバックループは短くできる一方、エンジニアを“監視係”に押し込むとメンタルヘルスに影響し得ます。重要なのは速度よりも、検証と判断の設計を人が受け止められる形にすることです。
cognitive debt（認知的負債）は、品質だけでなく認知負荷・評価不全も引き起こします。
