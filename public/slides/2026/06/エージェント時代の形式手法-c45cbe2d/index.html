02 / 16
Cover
形式手法の再評価
agent時代
エージェントによるコーディングが広がると、形式手法の価値評価は「高コストの特殊手段」から「普遍的に役立つ道具」へ移る可能性があります。
03 / 16
Problem
なぜ今、論点がずれる
費用対効果
従来の形式手法は数学的な証明を要し、コストが極めて高かったため、一般的な開発では費用対効果が合いにくいと見なされてきました。
形式手法
仕様（spec）を満たすことを数学的に証明
型システム
軽量な形式手法として振る舞いの保証を狙う
04 / 16
Evidence
“重い”代表例
seL4
seL4は厳密に検証されたマイクロカーネルとして有名ですが、25人年で8,700行のCを検証し、証明作業の負荷も非常に重い例です。
成果
厳密検証されたマイクロカーネル
コスト感
25人年 / 8,700行のC + 大量の証明作業
だからこそ「最重要システムですら、一般開発で同等の費用対効果は成立しにくい」という懐疑が合理的でした。
05 / 16
Architecture
転機は二つ
コスト と 利益
エージェント時代は状況を変え得ます。変化は大きく「コスト側」と「利益側」の2系統です。
i. コスト側
モデルが仕様理解・証明の下準備を支援
ii. 利益側
リリース可能品質までのギャップが検証ボトルネック化
06 / 16
Process
コスト構造の変化
支援で下がる
エージェント単独で「完全自動に任意の難しい証明」を作るわけではないとしても、モデル支援で人間の作業量を大きく減らせる可能性があります。
仕様理解
言語化・形式化の壁が低下（例：制約の整理）
証明の下準備
補題・手順のたたき台を作りやすく
人間の手間
「やるべき最終責任」に集中しやすい
07 / 16
Problem
品質の壁が増える
ギャップ
モデルが有用なコードを出せても、リリース可能な品質へ到達するにはギャップが残ります。特にエージェント生成コードは「スロップ（雑さ）」になりがちです。
頻出リスク
複雑すぎる / 変なバグ / コーナーケース混入
重要な失敗
コードベースの必須不変条件を守れない
08 / 16
Metaphor
レビューの労を圧縮
検証ボトル
最終的に人間が時間をかけて検証・レビューする必要が出るなら、形式手法はその負担を圧縮して「レビュー効率」を上げる道具になり得ます。
現状
生成物の品質が揺れる → 追加レビューが必要
期待
型・形式手法で不変条件の逸脱を早期に排除
09 / 16
Comparison
テストとの違い
∀保証
テスト（プロパティベーステスト、ファジング等）は重要ですが、探索できない状態空間が残ります。型は「∃ではなく∀に近い確実性」を提供し得ます。
テスト
到達したケースに依存する（全部は探索できない）
型・形式手法
条件を満たすことを普遍的に狙う（∀）
例：データ競合の排除、特定クラスの脆弱性の根絶に近い確実性。
10 / 16
Enumeration
フィードバックが効く
学習 の 反復
エージェントはフィードバックを必要とし、学習でもコーディングでも性能が押し上げられます。型システムと形式手法は強いフィードバックとして機能し得ます。
i
学習のフィードバック（正誤の指標）
ii
コーディングのフィードバック（型エラー等）
iii
探索の効率化（探索できない領域を減らす）
11 / 16
Evidence
なぜジェーンストリート
言語統制
ジェーンストリートが形式手法へ注力する背景として、言語への深い統制があり、型や証明技術を運用に合わせて調整できる点が挙げられます。
調整できる強み
言語・型システムを証明志向に合わせて設計
利用者の準備度
型機能を使う人が多い → フィードバックが回る
外部PLコミュニティ（Lean、Dafny、Coq、Agda、Iris等）とも連携。
12 / 16
Distinction
期待の裏の注意点
不確実性
ただし、エージェントがどの程度まで低コスト化するかの定量情報は文章から読み取りにくく、形式手法の導入・学習・仕様化が別のボトルネックになる可能性もあります。
未確定
どの仕様範囲・どの形式手法で、どれだけ削減できるか
別コスト
証明や仕様の表現が開発体験に噛み合う必要
「自動化の程度」と「人間が担う部分」と「仕様化コスト抑制」が勝負。
13 / 16
Closing
結論
普遍的道具へ
エージェントでコストと利益の構造が変わるなら、形式手法はレビュー効率と普遍保証（∀）の両面で、より「実務に効く」方向へ移行しうる、という主張です。
i. コスト
モデル支援で証明・仕様の下準備が軽くなる
ii. 利益
品質ギャップが検証ボトルネック化する
iii. 確実性
テストの外を、型で普遍保証に近づける
読む前の鍵：型による保証（∀に近い）と、生成コードの品質課題（不変条件の逸脱）。
