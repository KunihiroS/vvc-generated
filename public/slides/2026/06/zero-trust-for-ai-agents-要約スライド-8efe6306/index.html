Zero Trust / AI agents
02 / 16
AIを禁止しない
統制を設計する
Anthropicの「Zero Trust for AI agents」を、実務で迷わないための要点として整理します。
中心は「AIエージェントを主体として台帳化し、侵害時の被害を限定する」ことです。
Problem
03 / 16
前提が崩れる
時間が短縮
攻撃も防御も「時間感覚」が変わり、静的・月次運用では追いつきません。
フロンティアAIにより、攻撃（悪用）までの時間が短くなります。

Before

リスクが数か月単位で進行（想定しやすい）

Now

数時間単位で進行（前提・検知依存が危険）

Core idea
04 / 16
チャットではない
主体として扱う
AIを“入力に従う便利機能”として見ず、エージェントを新しい主体（entity/主体）としてゼロトラスト管理に加えます。
統制の焦点が「内容」から「誰が・どんな権限で・何をしたか」へ移ります。
Foundation
05 / 16
入口で強く
Foundation要求
AIエージェントの入口で、最低ライン（Foundation）を上げます。短命トークン、暗号学的に検証可能な固有ID、Deny by Default、エージェント単位の隔離などが中核です。

固有ID

cryptographically verifiable（暗号検証可能）な agent_id

固有ID（固有の身元）

短命化

short-lived token（短命トークン）

有効時間を短く

隔離

Deny by Default & agent isolation

拒否前提・分離
Entity ledger
06 / 16
台帳化する
管理対象
エージェントを、owner/allowed_tools/credential/approval_policy/memory_scope のように管理対象として棚卸しします。
“何を言ったか”より、“何を実行する権限を持つ主体か”を統制します。

主体

agent_id / owner

権限・経路

allowed_tools / credential / approvals

判断材料

memory_scope

出力統制

出力フィルタリングと検証可能性の確保

Distinction
07 / 16
一枚岩でない
Promptだけ不足
プロンプトインジェクション対策だけでは不十分です。攻撃は“実行経路”と“判断材料”を汚染して多層化します。

Tool Poisoning

ツール定義/入力の汚染で挙動を逸らす

Tool Chaining

複数ツールを連鎖して目的を達成

Memory Poisoning

メモリの汚染で誤った判断を誘発

NHI拡大

Non-Human Identity（人間以外の主体）が増える

Attack surface
08 / 16
経路が攻撃になる
実行経路を管理
攻撃面は「入力文章」だけでなく、ツール呼び出し、エージェント間通信、サプライチェーン（モデル・RAG・MCP等）まで広がります。
“どこで何が信用されるか”を明確にします。

Tool

MCP / API / 内部SaaS

Memory

retrieval / storage

Supply

model / RAG / MCP server

Design rule
09 / 16
摩擦では足りない
Impossible寄り
対策は“面倒化（tedious）”より、“成立しにくい設計（impossible）”を重視します。反復コストが下がるため、摩擦依存は中核防御として数えにくいからです。

tedious

レート制限などの摩擦：補助的に使う

impossible

ID/権限/分離/検証で“通らない”設計へ

Operations
10 / 16
人は残す
SOARは一次まで
Agentic SOARは一次トリアージまでが現実的です。証拠収集・ログ相関・調査文書化は自動化し、封じ込め最終判断や外部開示は人間承認に残します。

自動化

一次調査・証拠収集・影響候補提示・文書作成

承認

封じ込め・顧客通知・法務/規制対応（人間）

Foundation→上へ
Foundationを土台に、運用成熟度に応じて Enterprise / Advanced へ段階的に引き上げます。

Advanced

移行

Example
13 / 16
事故でも止める
被害限定
侵害・誘導・誤作動が起きても、エージェント単位の隔離、Deny by Default、監査可能ログにより被害範囲を限定します。
“検知してから対応”に寄り過ぎない設計が狙いです。
入力
入力検証（不正な意図・命令の入り口を狭める）
実行
allowlist（許可ツール）と短命資格情報（credential）で制御
監査
包括ログ＋トレースで“誰が何をしたか”を追跡
Start here
14 / 16
最短の出発点
台帳→権限→承認
エージェント導入前に「人間ID・端末・SaaS・NHI・AIエージェント」を同じ台帳で棚卸しし、権限を絞り、ログを残し、高リスク操作に人間承認を残すことが出発点です。

i 台帳

全部入りで棚卸し（identity inventory）

ii 権限

最小権限（Least Agency）へ絞り込み

iii 承認

高リスクは人間承認（封じ込め等）

Closing
15 / 16
結論
禁止ではない
「AIを禁止する/しない」ではなく、侵害・誘導・誤作動を前提に、主体・権限・経路・ログで統制し“被害限定”へ落とすのがAnthropicの要旨です。
まずは台帳と最小権限から始めてください。
