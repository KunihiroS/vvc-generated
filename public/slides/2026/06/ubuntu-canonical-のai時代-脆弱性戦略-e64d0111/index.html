COVER
02 / 18
AI時代の脆弱性対策
Ubuntu
Canonical（Ubuntu）の戦略は、脆弱性の“発見量の増加”を前提に、優先度・封じ込め・長期保守・CVD遵守を組み合わせて被害を最小化します。
キーワード：CVD / defense-in-depth / リスク評価 / AppArmor・LXD / Ubuntu Pro / Livepatch
PROBLEM
03 / 18
“見つからない”は幻想
前提
未知の欠陥は必ず存在し、AIにより脆弱性発見が“大量・高速”になるほど、対応能力（トリアージ/修復）がボトルネックになり得ます。
変化
報告が増え、短時間に集中しやすい
リスク
単発のパッチでは回らず、優先度判断が崩れる
ARCHITECTURE
04 / 18
防御を段階化
DEPTH
対応の核は defense-in-depth（多層防御）：パッチだけに依存せず、封じ込め・軽減策・運用継続までを同じ設計として扱います。
一次
脆弱性の修正（迅速なパッチ提供）
二次
隔離（AppArmor / LXD / snap）
三次
運用の継続（Ubuntu Pro / Livepatch / ワークアラウンド）
PROCESS
05 / 18
“量”に耐える運用
TRIAGE
AIで報告が増えても処理を落とさないため、インシデント対応チームの準備（トリアージと修復の運用設計）を拡張する方針です。
受ける
CVE/CVD枠組みで情報を受領（報告が増える前提）
捌く
優先度・影響・悪用可能性で分類（捨てる/急ぐを決める）
直す
パッチ提供＋代替軽減策（必要時に緊急リリースも）
確認
隔離状況・適用状況を踏まえ再評価
DISTINCTION
06 / 18
スコア至上主義を回避
RISK
CVSSの数値だけで順位を決めず、対象ユーザへの影響・悪用可能性・環境の広がりで優先度を設計します（“守るべき場所”へ資源集中）。
見ない
CVSS単独の優先度固定（ノイズ増に弱い）
見る
実脅威影響：影響範囲・悪用の現実性・運用条件
EVIDENCE
07 / 18
“効く確率”を下げる
ISOLATE
たとえ新たに武器化されても、カーネル境界・サンドボックスで被害範囲（blast radius）を抑える方針が強調されています。
AppArmor
権限/操作を制限して横展開を抑える
LXD
コンテナ/環境分離で被害を局所化
snap
封じ込め（confinement）で実行経路を制御
PROCESS
08 / 18
“パッチ遅延”の現実
MITIGATE
修正が間に合わない期間を前提に、補償的統制（補償措置）を計画し、運用者が導入手順を立てやすくします。
プロファイル
AppArmorで緩和（workaround提示を含む）
分離
snap/LXDの制限で影響を縮小
リブート回避
Kernel Livepatch（復旧を止めない選択肢）
可視化
CVEトラッカー上で実行可能な軽減策を提示
注意：これらは「パッチの代替」ではなく、時間差を埋める統制として扱う。
ARCHITECTURE
09 / 18
責任ある開示
CVD
Coordinated Vulnerability Disclosure（CVD）を守り、エンバーゴや同時公開（day zero）などの枠組みで、情報の出し方を管理します。
公平性
報告主体に関係なく、技術的妥当性と環境リスクで評価
透明化
USNでの情報提供や、データ配信（OSV/VEX/OVAL等）を通じた追跡性
COMPARISON
10 / 18
“特別扱い”しない
NoSP
Project Glasswingのような特定プロジェクトに“肩入れ”せず、早期アクセスを特別ルート化しない姿勢が明記されています。
やらない
特定AIベンダー/プロジェクトへのアウト・オブ・バンド更新
やる
CVD同じ枠組みで、CVE指定（CNAとしての役割）も含め調整
Bilingual（用語）: CVD = 責任ある共同開示（coordinated vulnerability disclosure）
ENUMERATION
11 / 18
長期運用で差をつける
LIFE
脆弱性が“古いもの”まで含まれ得るため、単発の修正ではなく、最大15年のセキュリティ保守（Ubuntu Pro）で運用者の負担とリスクを下げる方針です。
Ubuntu Pro
最大15年のセキュリティ保守
Livepatch
リブート不要の修正選択肢
継続性
“当て続けられる”体制で被害時間を短縮
METAPHOR
12 / 18
優先度は地図
MAP
CVSSは天気予報に近いが、実際に“壊れる場所”は地形（資産・環境・悪用の現実性）で決まる。Canonicalはそれで優先度を描き直します。
天気
CVSS（技術的な可能性の指標）
地形
影響範囲・対象の広さ・悪用の現実性
COMPARISON
13 / 18
強みと弱み（検証視点）
BALANCE
方針は“良い方向”だが、AI活用の誤検知率・監査可能性・KPI（実測）不足があり、運用品質の裏付けは追加確認が必要です。
強み
プロセス＋技術＋運用を統合し、CVDと優先度設計を両立
懸念
AI推奨の検証手順・再現性・是正の運用指標が具体化されにくい
EVIDENCE
14 / 18
安全性は“指標”で決まる
KPI
AI時代は情報量が増える分、修復の実測、軽減策の有効性、検証プロトコル、そして公開データ（USN/OSV/VEX/OVAL等）の整合性が重要になります。
要確認
修復までの実測データ / 軽減策の有効性
要確認
AIの誤判定からの是正 / 優先度取り違えの管理
要確認
公開データの整合（USN・OSV・VEX・OVAL）
効果
“効率化”が安全性に悪影響しない根拠
CLOSING
15 / 18
まとめ
Takeaway
Canonicalのポイントは、脆弱性が必ず見つかる前提で、CVD・多層防御・リスク基準の優先度・隔離と代替統制・長期保守を一つの設計にすることです。
責任ある開示
CVD遵守で信頼性と追跡性を確保
現実の脅威
CVSS単独でなく“影響×悪用可能性”で判断
被害を局所化
AppArmor/LXD/snap＋Livepatch＋Ubuntu Pro
Bilingual（用語）: defense-in-depth = 多層防御（多段階の安全策）
