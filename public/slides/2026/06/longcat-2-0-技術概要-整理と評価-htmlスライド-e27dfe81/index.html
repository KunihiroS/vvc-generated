LongCat-2.0 を読む

技術概要

LongCat-2.0の特徴を、MoE・長文注意・N-gram・ASIC運用・後段学習まで一続きで整理し、検証の限界も含めて評価します。
対象：公開資料ベース。用語の要点（MoE、attention/KV、LSA、AI ASIC superpods）を前提に、1メッセージ/1スライドで要点だけ濃縮します。

03 / 16
長文 × エージェント

難所

1Mトークン級の長文では、attention計算とKVキャッシュ、さらにprefill/decode時のI/O・通信が支配的なボトルネックになります。
attention負荷
候補選択（indexing）が律速になり得る。予測可能なメモリアクセスが鍵。
MoEの制約
活性化（どのエキスパートを使うか）とロードバランスが品質と計算量を左右。
計算基盤
GPU中心ではない AI ASIC superpods で学習・推論を成立させる必要。
後段学習
エージェントの実行・対話品質・安全性を、統合的に整える工程が必要。
//Architecture

全体像は 3層

統合設計

LongCat-2.0は、モデル（LSA/N-gram/MoE）・推論最適化（prefill/decode配備）・後段学習（複数エキスパート）を一体最適化する設計です。
モデル
MoE＋LongCat Sparse Attention（LSA）＋N-gram Embedding
推論
speculative decoding / MTP と分離並列戦略（TTFT/TPOT意識）
後段学習
Agent / Reasoning / Interaction Experts を MOPD で統合
//Enumeration
05 / 16
規模と活性化

MoE仕様

LongCat-2.0はMoEで、総パラメータ1.6兆・1トークンあたり約480億パラメータが活性化するとされています。
総パラメータ
1.6兆（Mixture of Experts / MoE）
活性化（当該トークン）
約48B/トークン（=計算量を抑えつつ表現力を確保）
MoEは入力ごとに一部エキスパートだけを活性化するため、長文でも“全体計算をしない”設計思想に整合します。
//Metaphor
06 / 16
候補選択が肝

LSA

LSA（LongCat Sparse Attention）は、attention高速化の“本丸”をindexer（候補選択器）の律速として捉え、複数手法で直交的に効率化します。
SI
Streaming-aware Indexing：ストリーミング文脈に最適化
CLI
Cross-Layer Indexing：層を跨ぐindexingの償却（サリエンス安定性）
HI
Hierarchical Indexing：候補集合そのものを縮小
//Process
07 / 16
indexingコストを分散

SI/CLI/HI

LSAは、メモリアクセスの予測可能性・層を跨ぐindexingコストの償却・候補集合の縮小を同時に狙うとされています。
メモリアクセス
予測可能にして、実行時の不確実性（待ち）を減らす方向。
償却（CLI）
隣接層のattentionサリエンスが安定する前提で、indexingを“使い回す”。
候補縮小（HI）
候補集合を狭めることで、attention計算そのものを減らす。
拡張
推論高速化（speculative decoding / MTP）にも展開可能とされます。
//Distinction
08 / 16
“増やす”ではなく“軸を変える”

N-gram

N-gram Embeddingは、MoEのエキスパート増量ではなく表現の別軸拡張として設計され、利益が出る比率を制限しています。
前提
MoE活性化が約97%疎性到達（=エキスパート増量の追加効果は小さめ）
設計制約
N-gram部分が総パラメータ比50%超で利益減 → LongCat-2.0は10%未満に制限
どのベンチマーク条件で最適性が確認されたかの詳細は、提示テキストだけでは追えません。
//Architecture
09 / 16
ASIC superpodsで完走

運用品質

GPU中心の世界ではない AI ASIC superpods 上で、決定性・数値信頼性・故障回復まで含めた運用品質が強調されています。
決定性
determinism：再実行で挙動が崩れにくい前提づくり。
数値信頼性
誤差蓄積低減・高精度ベースライン検証・ビット反転検知。
フォールトリカバリ
故障リンク切離しでも影響が出にくく、合格テスト後に復帰。
//Process
10 / 16
TTFT/TPOTで配備

推論設計

1.6T級を1Mトークンで扱うため、prefill（読み込み）とdecode（生成）を分離し、異なる並列戦略で成立させる方針です。
prefill側
CPP/SPやKVPなどの並列手法を使い分け、並列効率を確保。
decode側
TPOTやレイテンシ分布に配慮しつつ、別の並列（例：EP128等）で最適化。
ロードバランス
EPLBでサービス偏りを抑える。
注意点
p95/p99レイテンシや$ / tokenなどのコスト指標は、テキストから評価しにくい。

11 / 16
複数教師で整える

後段学習

Agent Experts / Reasoning Experts / Interaction Experts を作り、MOPDで統合してエージェント実行・深い推論・ユーザー体験をバランスする方針です。
Agent Experts
ツール呼び出し精度、マルチターンAPIの堅牢なパース、自己修正。
Reasoning Experts
深い推論とループ抑制を学習目標に含める。
Interaction Experts
アラインメント・幻覚抑制・安全性の境界（Interaction）を調整。
定量評価としてコード・エージェント系ベンチマーク比較やデモが提示される一方、失敗パターン内訳が不足している点は残ります。
//Comparison
12 / 16
強みと弱み（要点）

評価

LongCat-2.0の強みは体系的統合ですが、外部検証の透明性（アブレーション、前提条件、1M運用の分布）が不足し、解釈に注意が要ります。
強み
LSAがindexer律速に踏み込み、SI/CLI/HIで構造的に説明。N-gramは10%未満など制約ベース。
限界
検証条件（学習データ混合比、ベースライン、サリエンス安定性の成立範囲）が読み取りづらい。
安全・運用
幻覚抑制/境界づけ・決定性/ビット反転検知/故障回復など運用品質を重視。
追加で欲しい
アブレーション、評価設定の完全開示、失敗モード解析、推論コスト指標（p95/p99、$ / token）。

13 / 16
次の確認点

結論

LongCat-2.0は「長文＋エージェント＋独自ASIC基盤」を、モデルとシステムを一体最適化して成立させようとしている。
最重要チェック
LSA各要素の寄与（SI/CLI/HIのアブレーション）と、CLI前提（サリエンス安定性）の条件。
実運用の検証
1Mコンテキストのレイテンシ分布・コスト/トークン・再現条件の公開度（外部追試）。
体系的統合は魅力的。一方で、研究としての信頼度は“透明性の追加”で一段上がる余地が大きい、というのが整理結果です。
