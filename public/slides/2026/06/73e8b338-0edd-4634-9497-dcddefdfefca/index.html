人生相談を“初対面”から解放

取扱説明書

Claude Code と仕組み（Skills／MCP／メモリ）で、自己理解を運用する体験談を「再現しやすい設計」に圧縮します。

要点：最適化の中心は自動化ではなく、自分を言語化する強制力です。

Problem
03 / 16
毎回の前置き地獄

初対面

AIに相談すると、毎回「状況説明」から始まって疲れます（初対面問題）。

体験
相談のたびに自己情報を再入力
結果
考えるより“説明する作業”が増える
Metaphor
04 / 16
言語化＝自分のOS

地図

AIに渡すのは“人生の中身”だけでなく、参照すべき情報の地図（分類の軸）です。

self
変わらない自己（基礎）
direction
目指す方向（方針）
work
進行中の仕事（実行）
journal
日々の記録（振り返り）
knowledge
外部知識（判断の材料）
Evidence
05 / 16
勝因は自動化ではない

強制力

3か月運用で最大の効果は「自動化」ではなく、書くことによる言語化の強制力でした。

最初の動機
人生まで自動化してラクしたい
実際の転換
“書けない”詰まりが自己理解を押し広げる
Architecture
06 / 16
AIの使い方を部品化

Skills

Claude Code の Skills で、相談やタスク更新を“コマンド化”して運用を安定させます。

狙い i
やることを毎回同じ手順に揃える（再現性）
狙い ii
入力設計を崩さない（プロンプト以前の設計）
※ここでも中心はツールではなく「入力と参照の整備（運用設計）」です。
Process
07 / 16
地図を先に作る

時制

情報を混ぜないために、参照目的で時制の軸に分けます。するとAIが“何を重視すべきか”迷いにくくなります。

分類ルール
self／direction／work／journal／knowledge
効果
答えにくさが減り、助言の輪郭が濃くなる
Architecture
08 / 16
外部を繋いで記録を回す

MCP

MCPでカレンダー等と接続し、作業と記録が運用サイクルに入る状態を作ります。

接続
予定・予定周辺の情報を参照／更新
狙い
会話だけで終わらせず、運用に落とす
Architecture
09 / 16
参照先を覚えさせる

メモリ

メモリ機能で、好みやフィードバックをAI側に保持させます。これで毎回の“初対面前置き”が軽くなります。

保持する
好み（Preferences）／フィードバック（Feedback）
効き方
相談の文脈が繋がり、反復コストが減る
Distinction
10 / 16
主導権は人間へ

丸投げ禁止

ゴール（どこへ行くか）は人間が握るべきで、AIは実行や提案（どう進めるか）を担うのが安全です。

人間
価値判断の最終責任（方向づけ）
AI
整理・加速・編集・提案（実行の支援）
Evidence
11 / 16
見えないルールを出す

仕様化

自己理解が曖昧だと助言もぼやけるため、価値観や判断基準という“見えないルール”を明文化します。

書く対象
価値観／判断基準／守る制約（ガードレール）
狙い
攻略可能な仕様としてAIに渡し、ブレを減らす
Comparison
12 / 16
効く点／注意点

両面

良い点は実用性と運用品質、注意点は測定不足と誤記録・固定化などのリスクです。

強み
Lv1〜Lv4で段階導入／言語化の学習効果／ルール明文化
弱み
成果指標が少ない／誤記録の修正手順が薄い可能性／プライバシー運用は要設計
Closing

Lv1例
