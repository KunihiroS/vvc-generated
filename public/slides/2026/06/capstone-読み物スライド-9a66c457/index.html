capstone
02 / 16
軽量 × マルチアーキ
分解フレームワーク
Capstoneは、機械語バイト列を人が読めるアセンブリへ分解し、暗黙に参照・更新される情報に近いデータも提供する設計です。
バイナリ解析・リバースエンジニアリング（セキュリティ解析を含む）の基盤として使われます。
中心
disassembly / decomposer
実装
C + 多言語バインディング
problem
03 / 16
横断した解析が必要
壁は CPU差
アーキテクチャが違うと、命令デコードや意味推定の実装が分断されがちです。
その結果、解析基盤の保守コストと統合コストが増えます。
課題 i
命令セットごとの実装差
課題 ii
解析パイプライン統合の難しさ
metaphor
04 / 16
分解は “材料” 提供
アセンブリだけで終わらない
Capstoneの“decomposer”は、文字列の逆アセンブルに留まらず、命令が暗黙に読む/書くレジスタなどの手がかりを返すことが特徴です。
解析（データフロー推定など）の材料が増え、次工程へ渡しやすくなります。
入力
バイト列
出力
命令 + 詳細情報
価値
意味寄りの解析補助
architecture
05 / 16
統一APIで組み込む
横断性の 設計
多数のアーキテクチャを扱い、アーキテクチャ非依存のAPIを掲げています。
これにより、解析基盤側でCPU差を吸収し、ツール開発の手戻りを減らせます。
対応の例
ARM / ARM64 / RISC-V / x86(16/32/64)
開発メリット
非依存API → 実装差を局所化
enumeration
06 / 16
“意味情報” の要点
decomposerの出力
Capstoneは、命令が暗黙に関与するレジスタなど、意味推定に使える情報の提供が重視されています。
結果として、静的解析・セキュリティ解析パイプラインに乗せやすくなります。
i
暗黙のレジスタ読み（read）
ii
暗黙のレジスタ書き（write）
iii
命令詳細（デコード成果の充実）
iv
解析の次工程で再利用しやすい

万能ではない
精度は検証が必要
アーキテクチャ対応が広くても、実際の命令網羅率や精度はアーキテクチャごとのデコーダ状態に依存します。
