PROBLEM
03 / 16
監督が前倒し

認知負荷の形が変わる

エージェントは停止しないため、人の監督は「後工程の逐次確認」から「短いプロンプト投入前の前倒し準備」に移る。 その結果、速いがゆえに外れ続ける危険がある。

Before / 従来

設計→レビュー→テスト→運用を人が段階的に分担。

Now / 今

エージェントが即応し、監督論点が“プロンプト前”へ前倒し。

CORE IDEA
04 / 16
複雑性ではなく圧縮

中心課題は“量”ではない

記事の核は、難しさ（complexity）そのものよりも「認知負荷が圧縮されること」を重視する点にある。 人は“連続監督”を続けるため、何を考えるかの範囲を絞る必要がある。

Compress / 圧縮

逐次チェックより、構造的判断を前倒しで行う。

Focus / 焦点

人が扱うのは“論点のある決定”に寄せる。

Guardrails / ガードレール

安全性・信頼性を問い合わせ可能な形で委譲する。

FRAME
05 / 16
境界を切る

what vs who/how

エージェント基盤（agentic platform）は“必要なもの（what）”を定義する。 Team Topologiesは“誰が提供するか（who）／どう相互作用するか（how）”を設計する。

Agent Platform（what）

セキュリティ、信頼性、ブランド整合性、文脈注入、ツール連携、評価、CI/CD…を提供。

Team Topologies（who/how）

4種類のチーム型と、3種類の相互作用で“責任分離”を作る。

PLATFORM
06 / 16
守り＋手続き

プラットフォームが受け持つ

プラットフォームは、守り（security/信頼性/ブランド整合）と手続き（context、tool連携、evaluation、CI/CD等）を自己完結的に提供する。 これにより人の監督は「後からの逐次」→「前倒しの構造的判断」に縮退する。

守り / Guard

ガードレール、監査トレイル、透明性の確保。

手続き / Procedure

評価（evaluation）、CI/CD、文脈注入、ツール連携。

TEAMS
07 / 16
4つのチーム型

誰が何を担当する

チームは4型に分ける：ストリーム整合、プラットフォーム、イネーブルメント、複雑サブシステム。 それぞれの“責任の持ち場”を固定する。

① Stream-aligned / ストリーム整合

アプリを作るプロダクト（意図・合否判断＝what）。

② Platform / プラットフォーム

能力をX-as-a-Service化（保証＝how）。

③ Enablement / イネーブルメント

自律化までの橋渡し（縮む役割）。

④ Complex subsystems / 複雑サブシステム

AI基盤の専門部分を担う。

INTERACTIONS
08 / 16
相互作用は段階的

相互作用は 進化させる

チーム間の相互作用は、ファシリテーティング→X-as-a-Service→コラボレーションの流れで段階化するべきだ。 「任せてよい範囲」を広げ、監督対象を減らす。

i. Facilitating
支援→自律化

イネーブルメントが橋をかける（初期）。

ii. X-as-a-Service
自己提供

プラットフォームが“問い合わせ”として提供。

iii. Collaboration
構築期の深い協働

複雑領域での共同構築。

BOUNDARY
09 / 16
運用責任の分離

運用は分ける

ストリーム整合チームが“what”に責任を持ち、プラットフォームがデプロイ／監視／ロールバック等の“how”を保証する。 これにより業務チームが技術者でなくても制作に参加できる前提が成立する。

Stream-aligned：what

意図・品質の合否（ドメインの判断）。

Platform：how

デプロイ、監視、ロールバック、ガードレール適用。

MATURITY
10 / 16
測れる条件

成熟度を観測する

「理念」ではなく、観測可能な基準でプラットフォーム成熟度を測る。 ガードレール自動カバー、パイプライン成功率、自己サービス比率、監査トレイルなど。

Guardrails / カバー

ガードレールが自動で適用される。

Pipeline / 成功率

パイプライン成功率を追跡する。

Audit / 監査

監査トレイルを残す。

Self-service / 自己提供

自己サービス比率を高める。

Docs / ドキュメント

例付きで文書が揃っている。

Coverage / 適用範囲

守りの範囲が実運用でカバーされる。

GOVERNANCE
11 / 16
シャドーITを封じる

中央集約で 見える化

ガバナンスが欠けると、産業化されたシャドーITが生まれる。 プラットフォームがデプロイ／監視／利用指標を中央集約し、アプリ存続や廃止を見える化・自動フラグ化する。

Central Visibility
中央可視性

デプロイ、監視、利用メトリクスを集約。

Lifecycle Control
ライフサイクル制御

存続/廃止を判断しやすくする。

EVOLUTION
12 / 16
ガードレールの昇格

個別→全社へ

ガードレールが個別から全社のプラットフォームへ昇格する道筋が重要。 ただし、プラットフォーム製品オーナー（PO）がボトルネックになり得るため、昇格候補検出や優先度付けは自動化で緩和すべきだ。

Signal → Upgrade
昇格のシグナル

同じニーズが繰り返される（例：3回以上の反復）を目安に一般化。

PO Load → Automate
PO負荷の抑制

ポートフォリオ追跡をツール化し、手作業裁きを減らす。

CLOSING
13 / 16
要点まとめ

成功の鍵は 責任分界

エージェント時代は“作る速さ”より“人の意思決定スループット”がボトルネックになる。 whatはストリーム、howはプラットフォーム、相互作用は段階化して認知負荷を圧縮する。

Platform（what）/ プラットフォーム

守り＋手続きの自己提供で、監督対象を絞る。

Stream（how）/ ストリーム

意図と合否（ドメイン判断）に集中する。

Gov（進化）/ ガバナンス

中央集約でシャドーITを抑え、ガードレールを昇格させる。
