Exapanse
02 / 16
HPCのムダ削減

Expanse

GPUクラスター/SLURM/Kubernetesで起きる「必要以上の資源申請」を、予測・可観測性・失敗診断で減らす取り組みです。

問題
03 / 16
過剰申請の常態化

Overprovision

研究者は失敗回避のため、VRAM/CPU数/メモリ/walltimeを安全側に“過剰に”申請しがちです。

構造
04 / 16
安全性の非対称

Asymmetric Risk

失敗の損失
OOMなどで数日分の計算が無駄になり得る。
運用の損失
未使用リソースを抱えたまま走り、実効稼働率が伸びにくい。

結果として「利用者の安全側インセンティブ」と「データセンターの全体最適化」が噛み合いません。

核心
05 / 16
ジョブ要件を推定

Live Forecast

Expanseはジョブ投入前に、コード/投入スクリプト/テレメトリから「実際に必要な資源量」を推定します。

入力
06 / 16
コード×環境

Multi-modal

ジョブ側
ソースコード / 投入スクリプト
申請条件
VRAM / CPU / メモリ / walltime
環境側
DCGM / CUPTI / cgroups / ネットワーク/IO

テキストだけ（LLM）の“真空推論”ではなく、計算グラフと実測性能を結びます。

出力
07 / 16
確率で提示

Risk-aware

必要量の推定
p90 / 信頼区間などで資源レンジを提示。
失敗予兆
OOM等が起きそうなケースを事前に予測。

利用者は推定を“鵜呑み”せず、リスク許容度を選べます。

診断
08 / 16
当たった後も強い

Debug Assist

推定が外れて失敗しても、スタックプロファイルとハードウェア指標の相関解析で原因に寄り添います。

推薦
09 / 16
行単位の改善案

Code-level

分析
失敗時とテレメトリの相関を解析。
提示
（主張として）コード行レベルの修正提案まで。

単なる最適化ツールではなく、実務で“直せる”情報を返す設計です。

統合
10 / 16
スケジューラは置換しない

Side Intelligence

ExpanseはSLURM/Kubernetesの制御機構を置き換えるのではなく、申請とユーザ判断に介入する“知能レイヤ”です。

適応
11 / 16
クラスタ学習

Cluster Fine-tune

ベースライン
sacct平均 / 手書きルール / LLM推論と比較。
適応
クラスタ固有に微調整し、時間で精度向上。

ハード構成やワークロード変化への適応を狙います。

運用
12 / 16
顧客内デプロイ

On-prem / VPC

導入モデル
クラスタ単位の有償パイロット（例：2週間測定ウィンドウ）。
セキュリティ姿勢
テレメトリ外部送信を行わない（主張：egressなし / エアギャップ・VPC内）。

ただし実装詳細は本文抜粋だけでは不十分で、監査・契約確認が重要です。

所感
13 / 16
評価が分岐点

Measure & Trust

Practical（実務性）
提出フローにフックし、大改修なしで導入しやすい。
Safety（安全性）
p90等＋失敗予兆で“事故なし”に寄せる。
Security（セキュリティ）
送信しない主張は評価。ただし監査可能性の確認が必須。

成功の鍵は「どのジョブ種別で」「どれだけ申請を下げても失敗が増えないか」を測定ウィンドウで定量検証することです。
