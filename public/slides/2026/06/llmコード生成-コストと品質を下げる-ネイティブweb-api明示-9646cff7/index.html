02 / 16
LLMのコード生成
API明示で下げる
出力トークンが膨らみがちなLLMのコード生成は、「ランタイムのネイティブWeb APIを最初に指定」することで、コストと品質を同時に改善できます。
（英: Native Web API / 日: ネイティブWeb API）
Problem
03 / 16
コストが出力で増える
出力トークン問題
LLMの料金は入力と出力で分かれることが多く、コードが長いほど出力トークン費用が跳ねやすいです。 実装が「標準で済むのに手作り」だと、冗長化が構造的に起きます。
Problem
04 / 16
再実装が長文化
手作りが冗長化
例：URL/クエリパース、Formの状態管理、fetchのキャンセル、並列処理の失敗処理などは、学習データの“旧来パターン”に引き戻されやすく、コード量が増えます。
例（概算）
クエリ：約140トークン vs 標準API：約12トークン
例（概算）
Form：約200〜250トークン vs 標準API：約14トークン
Metaphor
05 / 16
LLMは勝手に最適化しない
前提は指示する
LLMは「自分が動く環境で、何がネイティブに用意されているか」を自動推定しません。 だから最初から正しいAPI（Web API）を指定して、生成空間を短い実装へ寄せます。
Architecture
06 / 16
核：ネイティブAPIへ誘導
Web APIを明示
「手作りの実装」ではなく、ランタイムの標準機能を使うようにプロンプトで制約すると、短いコードで同じ目的を達成しやすくなります。
URL / Query
URLSearchParams
Form
FormData
Cancel / Timeout
AbortController
Architecture
07 / 16
並列と失敗も短縮
Promise標準化
並列処理のエラーハンドリングや「全部失敗/一部成功」の方針は、標準のAPI側に寄せると実装が圧縮されます。 さらに仕様準拠のため、典型的な穴を減らせます。
推奨
Promise.allSettled
関連
AbortSignal.timeout
Evidence
08 / 16
品質は“バグカテゴリ減”
短くなるのに安全
手作りパースや状態管理は、プロトタイプ汚染、デコード取りこぼし、タイマリーク、アクセシビリティ破綻などを誘発し得ます。 仕様準拠の標準APIは、既知のエッジケースを前提に作られているため、その方向の事故を構造的に減らすとされています。
Distinction
09 / 16
コメントは“制約”になる
誤コメントが有害
コメントは単なる説明ではなく、LLMが意図として読み取り得る情報です。 誤った/古いコメントは、無コメントより悪化する可能性があるため、運用を「生成品質の制御」として扱うべきです。
Comparison
10 / 16
フォーマットは補助
インデント最適化の優先度
インデントや空行削減は入力トークンを減らせますが、出力トークンが高くつきやすい前提だと主戦場ではありません。 もっと大きいレバーは「API再実装を避ける」ことです。
フォーマット
入力削減：約24.5%（研究紹介）
API切替
出力差：最大で約85〜92%削減（提示の整理）
Process
11 / 16
実務での書き方（要点）
プロンプト設計手順
まず「環境が提供するネイティブAPI」を禁止/許可として明示し、同等の目的は標準手段で達成させます。
i
用途を定義（例：クエリ生成、Form送信、キャンセル）
ii
使うWeb APIを指定（例：URLSearchParams / FormData）
iii
手作り再実装を禁止（独自パース、独自タイマ、独自状態管理など）
Distinction
12 / 16
注意点：数値と柔軟性
導入のリスク管理
提示のトークン削減率は概算で、モデル・プロンプト・方針で効果は変わります。 また指示を固定しすぎると、既存の設計やテスト戦略を十分に織り込めない可能性があるため、社内規約とレビュー体制も設計します。
リスク
概算のため再現性は要検証（自チーム計測が必要）
副作用
プロンプト過固定で設計考慮が不足／コメント運用コスト増
Closing
13 / 16
結論
再実装をやめる
最大のレバーは「新しい技巧」ではなく、あなたの環境がすでに持っている標準APIを、モデルが分かる形で最初から指定することです。 それにより出力トークンが下がり、同時に安全性・保守性の土台も改善しやすくなります。
One-liner：Native Web API First（ネイティブWeb APIを最優先）
