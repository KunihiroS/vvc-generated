cotect
02 / 18
信頼より検証
verify 変更を人間のゲートへ
エージェントが書いた差分を、hunk単位でキャンバス表示し、承認・コメントまでをレビュー体験として先回りします。
Problem
03 / 18
追えない増加
Agent 変更が増えるほど検証が遅れる
エージェントは便利ですが、差分が一括で現れると「どこを見ればいいか」が散り、見落としや判断疲れが起きます。
Metaphor
04 / 18
ゲート制御
Canvas 出荷前に一手ずつ見る
承認されるまで“コミットに相当する最終化”を待つ発想で、レビューを手続きとして分割します。
Architecture
05 / 18
ローカル監視インスペクタ
cotect リポジトリ監視 → hunk提示
cotect はリポジトリ更新を監視し、差分を編集画面（canvas）上で逐次（hunk単位）にレビューできるようにします。
Core
06 / 18
Trust→Verify
trust ではなく、verify する
変更は“見た瞬間”に canvas に現れ、人間が承認・コメント・差し戻しを選べる設計です。
Read-only
07 / 18
勝手に書かない
read-only 作業ツリーを直接変更しない
ツールが裏で作業ツリーを書き換えるリスクを抑え、最終判断の主導権を人間側に残します。

cotect

read-first で、ユーザーが hunk ごとにレビューして判断

避けたい状態

ツール主導で自動更新・自動コミットが進むこと

Flow
08 / 18
hunk-by-hunk
hunk 論点を分割して承認
差分を一括で読むのではなく、論点のまとまりごとに承認・コメントできるため、早期に止めやすくなります。

i 提示

変更が canvas に逐次反映される

ii レビュー

行単位・ブロック単位で点検

iii 決定

承認 / コメント / 差し戻し

UX
09 / 18
止まらない検証
~20ms file opens の体験設計
live working tree のような体験でレビュー遅延を小さくし、実務の「見てられない」問題を下げる狙いがあります。
Signals
10 / 18
優先度の補助
Health 構造情報で焦点を絞る
差分だけでなく、インポートグラフやコードベース健全性（循環依存、ホットスポット、テスト欠如など）のヒントを提示します。

提示される観点

LLM-risky files / Review Priority / Missing Tests / Circular Dependencies など

ねらい

変更の影響範囲・再現性の低さを前もって見立てる

Boundary
11 / 18
万能ではない
Heuristics 硬い保証はない前提
提示情報は「heuristics」であり、セキュリティ脆弱性の自動発見や形式的検証のような“保証範囲”はこの情報だけでは判断できません。
Safety
12 / 18
基本線を押さえる
Accountability 最終責任は人間へ
read-only とゲート型レビューにより、「誰が最終的に何を変えたか」を説明可能にしやすい点が重要です。
Limits
13 / 18
過信を避ける
Missing 確認ポイントが別に要る
依存脆弱性、権限逸脱、データ漏えい、性能劣化などは追加ルールや機能に依存し、入力だけでは分かりません。

追加で知りたいこと

LLM-risky files の算出根拠 / セキュリティ・形式検証の有無

運用設計

レビュー結果の保存・監査ログ連携 / トリガ（ブランチ・PR等）

Takeaway
14 / 18
仕組み化の価値
Review “信頼の置き換え”ではなく“検証の仕組み”
cotect は差分表示を超えて、hunkごとの承認フローとコードベース健全性ヒントで、見落としを減らす土台になります。
References
15 / 18
参照（入力要約）
FAQ 要点だけ整理
・cotect はエージェントの変更を自動コミットしない（read-first）
・変更は canvas に即時反映され、人間が承認/コメント/差し戻しを行う
・LLM-risky files はリポジトリ自体から算出（ただし heuristics）
