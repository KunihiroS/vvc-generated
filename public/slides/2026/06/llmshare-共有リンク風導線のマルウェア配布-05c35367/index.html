02 / 16
共有リンク悪用
LLMShare報告

正規のチャット共有リンク風UIと検索広告を組み合わせ、ユーザーの「普通の操作」をマルウェア配布へ転換する手口が報告されました。

狙いは“偽サイト”ではなく、“安全に見える導線（share link）”の連鎖を作ることです。ねらい（Goal）: 正しく見せて実行を促す
Problem
03 / 16
防御がすり抜ける
見た目問題
危険は「怪しいドメイン」に潜むだけではありません。正規っぽい画面（共有リンク風）から始めると、入口のブロックが効きにくくなります。

攻撃の核（Core）

chatgpt.com/s/[unique-id] のような共有リンク“らしさ”を偽装

偽UIで誤認させる

防御の弱点（Weak）

ドメイン単位の制御だけでは、入口の段階で断ちにくい

入口≠配布先
Metaphor
04 / 16
連鎖の罠
入口は正規
ユーザーの行動は同じでも、背後で配布先・挙動が切り替わります。つまり“見た目”が合図になり、“次の操作”へ誘導される構造です。

share link風表示（chatgpt.com/s/[id]） → 「ダウンロード」操作 → 外部配布先へ切替 → OS別ペイロード

ポイント：入口で止めるだけでなく、「操作後にどこへ飛ぶか」「何を要求するか」を監視する必要があります。
Architecture
05 / 16
攻撃フロー
LLMShare導線
スポンサー検索広告が“正規入手の意図”に刺さり、共有リンク風の画面へ到達させます。そこから外部ドメインへ転送され、ペイロードが配信されます。

i. 広告（Ads）

「ChatGPT desktop app / download」系の検索語を狙う

検索意図に擬態

ii. 表示（UI）

正規っぽい共有リンク風：chatgpt.com/s/[id]

安全誤認

iii. 転送（Redirect）

外部ドメインへ移動（例：openew[.]app）

配布先へ切替
Enumeration
06 / 16
説得の演出
クラッシュで急ぐ
攻撃ページは“メンテナンス中”“アクセス集中”など、実行を正当化する文面でユーザーの心理を動かします。次の行動は「ダウンロード」という通常操作です。

文面（Message）

システムメンテ風の警告で“今すぐ対応”に見せる

緊急性を演出

行動（Action）

デスクトップアプリのダウンロードを促す

通常の取得フローに沿う
技術だけでなく認知（認識の順番）を狙う点が、被害拡大に直結します。
Evidence
07 / 16
OS横断
Windows / macOS
同系統の罠が複数OSで成立し、単一プラットフォーム対策だと取り逃しが起き得ます。Windowsでは環境判別、macOSでは機微情報窃取の系統が報告されています。

Windows側

物理/仮想やセキュリティ関連レジストリキー等を手掛かりに判別

解析回避（Avoid）

macOS側

Odyssey Stealerを落として機微情報を窃取（報告）

情報窃取（Steal）
横断配布の示唆：組織内の混在環境前提で監視が必要です。
Distinction
08 / 16
解析を遅らせる
サンドボックス見分け
自動解析環境（サンドボックス）を検知すると、解析者には無害に見えるモックや挙動変更が出ます。結果として、封じ込めの速度と確度が落ちます。

検知（Detect）

仮想/解析環境の手掛かりを収集して識別

偽の挙動に切替

影響（Impact）

解析再現がズレる／報告が遅れる

実被害に追いつかない
防御は「検体が動いた後」だけでなく、「通信・実行要求・転送」を早期に見る必要があります。
Comparison
09 / 16
単純なURLブロック
だけでは足りない
この手口は「正規ドメインに見える」段階と「実際の配布先」が分かれるのが特徴です。だから“入口のドメイン”だけで止めにくく、連鎖全体で監視が要ります。

入口対策（入口）

URLフィルタ / DNSブロック：一部は効くが限界

見た目が正規

多層監視（全体）

最終ダウンロード先・挙動（実行ファイル要求等）まで含める

操作後を追う
Process
10 / 16
現場での抑止
やること（要点）
ユーザーと組織で、同じ目的を別の層で達成します。結論は「公式以外からのアプリ取得を避ける」「ダウンロード誘導文言に反応しない」です。

ユーザー（User）

公式以外からアプリを落とさない／検索広告経由の“ダウンロード”に警戒

拒否をデフォルト化

組織（Org）

ブラウザ保護・アプリ制御・EDRの振る舞い検知・導線（メール/広告/URL）監視

連鎖の各段で止める
実行前に署名や提供元を確認することも有効です。
Distinction
11 / 16
原因の断定には注意
サービス側の責任範囲
記事の提示内容だけでは「OpenAIの不具合」と断定できません。少なくとも共有リンク風UIを悪用している点は読み取れますが、成立経路（悪用・設定・運用の穴）は別途確認が必要です。

言えること

正規に見えるURL/画面を入口にしている（報告）

観測ベース

断定できないこと

どこまでがサービス機能の脆弱性か／公式の対応状況

追加情報が必要
防御は“確実に当たるルール”へ落とすため、追加の技術報告を前提に設計してください。
Evidence
12 / 16
考え（私見の要点）
配布戦略が変わった
従来は怪しいドメインやメール添付が中心でしたが、今は「正規サービスの共有リンク風UI」＋「検索広告」で注意力を奪いに来ます。

Security

UIが正しいため危険判断が遅れる

見た目の欺き

Operational

サンドボックス検知で封じ込めが遅れやすい

対応の遅延

Strategic

広告→共有風UI→外部配布の“連鎖”を各段で監視すべき

多層監視
