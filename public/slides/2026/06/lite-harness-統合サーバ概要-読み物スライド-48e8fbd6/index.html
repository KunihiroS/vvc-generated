lite-harness

統合サーバ概要

複数のAIコーディングエージェントを、同じ統一サーバ経由で運用するための設計思想を素早く把握します（統合基盤 / Integration Server）。

Problem / 課題
03 / 12

別々運用が重い

エージェントごとに別サーバを立てると、API仕様差（入力・プロンプト・MCPツール）やセッション管理の不安定さが運用負担になります。

Difference / 差分
入力仕様・プロンプト・MCPツールの扱いが揃わない
Sessions / セッション
セッション管理が難しく、運用が安定しにくい
Core / 要点
04 / 12

OpenCode互換サーバ化

lite-harnessは、複数エージェントを「統一サーバ」にラップし、OpenCode互換として同一Dockerイメージで維持できるようにします。

統合 / Unified
Claude Code / Codex / OpenCode / Copilot を同一導線へ
互換 / Compatible
OpenCode互換サーバとして運用体験を寄せる
維持 / Maintain
別々サーバ分の保守差分を減らす
Usage / 利用
05 / 12

CLI・API・Slack

エージェントはCLI / API / Slack経由で利用・デプロイでき、チームで同じ運用体験を共有しやすくなります（提供手段 / Channels）。

CLI
手元から起動・管理（運用の入口）
API
Open-code API Spec / 統合サーバとして呼び出し
Slack
ワークフローを会話内で扱う（返信のストリーム等）
統一体験
仕様差・セッション差の吸収を狙う
Schedule / 実行
06 / 12

cron × 隔離サンドボックス

定期実行はcronスケジュールで行い、isolated Linux sandbox（E2BまたはDaytona）で動かすことで、事故リスクを安全寄りに設計します（隔離 / Isolation）。

cron
平日4時間おきなどのスケジュール実行
Sandbox
E2B_API_KEY / DAYTONA_API_KEYで隔離環境を適用

※自動化は外部へ影響し得るため、隔離が重要という前提で整理されています。

Human-in-the-loop

実行内容が外部へ波及する局面（例：送信前）では、人の承認（Human-in-the-loop）をUIフローに組み込み、意図せぬ操作を抑えます。

When / いつ
外部アクションの前など、高リスク局面
Who / 誰が
Inbox UIで承認する運用者・担当者
Security / 安全
09 / 12

Vault と 永続化

VaultでAPIキーなど秘密情報を安全に保持し、DB_PATHなどで永続化すれば、再起動後も履歴やモデルコンテキストを復元できます。

Vault / 秘密
BROWSER_USE_API_KEY、LINKEDIN_PROFILE_ID等を参照
Persist / 継続
DBマウント/DB_PATHで再起動後の復元を可能に
Decision / 判断軸

採用前に確認したい

README抜粋だけでは「セキュリティ保証の範囲」「承認ログの粒度」「互換性の限界」が不明なため、運用採用前の追加確認が必要です。

Threat Model

Takeaway / 要約
11 / 12

統合で「社内サービス化」

lite-harnessは、複数エージェントを統合サーバ化し、共通運用（CLI/API/Slack）、cron実行、隔離、Vault、承認、永続化で「運用可能な社内サービス」に近づけます。

戦略
別サーバ保守の困難さを統合で吸収
安全
隔離サンドボックス + Vault
運用
cron + 承認 + 永続化（DB_PATH）
