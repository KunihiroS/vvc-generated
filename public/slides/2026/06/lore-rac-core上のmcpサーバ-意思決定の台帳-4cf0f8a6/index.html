Lind: Decision Ledger
02 / 16
AIが探す前に、決めを返す

Lore 意思決定の台帳

Loreは、チームの判断（ADRs・要件・設計・ロードマップ等）を typed Markdown として保存し、MCP経由でエージェントに最新の“決定”を確実に根拠提示します。（lore / 決定の法定記録）

Target: re-readable “source of truth” for coding agents（Claude Code / Cursor / Claude Desktop等）
Problem
03 / 16
曖昧検索の副作用

RAGが揺らす

RAGは類似度でヒットするため、過去に却下した方針を再検討したり、最新の決定に追従できないリスクが残ります。（retrievalの揺れ＝意思決定の不一致）

起きがち

“それっぽい文章”に引っ張られる

困る

却下案を再実装 / 方針を更新し続けられない

Goal: “探す”ではなく “決定を返す” （deterministic / 再現可能）
Metaphor
04 / 16
類似検索より、判決文

ニュースではなく 判例

Loreは、過去の経緯を“文章の束”として渡すのではなく、決定を決定として固定し、読む側（エージェント）が参照すべき判決文を返します。

RAG / 検索

類似の手がかりを集める

Lore / 台帳

いま有効な“意思決定”を確定して返す

Bilingual: 決定（decision）台帳（ledger）
Architecture
05 / 16
MCP＋read-only境界

参照は 安全

Loreは Lore/RAC 上のMCPサーバとして動き、エージェントからは読み取り専用（read-only）で参照可能にします。信頼境界は最終的に人間のPRレビューに置きます。

Storage

リポジトリ内 typed Markdown

API

MCP経由で read-only 配信

Governance

書き込みは人間のPRレビュー

目的: エージェントの誤更新・勝手な上書き・改ざんを構造的に抑制
Architecture
06 / 16
typed Markdown / スキーマ

“種類”で守る 整合性

Loreは、決定・要件・設計などを typed Markdown として保存し、RACがスキーマ適合を検証します。曖昧な書き方や壊れた構造を減らし、参照品質を安定させます。（スキーマバリデーション）

typed Markdown

“形式（schema）”が文章を縛る

rac validate

保存・取り込み時に適合をチェック

Bilingual: 検証（validation）／適合（conformance）
Process
07 / 16
CIゲートで“壊れ”を弾く

rac gate 決定だけを通す

Lore/RACは rac gate により、無効・曖昧・参照不能な意思決定をCIで拒否する構想です。これにより “knowledgeの整合性” を運用の中心にできます。

拒否1

フォーマット不正

拒否2

壊れたリンク / 無効参照

拒否3

superseded（無効化）参照

注: 詳細なエラー条件は公式ドキュメント（Masked）依存の可能性
Distinction
08 / 16
RAGの探索を否定しない

併用は 探索→検証

LoreはRAGの代替ではなく役割分担です。必要ならRAGで探索し、その後Loreで“決定としての正しさ”を確認するワークフローが推奨されています。

RAG（探索）

質問に近い情報を探す

Lore（確定）

いま有効な意思決定を返す

Bilingual: 検証（verification）／確定（confirmation）
Distinction
09 / 16
LLM呼び出しを避ける

エアギャップ志向 no-LLM

Lore/RACのエンジンはノーLLM・ノーネットワークとされます。信頼境界は人間レビューが担い、外部通信を最小化する方向性です。

思想

外部の推論や通信で意思決定を揺らさない

境界

最終責任はPRレビュー（人間）

企業導入で懸念されがちな点（セキュリティ/監査）への配慮
Evidence
10 / 16
“決定だけ”を安定供給

エージェントの 判断根拠

Loreの最大の主張は、エージェントが“関連しそう”を探すのではなく、いま有効な最新の意思決定を決定的に返すことです。再実装や方針の再検討を抑えるガバナンス基盤になります。

返すもの

意思決定（decision）そのもの

性質

deterministic / 再現可能

狙い

却下の再検討を減らす

“確定分類＋検証”が一体で機能するイメージ
Comparison
11 / 16
Lore vs RAG vs メモリ

役割は 別

Loreは正本（source of truth）としての役割に寄せ、RAGや外部メモリは必要に応じた探索の補助として位置づけます。移行や併用の前提も、この考え方に沿っています。

Lore

最新の決定を“決定として”返す

RAG / 外部メモリ

探索・補完（インデックス/メモリ）

Bilingual: 正本（source of truth）／補完（complement）
Ops
12 / 16
取り込みとエクスポート

既存資産は 型へ

既存のADRsやドキュメントは import（rac-importスキル等）で typed Markdown に変換し、バリデーションへ進む流れが書かれています。Loreの内容は export（--documents / --graph / --okf）で外部ツールに渡せます。

import

既存→typed Markdown化（人間レビュー前提）

export

外部メモリ/RAG/グラフ形式に変換（OKF等）

OKFはGoogleのOpen Knowledge Format（互換の道）
Closing
13 / 16
導入評価の焦点

“効くか”を 検証

LoreはAIに探索させるより、チームの意思決定を手続き化して監査可能性を上げる方向性です。評価では、(1) 移行体験、(2) rac gateでの弾かれ頻度と改善サイクル、(3) 参照失敗時の挙動（参照不能をどう扱うか）を確認するのが重要です。

価値

意思決定の再実装・再検討を抑制

確認点

superseded定義・スキーマ詳細・失敗時挙動

次アクション例: まず小さくADRsをtyped化→CIゲート→エージェント参照の成功/失敗を計測
