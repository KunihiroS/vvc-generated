予測 × 計測 × 診断
Expanse
Expanseは、GPUクラスターで起きる「過剰見積もりによる浪費」を減らすための、投入前の推定と実行中の可視化・失敗解析を一体化したレイヤです。（Prediction / 観測 / Diagnosis）
元投稿：HN「Launch HN: Expanse (YC P26) – Unlock Wasted GPU Capacity」
本資料：要点圧縮（日本語 / English用語併記）
問題
03 / 16
30–40% しか回らない
過剰見積もり
HPCのジョブ投入では、GPU枚数・CPU数・メモリ・ウォールタイムを先に申告します。ところが実運用ではクラッシュ回避のために多めに申請され、クラスターの実効稼働率が30〜40%に留まります。
原因（リスク非対称）
足りない→途中で死んで損失大／余る→損失小で、ユーザーは保守的になる。
観測される浪費
余剰分が「予約されたアイドル」ではなく「割当の中で実際に使われない部分」に寄る。
例：申請が2〜3倍になる傾向（投稿・コメント要約）
浪費→機会損失（自社/クラウド双方）
根拠
04 / 16
数字で見る損失
浪費の規模
投稿では、実効稼働が伸びない背景として浪費割合を具体化しています。ある国規模HPCで122kジョブ中59%の計算が浪費された可能性、オンデマンド相当では月あたり約8.5Mドル規模の浪費になりうると述べられています。
実効稼働
30〜40%
浪費割合
122kジョブ中 59%
金額換算
約 $8.5M / 月（オンデマンド相当）
定義：予約アイドルより「使われない割当内部」を重視
目的：コストと機会損失を圧縮
直感
05 / 16
見積もりを賢くする
Trip planner
Expanseは「出発前の所要時間推定」だけでなく「走行中の計測」と「失敗した理由の特定」まで行う旅程プランナーに近い発想です。（Prediction / Observation / Diagnosis）
予測（前）
必要資源と失敗リスクを推定
観測（中）
低オーバーヘッドでプロファイル蓄積
診断（後）
失敗原因を相関で絞り、修正提案
投下→実行→学習のループ
手作りルール/単純平均から脱却
中核
06 / 16
モデルはLLMじゃない
多様入力
Expanseはジョブのソースコード・投入スクリプト・クラスタのトポロジ等を“マルチモーダル入力”として扱い、DCGM/CUPTI/cgroups/ネットワーク/IOのテレメトリからクラスタ固有の振る舞いを学習して推定します。
入力（Job side）
コード / 投入スクリプト / ワークロード特徴
入力（Cluster side）
DCGM, CUPTI, cgroups, ネットワーク, IO など
LLMは比較基準として言及（コアは独自モデル）
狙い：性能予測に効く特徴設計
フロー
07 / 16
投入前に推奨
Predict → Recommend
Expanseはスケジューラを置き換えず、SLURM/K8sのライフサイクルにフックして基本は「予測・推奨」層として動作します。ジョブが必要とするGPU/CPU/メモリ/ウォールタイムを、失敗リスクとセットで提示します。
1. 予測（前）
必要量（量）とOOM等の失敗確率
2. 推奨（前）
申請値の調整案（過剰を抑える）
3. 学習（後）
実行プロファイルでクラスタ固有に更新
導入障壁：制御不能な置換を避ける方針
運用：ユーザーの申請見直しが鍵
違い
08 / 16
予測と診断
After-fact
Expanseの“失敗診断”は、投入時の予測（failure risk推定）と別物です。失敗後にテレメトリとプロファイルの相関を取り、“なぜ落ちたか／どう直すか”をライン単位の修正提案として提示します。
予測（Prediction）
投入時点：OOMなどの失敗リスクと必要量
診断（Diagnosis）
失敗後：相関から原因を絞り、修正提案
目的：手戻り削減と学習ループ促進
価値：精度だけでなく実務改善
学習
09 / 16
クラスタごとに適応
Cluster-specific
平均値ベース（例：sacctの過去平均との差）では、新しいジョブ種別やコード変更で精度が崩れます。Expanseはクラスタごとに学習・ファインチューニングし、実行データが増えるほど“より当たる”方向へ更新します。
課題
世代更新・設定変更・ワークロード変化が頻繁
限界（平均）
過去平均との差分は新規で崩れる
解決（適応）
クラスタ固有にファインチューニング
ここが成否：モデル更新の仕組み
投稿の意図：精度を運用で維持
比較
10 / 16
LLM頼みではない
Not-LLM core
ExpanseのコアはLLMではなく独自の機械学習モデルです。LLMはベンチマーク比較の基準として使われ、クラスタごとに微調整する“性能予測”の設計思想が強調されています。
コアモデル
独自アーキテクチャ（マルチモーダル＋テレメトリ）
LLMの位置
比較基準／参照（「LLMで推論」そのものではない）
投稿：LLMに対し8倍改善を主張（条件の読み取りは別途）
論点：測定条件の透明性が重要
セキュリティ
11 / 16
エグレスなし
VPC / Air-gap
HNコメントで“テレメトリ漏洩”への懸念が出たのに対し、Expanse側はデータを外部SaaSへ送らずエグレスがない（エアギャップやVPC運用）と明言しています。HPC/金融/バイオでは推論可能性自体がリスクになり得るため、導入判断に直結する論点です。
対外送信
外部SaaSへ送信しない（エグレスなし）
運用形態
顧客VPC / エアギャップ / 監査ログ
未確定（本文不足）
保存期間・暗号化・最小化など詳細は要確認
導入前確認ポイント：監査範囲とデータ保全
攻める前に統制設計
注意点
12 / 16
精度の測り方が鍵
Pilot required
投稿文だけでは、推定誤差（VRAM/ウォールタイム）、OOM回避率、オーバーヘッド条件などの定量が十分に見えません。また推奨が申請に反映されなければ効果は出ず、段階適用やガードレールが重要になります。
技術リスク
予測が外れるとOOM/中断増 → 信頼区間と閾値設計が必要
運用リスク
ユーザーが“推奨を採用”する文化・承認プロセスがボトルネック
追加で欲しい情報：対象クラスター構成・失敗ケース偏り
追加で欲しい情報：セキュリティ実装詳細
結論
13 / 16
実務を変えるか
Takeaway
Expanseは「予測→実行中計測→失敗診断→クラスタ固有に更新」というループで、GPU資源の過剰見積もり由来の浪費を減らすことを狙っています。スケジューラ置換を避けた設計と、マルチモーダル＋テレメトリ中心の学習が強みで、導入効果はパイロットで確かめるのが筋です。
強み
コード/スクリプト/トポロジ＋DCGM等で推定
価値
精度だけでなく、失敗ログで修正まで導く
次の確認
誤差・失敗削減・オーバーヘッド・セキュリティ詳細
