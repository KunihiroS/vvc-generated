コミットより、
操作と会話
DeltaDB構想
Git中心の“コミットの連なり”ではなく、エージェント協働で生まれる「操作（operation）」と「会話」を一次的に扱う新世代のバージョン管理を目指す。
Problem / 問題
03 / 16
時系列のズレ
会話が後回し
Gitのコミットは“静止画”になりがちで、重要なやり取り（会話）が編集と時間的に切れて追跡しにくい。
Core / 中核
04 / 16
単位を変える
operationが主語
DeltaDBでは、コミットではなく「すべての操作（operation）＝細かなデルタ（delta）」を連続ストリームとして記録し、過去の瞬間を一意に参照できるようにする。
Metaphor / 比喩
05 / 16
静止画→動画
差分が流れる
コミットを“切り出した写真”と見るのではなく、編集の連続と、その間の出来事を“つながった映像”として残す。
Evidence / 根拠
06 / 16
双方向に辿る
行→会話／会話→行
会話の任意の箇所から、その時点のコードへ（または現在のコードとして）飛べるだけでなく、コードの箇所から作った会話や以後の履歴へ戻れる。
Process / 流れ
07 / 16
同じ基盤へ
会話＋編集を併記
メッセージ（会話）と、それが生んだ編集を同じ場所で並べて保存し、片方が解離する問題を減らす。
Git寄り
議論が別の場所に残り、後付けで結び付けが必要になりやすい。
DeltaDB寄り
会話と編集が同じ基盤で記録され、結び付け作業（儀式）が薄れる。
Distinction / 違い
08 / 16
行番号は不安定
参照をdeltaへ
行番号ベース参照は編集でズレる。DeltaDBは参照のアンカーをデルタに置き、コードの移動・改変に耐える前提を作る。
Comparison / 比較
09 / 16
儀式を減らす
コミット前提を緩和
共同編集では、いちいちコミットしてプッシュを待つ必要がない方向を示し、エージェント協働の速度と心理負担に効く。
従来の前提
レビュー・コメントはコードと別になり、後で“つなぐ”手順が要る。
DeltaDBの前提
会話と操作が同じ場所に記録され、後付け結合が不要に近づく。
Architecture / 仕組み
10 / 16
衝突回避の発想
CRDT的な同時編集
競合なしの複製ワークツリー（conflict-free replicated worktrees）を内包する考え方が示され、複数人・複数エージェントが別マシンで同時編集できる余地がある。
Enumeration / 利点
11 / 16
価値の核
何が得か
i 追跡性
操作単位の記録で、過去の瞬間に根拠を結び付けやすい。
ii 双方向参照
会話⇄コードの往復が可能で、理解と説明責任を支える。
iii 編集耐性
行番号ではなくデルタにアンカーし、コード移動に強くする。
iv 協働の速度
コミット待ちの心理的負担を下げ、会話と編集の連続性を保つ。
Distinction / 注意
12 / 16
見えていない課題
実装・監査・安全
本文は方向性が中心で、デルタ粒度、性能、ネットワーク同期、既存CI/CD連携、権限、セキュリティ、衝突時UXが未確定。
ガバナンス
レビュー承認（誰がいつ承認したか）を別メカニズムで満たす必要が出る可能性。
セキュリティ
会話（プロンプト、設計意図等）が体系的に保存され、機密情報の取り扱い設計が重要。
Closing / まとめ
13 / 16
最終メッセージ
会話が生成源
DeltaDBは、会話と操作を一体化し、デルタを基準に参照できる履歴設計で、エージェント時代の“理解しやすい開発”を狙う構想。
