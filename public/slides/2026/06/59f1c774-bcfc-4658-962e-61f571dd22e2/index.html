LLM × Verus（自動証明合成）
verus-proof-synthesis
VerusでRustの形式的証明を自動生成・修復するための基盤。AutoVerus と VeruSAGE を中心に、ベンチマークまで揃える。
読みどころ：生成だけでなく、検証エラー駆動の修正まで

problem
03 / 16
なぜ難しい
証明は連鎖する
Verusでは、型・仕様・不変条件（invariant）が噛み合わないと検証が失敗する。LLMの一発生成では、その連鎖を最後まで通しにくい。
失敗の形
エラー・不整合が出るが、原因は局所とは限らない
対策の核
検証ログを見て、推論→修正を繰り返す
結論：自動化は「生成」だけで完結しない
——
metaphor
04 / 16
発想
Repair が要
最初の候補を作って終わりではなく、Verusの検証エラーを起点に“直す”設計が中核。これにより成功確率を上げる。
Inference
few-shotで初期候補を生成
Refinement
改善して候補を更新
Repair
検証エラーをデバッグして修正
AutoVerusの三相：生成→洗練→修復

AutoVerus
小さく攻める
AutoVerusは“アルゴリズム的コード”向けに、少数ステップで証明を作りに行く。少しでも通りやすくするため、3段階プロセスで候補を磨く。
狙い（日本語）
少規模問題での通過率を上げる
狙い（English）
success probability via multi-stage correction
ポイント：エラー駆動のRepairが明示される

architecture
06 / 16
VeruSAGE
大規模を回す
VeruSAGEは“大きなシステムプロジェクト”の検証に強いとされるエージェント型。単発生成より、検証結果を見て状況依存で進める。
ループ
observation → reasoning → action
観察材料
検証ログ（エラー等）を手がかりに更新する
適材適所：依存関係が多いほど“ループ型”が効く

comparison
07 / 16
選び方
どちらを使う？
READMEの説明では、AutoVerusは小規模アルゴリズム、VeruSAGEは大規模システム。入力Rustの規模と依存関係を基準に選ぶのが自然。
AutoVerus
小さな“アルゴリズム寄り”入力
VeruSAGE
“システム寄り”で関連が多い入力
規模 × 依存関係：それが最大の判断軸

08 / 16
評価基盤
ベンチマーク2系統
Verus-Benchは150タスク、VeruSAGE-Benchは849タスク。単に数が多いだけでなく、難しさの性質が異なる領域を分けて評価する。
Verus-Bench
150（古典CS、仕様問題、配列/ループ、チュートリアル例など）
VeruSAGE-Bench
849（分散、KV、OSカーネル、マイクロカーネル、ストレージ、シリアライゼーション等）
注意：“同じ尺度で単純比較”は危険になり得る

11 / 16
品質境界
過信しない
README断片からは、成功確率や失敗時の残存誤り、部分成功の扱いが十分に確定できない。採用基準は基本的に「Verus検証が通ったもの」を守る必要がある。
不明点
どの程度で“最終的に通る”のか、部分成功をどう扱うのか
推奨
Verus検証合格を採用条件にして監査可能性を確保
安全用途：ログ・差分・根拠の保持が重要

強みと限界
