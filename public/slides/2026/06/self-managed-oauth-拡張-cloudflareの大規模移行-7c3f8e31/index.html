Report
02 / 17
Cloudflare OAuth / self-managed
全員が使うOAuth

CloudflareのOAuthを、従来の一部連携から「すべての顧客が自分でOAuthクライアントを管理できる形（self-managed）」へ拡張した取り組みです。

Problem
03 / 17
なぜ“全員化”が難しい？
同意と取り消し

OAuthは「どのアプリが・どの範囲を・どう許可するか」を管理できる一方、同意（consent）や取り消し（revocation）の安全設計が要になります。

従来の課題
第三者OAuthが手作業オンボード中心 / APIトークン依存が強い
委任アクセス
同意・取り消しの可視性が弱いと、誤認同意や運用事故につながる
Core Idea
04 / 17
解決の中心
self-managed開放

self-managed OAuthを全顧客向けに開放し、標準フローでスコープ付きの委任同意、分かりやすい同意、容易な取り消しを実現します。

同意
スコープ付き委任（ユーザーが理解可能）
取り消し
ダッシュボードから操作しやすい
権限制御
アプリ側で範囲を設計・制限できる
Architecture
05 / 17
OAuth基盤の本丸
Hydra大規模移行

Cloudflareが内部で利用するOAuthエンジン（Hydra）を大規模アップグレードし、1.x→2.xの移行を安全に成立させました。

理由
1.x→2.xはDBスキーマ変更が大きく、停止や長時間の影響を避ける必要
前提
OAuthはトークン発行/更新（refresh）/取り消しが絡むため、整合性が最重要
Process
06 / 17
段階的に“切り替える”
blue-green

段階的アップグレード（1.x→評価→2.x）とblue-green戦略で、本番DBと新バージョンを段階的に切替しました。

i
1.xを維持しつつ新経路を準備
ii
評価→2.xへ移行を段階化
iii
blue-greenで段階的に実切替
Risk
07 / 17
最大級の事故
revocation復活

移行中にrevocationが失われると、ユーザーが拒否したはずの権限が“復活”し得て、信頼性に直結する重大事故になります。

危険
取り消し漏れ → 権限が復活 → ユーザー意思に反する挙動
要求
切替後に取り消しを必ず反映（drain→replay）
Evidence
08 / 17
対策の核
Queuesで再生

Cloudflare Queuesを使い、取り消しイベントをキューイングして切替後にリプレイする設計で、取り消し漏れを防ぎます。

enqueue
revocationイベントを記録
drain
切替前後で取りこぼしを抑制
replay
切替後に確実に反映
Compatibility
09 / 17
互換性の落とし穴
refresh差分

1.x移行後にrefresh tokenエラーが増え、refreshトークン再利用が鎖全体を無効化するように厳格化された点が原因とされています。

影響例
Wrangler / MCPのようなリクエスト頻度の高いクライアントで再試行・再利用が起きやすい
緩和
Worker側で再利用検知、2.x側はgrace periodで調整
Outcome
10 / 17
良い結果と検証
3時間で安定

2.x移行は本番で約3時間、完了後はOAuthトラフィックが安定し、API p95が45%改善などの指標向上が報告されています。

移行時間
約3時間
性能
API p95：45%改善
資源
メモリ・CPU使用を大幅削減
Incident
11 / 17
同時に起きた問題
403増加

切替後に認可サービス側のデータクリーンアップが過剰に動作し、有効セッションが無効扱いになって403が増加しました。

原因
Hydra移行の一部の問題 → 認可サービス側との不整合
対応
復元対応を実施（“成功”に見えても検証が必要）
Takeaway
12 / 17
学びの要点
安全に拡張

self-managed OAuthは「可視化（同意）＋制御（取り消し）」を前に進める一方、大規模移行ではrevocation/refresh/整合性が最大論点になります。

Security
同意と取り消しの保証が設計の中心
Reliability
blue-green＋revocationリプレイが要
Compatibility
refresh運用の差分はクライアント相互作用が鍵
