COVER
02 / 16

端末内に“仕組み”がある

Stella顔認識パイプライン

Android版Stellaには、顔検出→整列→顔埋め込み→ベクトル検索→通知→未一致保存までの一連が“接続された形”で搭載されている可能性があります（ただし一般運用の有効性は未実証）。

PROBLEM
03 / 16

重要なギャップ

実装 ≠ 有効化

端末に機能が組み込まれていることと、一般の利用者が日常的に他人の顔を識別される状態で運用されることは別問題です。

できること（事実）

端末内でパイプラインが“動作する”ことを、テスト文脈で確認

言えないこと（未証）

通常ユーザーでのUI導線・通知の到達先・サーバー投入の実測

ARCHITECTURE
04 / 16

処理の流れ

検出→照合→通知

解析で、顔検出・整列・埋め込み生成（2048次元）・ベクトル検索（cosine等）・通知表示・未一致時保存の“つながり”が示唆されています。

i. 検出（Detection / 検出）

SCRFD など

ii. 埋め込み（Embedding / 埋め込み）

SFace：2048次元

iii. 照合（Search / 検索）

SQLite + sqlite-vec / person_profiles

EVIDENCE
05 / 16

“部品が噛み合う”

2048次元整合

顔埋め込みが float[2048] で設計され、検索側も同次元・距離指標（例：cosine）で一致させる構成が確認され、単なるデッドコードより“実運用想定”に寄ります。

データ形（Data shape）

face_mediaPath_vec：float[2048]

照合形（Matching）

sqlite-vec + cosine 距離で近傍探索

METAPHOR
06 / 16

連結された道具箱

“鍵穴まである”

顔検出・整列・埋め込み・検索インデックス・通知文言・未一致保存が揃うと、後から誰かを照合するための道具箱として最小限の連結が成立します。

モデル群（Models）

SCRFD / KPSAligner / SFace

運用部品（Ops）

通知チャネル・深リンク・保存分岐

ENUMERATION
07 / 16

未一致時の保存

未ラベルの“原材”

未一致（候補が特定できない等）の場合、クロップ画像と埋め込み（.jpg と .emb、2048次元）が NameTagsPending/ に保存される分岐が示唆されます。

保存先（Path）

NameTagsPending/

保存物（Artifacts）

.jpg（画像） + .emb（2048次元）

DISTINCTION
08 / 16

通知文言の存在

“Person recognized”

通知は「Person recognized」「Recognized <name>」のような文言と、深リンク（例：fb-viewapp://...）で人名表示へ誘導する設計が見えます。

文言（Text）

Person recognized / Recognized <name>

導線（Deep link）

深リンクで表示画面へ

※ただし該当画面が当該ビルドで欠落しており、通知は出ても導線が完結しない可能性が報告されています。

COMPARISON
09 / 16

できた/できない

エンドツーエンド検証

解析では、テスト文脈での“動作確認”はある一方、通常利用者が常時その識別を受けることを裏付ける観測（UI・サーバ投入）が提示されていません。

観測できた

ローカルで検出〜検索〜通知（少なくともハンドラ起動条件）

観測できない

一般ユーザーのUI表示・通知到達先・person_profilesへの実投入

PROCESS
10 / 16

理解のための前提

見るべき要素

この種の解析は、(1)ローカルSQLite、(2)埋め込み次元、(3)cosine類似度検索、(4)通知チャネル、(5)未有効化/不可視の可能性、を押さえると誤解が減ります。

SQLite（保存）

person_profiles / objects.db

埋め込み（表現）

2048次元（SFace）

通知・分岐（運用）

通知テンプレ + 未一致保存

DISTINCTION
11 / 16

リスク評価の注意

“確実同定”ではない

ベクトル検索があることは「候補を見つける」ことを示しますが、同定（確信）には閾値、誤一致率、ラベル付与の正確さが必要です。この入力では精度条件は読めません。

同定（Identification）

閾値・精度・ラベルの整合が鍵

提示されたもの

動作確認の中心で、誤認率や閾値の実測は未提示

CLOSING
12 / 16

結論（可能性と未確定）

仕組みは濃厚、運用は未確定

端末内の“連結された顔認識パイプライン”は強い示唆ですが、一般運用で他人の顔がどの条件でどれだけ識別されるかは、追加の検証と公開情報が必要です。

強く言える

モデルとデータ構造・検索と通知・保存分岐が整合

慎重に言う

UI有効化・サーバ同期・閾値・誤認・保存方針が未確定
