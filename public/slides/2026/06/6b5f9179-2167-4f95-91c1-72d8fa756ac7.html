道具の所有感

piが残った理由 pi

pi は「エンジニアが自分の道具を所有し続けるためのコーディング・エージェント」だ、というのが筆者の主張です。

PROBLEM
02 / 15

厚いハーネス問題

運用が固定される×

厚いハーネス（thick harness）は、計画モードやUI、サブエージェントなど“手順そのもの”を内蔵し、ユーザーのメンタルモデルを狭めます。

結果
適応のコスト（“製品に合わせる”）
副作用
方向一貫の追従負担／許可プロンプトの不満
METAPHOR
03 / 15

薄い道具、太いスキル

harness vs skills

薄いハーネス（thin harness）は入出力と実行基盤を最小にし、ワークフローは“拡張（スキル）側”で作る前提になります。

thin harness（薄い基盤）
fat skills（太いスキル）
ユーザー主導
ARCHITECTURE
04 / 15

4操作だけ

read / write / edit / bash

pi はツールをこの4つに絞り、計画モード、サブエージェント、To-do、MCP、権限ポップアップなどは“最初からは”持ちません。

提供するもの
read/write/edit/bash
任せるもの
計画・タスク・運用は拡張で実装
PROCESS
05 / 15

ビルド不要の拡張

TypeScript を即投入

拡張は TypeScript をランタイムロードし、ビルドや npm publish なしで次のターンに新しいコマンドとして使えます。

Step i
短いコードで拡張を書く
Step ii
拡張ファイルを書いて
Step iii
/reload で反映
EVIDENCE
06 / 15

探索を残す

セッションはツリー

pi のセッションは直線ログではなく木（tree）構造で、分岐・比較・並行検討ができるため文脈喪失が起きにくいとしています。

例
方針 A / 方針 B を並べて保持
得
戻りコスト（再説明）を減らす
COMPARISON
07 / 15

“初期体験の厚み”不足

便利機能は薄め

一方で pi は、Claude Code のような高度なセッション可視化TUI、git worktree分離、サンドボックス、ダッシュボード知能、大規模コミュニティ/チュートリアル等を欠くと明確に述べます。

DISTINCTION
08 / 15

期待のズレ

運用を“面倒見ない”

pi はワークフロー設計をユーザーに委ねるため、タスク管理やIssue tracking、意見のあるプロジェクト構造を期待するとフラストレーションになり得ます。

向く人
自分の運用を言語化して組める人
合わない人
“全部やってくれる道具”を求める人
ENUMERATION
09 / 15

設計論点の整理

piの論点は5つ

この記事の要点は「ハーネスが固定する影響」「4操作最小構成」「即時プロダクト化」「ツリーで探索」「面倒見の薄さ」です。

i
厚い運用＝学習方向が固定
ii
4操作＝拡張で自己設計
iii
ツリー＝分岐を損しない
EVIDENCE
10 / 15

リスクは“検証待ち”

安全設計の不明点

筆者は pi についてサンドボックスや自動隔離が欠ける可能性をリスクとして挙げますが、この記事だけでは安全性の担保範囲が不明です。

懸念
bash を伴う実行での事故
必要
権限の粒度・隔離・失敗時の扱いの確認
DISTINCTION
11 / 15

結論の読み方

優劣ではなく“好み”

この記事は製品の勝敗というより、設計思想の選好として読むのが妥当だと締めています。pi は強力だが万能ではなく、運用・拡張・セキュリティはユーザー側が引き受ける覚悟が必要です。

CLOSING
12 / 15

あなたはどっちが得意？

thin harness / fat skills

“道具に合わせる”のが苦ではないなら厚いハーネスが時短になり得る一方、“道具を自分に合わせる”なら pi が価値になります。

選び方（目安）
適応より設計（適応より設計できる）
次の一歩
まずは Claude Code 等で体験を掴み、その後 pi へ（比較学習）
