Evaluation’s Blind Spot
Trajectory

LLM-as-judge（LLMに採点させる方式）だけに頼ると、「何を読めたか／読めなかったか」という根拠の経路（trajectory）が検査できません。

その結果、境界違反（アクセス権・時点・証拠可視性）を見逃す評価が起きます。根拠の経路 = trajectory（軌跡）

problem
03 / 16
Judge Sees Destination
Not Route

judgeは多くの場合、最終回答が期待回答と一致するか（destination一致）だけを見ます。

Judge視点
最終回答の整合性（looks-right）
destination-match（目的地一致）
見落とし
途中で読めた証拠・未読・未探索の状態
trajectory / state（経路・状態）

経路

Failure Class
Absence

「存在しない」という主張（absence claim）が成立するには、本来“あり得る場所”を検索して否定を裏取りする必要があります。

正しい前提
検索 → 不在確認 → 否定の成立
裏取り（verification）
judgeが見逃す
未探索でも、それっぽい否定で前に進む
沈黙からの否定は危険

Final Text ≠ Proof
Repro

最終回答が“正しい文章に見える”だけでは、再現性や安全性の要件（アクセス権・時点・可視性）を満たした保証になりません。

Access

Trajectory Bench ≠ Governance
別軸

既存で測れる

state validity（状態の妥当性）

事前に定義

contract（契約）

11 / 16
Judge for Destination
Contract for Route

judgeは目的地一致、契約は経路妥当性。評価の問い（what we ask）を分けることで、抜けを埋められます。

目的地評価
最終回答の一致（LLM-as-judgeでも可能）
destination-match

route-state-check
closing

Takeaway
評価を変える

リーグ表の高スコアがあっても、境界違反がスコアに現れないことがあります。評価は「正解当て」から「制約下で成立」を測る方向へ拡張すべきです。
