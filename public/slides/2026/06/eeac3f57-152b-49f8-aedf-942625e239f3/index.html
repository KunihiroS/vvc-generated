脆弱性探索の“雛形”

Defending Code Reference Harness

Claudeを使って、脆弱性の探索から検証・レポート・修正案までを回す参照実装の全体像です。安全のため、実行を伴う段階はサンドボックス（gVisor）前提で設計されています。
対象
オープンソース参照実装（移植は前提あり）
狙い
“候補”ではなく“再現可能な根拠”まで積み上げる
Open-source reference implementation · Pipeline template
Problem / 課題
03 / 18
誤検知と危険な実行
“指摘”が止まらない
LLMの脆弱性指摘は候補が多くなりがちで、実行条件が不明確なため誤検知が混ざります。さらに、検証のためにコードを“実行”するとリスクが増えるため隔離が必須になります。
誤検知
静的スキャンだけでは実行時の事実が確定しない
安全
検証でターゲット実行が発生する可能性
Threat model · static vs dynamic · sandbox
Metaphor / 比喩
04 / 18
仮説→再現→確信
推理ではなく
実験へ
静的に見つけた“疑わしい箇所”を、クラッシュ再現で裏取りします。verifyは別環境で再現して、findの偶然性を排除する役割です。
recon
入力パース等を分解し、観測点を作る
find
並列に入力を生成し、クラッシュを探索
verify
新しいコンテナで再現して真偽を確かめる
detection → reproduction → confidence

入口
静的スキャン→トリアージ（スコープ絞り）
終端
patchの“妥当性確認”までパイプライン化
pipeline template · reusable workflow skeleton

安全寄り
ファイルI/O主体：/quickstart · /triage
隔離必須
実行を伴う探索・検証：gVisor（sandbox）前提
execution phases · reduce operational risk
Process / 運用
07 / 18
並列探索＋独立再現

偶然性を削る

reconで入力パースサブシステムを分割し、findが並列に異なる領域を探索します。さらにverifyはfindが触れていない新しい環境で再現し、状態依存の錯覚を減らす狙いです。
recon
分割：観測点を増やす
find
並列：クラッシュ探索
verify
独立環境：再現性確認
parallelism · independent verification
Evidence / 証拠
08 / 18
クラッシュを根拠へ

dedupe と再現性

dedupeでは既知バグとの比較で新規・改善・重複を判断します。これにより、レポートの“重さ”を下げ、検証の価値が伝わる形に整理します。
dedupe
既知バグ比較で重複除去（new / improved / duplicate）
目的
“候補”→“事実”への変換を促進
reduce noise · focus verification effort
Architecture / 構成
09 / 18
レポートは“意思決定用”

悪用可能性を構造化

reportは原始型（primitive class）や到達可能性、エスカレーションパス、重大度などを構造化して説明します。単発の提案ではなく、開発側が優先度付けしやすい形に寄せています。
primitive class
脆弱性の種類（例：メモリ系）
reachability
到達可能性の説明
escalation
悪用時のエスカレーション経路
structured report for triage + planning
Comparison / 差分
10 / 18
静的 vs 動的

“見つける”より

“確かめる”

静的スキャンは実行時条件や副作用を確定できず、偽陽性が増えがちです。find/verifyでクラッシュ再現を積み、信頼性を上げる方向性が明確です。
静的スキャン
候補が出る（誤検知は増えやすい）
動的検証
再現できる事実で裏取り（verifyが鍵）
static → hypotheses; dynamic → evidence
Evidence / 検証
11 / 18
patchの“再検証”

直した後も確かめる

patchではビルド成功、PoCでクラッシュしないこと、テストスイートの通過、修正回避の探索まで検証します。これにより、修正案を“採用候補”として扱える状態に寄せます。
build
ビルドが通る
poc
PoCでクラッシュしない
tests
テストスイート通過＋回避探索
patch validation · avoid “broken fixes”
Distinction / 限界
12 / 18
自律判断の未完

トリアージは

人と環境の領域

自律トリアージ/パッチは完全ではなく、重要度や優先度は環境依存の評価が残ります。検証済みのパッチでも、アップストリームに受け入れられない可能性があると明確に注意しています。
残る判断
重大度・優先度・取り込み可能性は依然として環境依存
実務影響
“パイプライン成功”＝“マージ成功”ではない
triage & patch autonomy · open challenge
Security / 安全
13 / 18
隔離と通信制限

gVisorで“実行”を封じる

実行を伴う段階ではgVisor隔離なしでは拒否され、エージェント起動もサンドボックス外では抑制される方針です。加えて許可された通信先（Claude API）に制限することで危険な挙動の抑制を狙います。
sandbox
gVisor隔離（実行フェーズの必須条件）
network
通信先をClaude API等に限定（許可制）
risk containment · safer autonomous workflows
Closing / まとめ

review · prioritize · decide
