チェーン型API
端点で 実行
Bun.Imageは「読み込み→変換→書き出し」をチェーンで合成し、`bytes()` / `buffer()` / `blob()` / `write()` / `dataurl()`などの“端点”を`await`した時点で初めて処理が走ります。
1) Input / 入力
パス・バイト列・Blob・BunFile・S3Fileなど
2) Transform / 変換
resize / rotate / flip / 色調補正 等
3) Terminal / 端点
`await`→ネイティブパイプラインで実処理

入力はバイトで扱う
パス直渡しは 危険
仕様上、パス文字列はファイルシステムパスとして扱われます。ユーザー由来なら、パスをそのまま渡さず「バイト列として受けて検証→渡す」が警告されています。
Do / 推奨
ユーザー入力→バイト→自前バリデーション→Bun.Imageへ
Don’t / 注意
ユーザー制御のパス文字列をそのまま渡す

メモリ暴発を封じる
maxPixels / デコンプレッションボム
画像はヘッダ上の寸法が不正な場合があり、巨大展開でメモリを枯渇させる恐れがあります。`maxPixels`は拒否を早めるための必須安全策です。
Why / 理由
ヘッダ読取後にバッファ確保しうるため
Guard / 対策
`maxPixels`でピクセル数の上限を設定して拒否

OS依存の現実
ERR_IMAGE_FORMAT_UNSUPPORTED
HEIC/AVIFなどはOSバックエンドに依存し、扱えない場合があります。フォーマット不対応は`ERR_IMAGE_FORMAT_UNSUPPORTED`になるのでフォールバック分岐が現実的です。
When / いつ
OS・環境差でデコード/エンコードが不可
How / どうする
エラー捕捉→例：AVIF不可ならWebPへ

10 / 15
サーバ統合の注意
await → Response
`Bun.serve`では`new Response(img)`でも動く可能性がありますが、ボディ初期化のタイミングで同期的にエンコードされる恐れがあります。サーバでは端点を先に`await`してからResponseに渡すのが推奨です。
Risk / リスク
Responseボディ初期化で同期エンコード
Best practice
`await`で先に出力→`new Response(out)`

12 / 15
まとめ：使いどころ
設計は 前提
Bun.Imageはネイティブ高速パイプラインで、端点`await`まで処理が走りません。実務では安全（パス・maxPixels）、互換（フォールバック）、サーバ同期回避（Response前await）を最初に組み込みましょう。
Performance
オフスレッド/ネイティブ連鎖（端点で実行）
Safety
パス直渡し禁止 / `maxPixels`でデコンプレッションボム対策
Compatibility
HEIC/AVIFはOS依存→`ERR_IMAGE_FORMAT_UNSUPPORTED`捕捉
Server
`await`してから`new Response(out)`
