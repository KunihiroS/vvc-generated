どこまで小さくできる？

最小オーケストレーション

エージェントの「考える→ツール選ぶ→実行→観測→再計画」を回すモデルは、能力ではなくツール呼び出し規律が先に崩れやすい。
DESIGN QUESTION
02 / 17
分けるべき責務

生成と 制御

エージェントは大きく、①重い一発生成（コード生成）と、②オーケストレーション（ReActループ）に分けられる。投稿は②だけを対象に、モデルを小さくする限界を探る。
① heavy generation / 重生成（a）
400行モジュール、巨大リファクタ等
専用で大きめモデルへ
② orchestration / 制御（b）
意図→ツール→実行→観測→再計画
小さくできる余地を検討
LOOP MINDSET
03 / 17
折れやすい関所

賢さより 規律

ループが破綻する瞬間は「推論が外れる」より先に、ツール呼び出しの引数・パラメータ規約が守れないことになりがち。
ReAct loop / 反復
考える → ツール選ぶ → 実行 → 観測 → 再び考える
Tool-call discipline / 規律
存在しない引数、誤型、結論をツール扱い 等
WHAT FAILS
04 / 17
劣化ポイント（例）

誤呼び出しが先

小さなモデルで起きやすいのは、タスク意図は近いのに「ツール呼び出し規約」が崩れる誤り。
i
存在しないパラメータを渡す
例：overwrite=true、output_mode など
ii
ツールでない“結論”を呼ぶ
「ツールとして出力」してしまう
iii
失敗から学習しない / 逆に反復
別ツールにも同じ誤引数を適用
iv
状態遷移が不安定
観測が取れずループが収束しない
LOCAL CONSTRAINT
05 / 17
大きくできない現実

“最小構成”を探る

llama.cpp のようなローカル推論環境では、VRAMや速度の制約が強い。だから「大きいモデルで解決」ではなく、最小のボトルネックを特定したい。
観点
モデル能力ではなく“仕様遵守”
tool-call discipline を主対象に
目標
ループ制御として実用性を確保
信頼できる最小モデルを絞る
COUNTERMEASURE
06 / 17
まず“署名”で止める

ツール 署名提示

システムプロンプトにツールの正確な引数リスト（ツール名と引数・デフォルト）を提示すると、存在しないパラメータを避ける方向に効き得る。
例（イメージ） / English + Japanese
tool_name(arg1, arg2, opt=default) // 引数スキーマ（引数一覧）
Non-existent parameters / 存在しないパラメータを“暗に学習”しないよう、情報として扱わせる。
RECOVERY LOOP

失敗の反復を断つ

ウォッチドッグ

同じ（tool, args）の失敗が繰り返されるのを監視し、戦略転換を促すことで応急処置としてループを守れる。ただし根本解決ではない。
detect / 検知
(tool, args)の反復失敗
switch / 切替
ツール変更、手順の再設計
CORE CLAIM
08 / 17
サイズより設計

“小さくする”より 制御を堅牢化

結論の仮説は「モデルを増やす前に、制御の正確さ（規律）を支える仕組みを先に作る」こと。そうすれば想像より小さくできる可能性がある。
Model scaling / モデル増
確実性は上がるが高コスト
ローカル制約では難しい
Control robustness / 制御強化
スキーマ提示・検証・回復
規律を上げて破綻を減らす

定量データの 不足

署名でどれだけ改善したか不明
Needed / 必要なもの
タスク多様性と境界条件
“35B-A3Bが境界”の根拠
RISK SAFETY
11 / 17
安全は別レイヤ

実行前バリデーション

ツール呼び出しの誤りは、誤操作や意図しないファイル変更につながり得る。モデルを小さくする前に、ツール実行前の引数検証（スキーマ検証）が重要になりうる。
Stronger than watchdog / 応急処置より
不正パラメータを“自動修復”ではなく“明確なエラー”として扱う
ループに正しい形式へ戻す誘導を組み込む
FAILURE MODES
12 / 17
誤りの切り分け不足

“引数だけ”か？

誤りがツール引数の捏造・過剰一般化に限定されるのか、ツール選択ミス、状態管理ドリフト、停止条件の設計など他の失敗モードが混在するのかが読み取れない。
Maybe / 可能性
ツール選択の誤り
“正しい呼び出し”に到達しない
Maybe / 可能性
状態ドリフト / 停止条件
収束・安全停止が崩れる

最小信頼点の指標

評価軸を揃える

投稿者の問題意識（tool-call discipline）を評価するには、少なくともツール呼び出し成功率、引数の妥当性、収束までのステップ数、失敗時回復性能が重要。
i
ツール呼び出し成功率
実行に到達できた割合
ii
引数の妥当性
存在しないパラメータ率、型ミスマッチ率
iii
収束までのステップ数
ループの長さ・打ち切り
iv
失敗時の回復性能
別手段に切替できるか

要点まとめ

小さくする鍵は 規律設計

オーケストレーションモデルの限界は「推論力」より「ツール呼び出し規律」で先に出やすい。ツール署名提示や検知・回復に加え、実行前のスキーマ検証まで入れると、より小さい構成が成立しやすい。
